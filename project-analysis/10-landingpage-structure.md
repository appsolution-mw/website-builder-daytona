# Landing Page Structure: Website Builder Daytona

Date: 2026-05-10
Scope: Premium SaaS landing page structure and conversion argument. This is a content and messaging structure, not UI implementation.

## Strategic Premise

Website Builder Daytona should not be sold as another AI website generator. The stronger landing page story is that AI web work becomes commercially useful only when it is isolated, visible, interruptible, inspectable, and handoff-ready.

The page should frame Daytona as:

> An AI web-production workspace that turns website change requests into isolated previews, reviewable code changes, and GitHub pull requests.

The buyer is not trying to avoid engineering forever. They are trying to remove the slow, lossy handoff between marketing or product intent and production-bound web code. The landing page must therefore sell governed throughput, not magic generation.

Primary conversion goal:

- Request demo or join beta for a controlled AI web-workspace workflow.

Secondary conversion goal:

- See a product walkthrough showing a real GitHub-backed project moving from prompt to preview to diff to pull request.

## Page-Level Argument

The page should make one cumulative argument:

1. Website teams are blocked by small requests that still require real engineering workflow.
2. Prompt-to-page tools do not solve the hard part because they stop before repo, preview, diff, cost, and PR.
3. Daytona gives the agent a real isolated project workspace, not a detached chat.
4. The user can supervise the work through live preview, visual context, files, terminal, console, queue state, and history.
5. The output becomes reviewable code and a GitHub pull request, not a screenshot or chat transcript.
6. Teams can reuse approved agent rules, workflows, and runtime choices while tracking cost.
7. The result is faster web-change throughput with less risk, less coordination drag, and more engineering trust.

## Section Sequence

### 1. Hero: The Trust Ceiling

Section goal:

- Establish the category and immediately separate Daytona from generic AI website builders.

Argument being made:

- The valuable outcome is not "AI made a page." The valuable outcome is "a web request became a reviewable code change."

Buyer psychology:

- The buyer already believes AI can generate something. Their unresolved fear is whether the result can safely enter their actual workflow.
- The hero should produce relief: this product understands the gap between impressive demos and production-bound work.

Key copy points:

- Turn website requests into reviewable code changes.
- Run AI web work in isolated project workspaces with live preview, code access, usage visibility, and GitHub handoff.
- Keep your repo, your review process, and your ability to inspect what changed.

Proof needed:

- Product screenshot or walkthrough frame showing chat, preview, code/history, and PR action in one workspace.
- Short evidence row: Isolated sandboxes, GitHub import, live preview, commit history, pull requests.

CTA placement:

- Primary CTA above the fold: "Request a workflow demo".
- Secondary CTA: "Watch prompt to PR".
- CTA should imply a serious workflow, not instant signup for a toy generator.

What not to say:

- "Build any website instantly."
- "No developers needed."
- "The autonomous AI developer for your whole company."
- "Launch your site in seconds."

### 2. Problem: The Website Backlog Is Not Really About Coding

Section goal:

- Name the true before-state: small web changes become slow because they require translation, setup, preview, review, and governance.

Argument being made:

- The expensive part of web work is the coordination loop around a change, not always the edit itself.

Buyer psychology:

- Growth and marketing buyers feel blocked by engineering queues.
- Engineering buyers feel protective of code quality and tired of low-leverage interruptions.
- Agencies feel margin pressure from repeated setup, QA, and review overhead.

Key copy points:

- A "simple" landing page change still needs a branch, running app, preview, visual review, console checks, diff review, and PR.
- Tickets lose visual context. Screenshots drift from the real app. Local agent sessions leave no team-visible record.
- Generic generators create artifacts that often have to be rebuilt before they can reach the repo.

Proof needed:

- A workflow contrast, not vague pain bullets: ticket -> local setup -> preview -> revision -> diff -> PR.
- Customer language examples: "I can see the issue, but describing it precisely is exhausting"; "I need a PR, not a pretty screenshot."

CTA placement:

- No hard CTA yet. A subtle inline link can point to the walkthrough: "See how Daytona keeps the loop in one workspace."

What not to say:

- Do not insult engineering teams as slow.
- Do not frame marketers as incapable.
- Do not say AI replaces the web team.

### 3. Product Reveal: A Real Workspace For Agent-Run Web Work

Section goal:

- Show the product as the missing operating layer around AI coding agents.

Argument being made:

- Daytona gives AI agents the same things serious web work needs: a repo-backed project, isolated runtime, live preview, files, terminal, console, history, and handoff path.

Buyer psychology:

- Buyers need a concrete mental model. "Workspace" is more credible than "magic builder."
- Technical evaluators want to see that this is not just a chat UI with an iframe.

Key copy points:

- Start from a template or connected GitHub repository.
- Provision an isolated Docker-backed project sandbox.
- Prompt agents with text, screenshots, or captured preview regions.
- Inspect the live app across desktop, tablet, and mobile frames.
- Open files, terminal, console, environment settings, and agent config without leaving the workspace.

Proof needed:

- Product surface breakdown with real workspace capabilities.
- Screenshot callouts should emphasize working surfaces: chat, preview, file tree, terminal, history.

CTA placement:

- Secondary CTA after feature reveal: "Explore the workspace".

What not to say:

- Do not over-index on infrastructure acronyms in buyer copy.
- Do not lead with worker pools, HMAC, Tailscale, or provider internals.
- Do not present terminal/code access as required for every user; frame it as control when needed.

### 4. Workflow: Prompt To Preview To Pull Request

Section goal:

- Turn the abstract promise into a believable sequence.

Argument being made:

- Daytona compresses the complete web-change loop into one controlled workspace.

Buyer psychology:

- Buyers need to imagine their next request moving through the product.
- The page should reduce perceived risk by showing clear stopping points and review gates.

Key copy points:

1. Import or create a project.
2. Ask for the change with text, images, or captured preview context.
3. Watch the agent run inside the project sandbox.
4. Inspect the live result and browser console.
5. Review changed files, commits, runtime/model metadata, and cost.
6. Open a GitHub pull request when the work is ready.

Proof needed:

- A real or demo walkthrough with one concrete task, such as "Update the pricing hero and fix mobile spacing."
- Show status states: queued, running, blocked, retry, skip, completed.
- Show a commit or diff, not only the final preview.

CTA placement:

- Primary CTA at the end: "Book a prompt-to-PR demo".

What not to say:

- Do not imply every run will be perfect.
- Do not hide the review step.
- Do not promise production deployment unless deployment is actually in scope.

### 5. Control Layer: Let The Agent Work Without Losing The Wheel

Section goal:

- Address the central adoption anxiety: "What if the agent does something wrong?"

Argument being made:

- Daytona is built around supervision, recovery, and auditability.

Buyer psychology:

- This section is for engineering, security, and technical founders. It should sound calm and specific.
- The emotional promise is: "You can delegate without surrendering control."

Key copy points:

- Runs are queued per project so overlapping agent writes do not collide.
- Active runs can be cancelled.
- Failed runs surface blocked states with retry or skip options.
- File editing locks while the agent is active.
- Agent output is persisted as events, commits, changed files, and usage records.
- Terminal, code editor, and browser console remain available as escape hatches.

Proof needed:

- Product screenshots of run state, changed files, history, and diff inspection.
- Technical proof list: isolated sandbox, serialized queue, persisted events, commit history, GitHub PR handoff.

CTA placement:

- No aggressive CTA. Link to "Review the controls" or "Talk to us about your workflow" near the end.

What not to say:

- Do not say "safe AI coding" as a broad claim.
- Do not claim autonomous production changes.
- Do not imply Daytona removes code review.

### 6. Visual Feedback Loop: Point At The Problem, Then Let The Workspace Change

Section goal:

- Show why Daytona is specifically strong for websites and web-app UI work.

Argument being made:

- Website work is visual, subjective, and iterative; Daytona keeps that feedback inside the same live project loop.

Buyer psychology:

- Designers and growth marketers need to feel seen here. They do not want to write CSS tickets to describe visual intent.
- Engineers should see that visual feedback becomes project context, not vague comments.

Key copy points:

- Capture a region from the live preview and attach it to the next prompt.
- Add screenshots, mockups, and visual references directly to the agent conversation.
- Check desktop, tablet, and mobile preview frames before requesting review.
- Use browser console capture to spot runtime errors without a separate debugging session.

Proof needed:

- A before/after visual correction example.
- Show captured preview region becoming a chat attachment.
- Show device-frame preview, ideally with the same page across sizes.

CTA placement:

- Secondary CTA: "See visual iteration in action."

What not to say:

- Do not make generic claims about "beautiful designs."
- Do not compete with v0 or Webflow on first-draft visual generation.
- Do not promise brand-perfect output without showing brand rules or review.

### 7. Governance: Reusable Rules, Workflows, And Runtime Choice

Section goal:

- Elevate the product from individual tool to team-operating system.

Argument being made:

- Teams need repeatable agent behavior, not one-off prompt luck.

Buyer psychology:

- Platform and agency buyers need confidence that successful workflows can be reused across projects and people.
- Runtime choice should feel like resilience and task-fit, not extra complexity.

Key copy points:

- Project-specific AGENTS.md content lets teams encode local rules.
- Skills, agents, and workflow presets can be versioned, published, rolled back, and reused.
- Runtime and model choices can be selected per session while the project remains the durable center.
- Library snapshots help preserve which workflow influenced a session.

Proof needed:

- Library and agent-config screenshots.
- Concrete examples of reusable workflows: landing page implementation, visual QA pass, accessibility repair, PR-ready change.
- Show versioning or revision history where possible.

CTA placement:

- CTA for team buyers: "Design your team workflow with us."

What not to say:

- Do not ask non-technical buyers to care about raw model lists.
- Do not market "more settings" as the benefit.
- Do not imply all runtimes have identical policy, cost, or capability behavior unless that parity is proven.

### 8. Cost And Operations: See What The Work Consumes

Section goal:

- Address AI cost uncertainty and operational maturity.

Argument being made:

- Agentic web work should be metered, observable, and governable by project and turn.

Buyer psychology:

- Economic buyers fear surprise bills.
- Agency operators care about margin by client/project.
- Engineering buyers want operational seriousness.

Key copy points:

- Track token usage and cost by project and completed turn.
- See input, output, cache write/read tokens, turns, and recent activity.
- Admin and operator surfaces support worker capacity, sandbox lifecycle, readiness, cleanup, and retries.
- Daytona turns AI web work into an accountable operating workflow.

Proof needed:

- Usage dashboard screenshot with project-level cost.
- Worker/admin proof can be mentioned as an enterprise/operator capability, but should not dominate the public page.

CTA placement:

- Soft CTA: "Talk through pricing and capacity needs."

What not to say:

- Do not promise exact savings unless backed by data.
- Do not claim all runtime costs are fully known if some providers do not return complete priced usage.
- Do not make infrastructure management sound like something every buyer must handle.

### 9. Use Cases: The Work Daytona Should Own First

Section goal:

- Make the product feel immediately applicable without becoming a generic feature grid.

Argument being made:

- Daytona is best for recurring, production-bound web changes where speed and control both matter.

Buyer psychology:

- Each ICP should see a familiar high-value job.
- The section should avoid "for everyone" breadth.

Key copy points:

- Growth teams: landing pages, pricing copy, CRO variants, SEO page updates, campaign pages.
- Web platform teams: controlled AI workspaces for non-engineering requestors, with reviewable output.
- Agencies: client repo sandboxes, reusable playbooks, previewable revisions, PR handoff.
- Technical founders: faster product-site and dashboard UI iteration without losing code ownership.

Proof needed:

- Use-case cards should include input, workspace action, and output:
  - Input: screenshot and request.
  - Workspace action: agent run with preview and diff.
  - Output: reviewed commit or pull request.

CTA placement:

- CTA after use cases: "Map Daytona to our web workflow."

What not to say:

- Do not list every possible software task.
- Do not position backend-only or mobile-only teams as core buyers.
- Do not present no-code brochure-site creation as the main use case.

### 10. Differentiation: Not A Generator. Not A Local Agent Session.

Section goal:

- Clarify category boundaries and prevent commodity comparisons.

Argument being made:

- Daytona sits between pure AI builders and raw coding agents: more production-aware than a generator, more collaborative and visual than a local CLI session.

Buyer psychology:

- Buyers compare against Webflow, Lovable, Replit, v0, GitHub Copilot, local Claude Code/Codex, and internal scripts.
- The page should help them understand when Daytona is the right tool.

Key copy points:

- Compared with prompt-to-page tools: Daytona keeps the repo, runtime, diff, and PR path in view.
- Compared with local coding agents: Daytona makes the run visible to the team and tied to preview, usage, and history.
- Compared with broad app builders: Daytona focuses on controlled web and UI work where preview and review matter.
- Compared with no-code builders: Daytona preserves real code ownership and GitHub workflow.

Proof needed:

- A concise comparison table based on workflow outcomes, not attack-copy.
- Rows: existing repo, isolated runtime, live preview, code/terminal access, persisted agent runs, commit history, GitHub PR, reusable agent rules, usage visibility.

CTA placement:

- No hard CTA needed if the next section is proof.

What not to say:

- Do not name competitors in a hostile way.
- Do not claim Daytona is universally better.
- Do not compare on vague "AI quality"; compare on workflow control.

### 11. Proof: Show The Product Doing A Real Job

Section goal:

- Replace abstract claims with grounded evidence.

Argument being made:

- Daytona's differentiator is visible in the workflow, so the proof should be a job story, not logo wallpaper.

Buyer psychology:

- Serious buyers need to see the system handle the messy middle: run state, files, preview, diff, cost, PR.
- Proof should feel operational and specific.

Key copy points:

- "A campaign page request becomes a sandboxed implementation, responsive preview, changed-file diff, costed run, and PR."
- "A visual spacing issue is captured from preview, attached to chat, fixed, and recorded as a commit."
- "A team preset applies approved agent instructions across similar web tasks."

Proof needed:

- Best: short product video or interactive walkthrough.
- Strong: annotated screenshots from one coherent scenario.
- Later: customer quotes with specific outcomes, such as cycle time reduction, fewer engineering interruptions, more landing page variants shipped, or lower agency delivery overhead.

CTA placement:

- Primary CTA after proof: "Request the walkthrough."
- Secondary: "Bring your GitHub workflow."

What not to say:

- Do not fabricate customers, logos, metrics, or testimonials.
- Do not use generic review quotes like "game changer" without concrete context.

### 12. Security And Trust Notes: Specific, Measured, Credible

Section goal:

- Give technical evaluators enough confidence to continue, without turning the page into a security whitepaper.

Argument being made:

- Daytona contains agent work and routes it through observable review steps.

Buyer psychology:

- Security and engineering reviewers punish vague claims.
- Specific constraints are more persuasive than sweeping promises.

Key copy points:

- Project work runs in isolated sandboxes.
- Host-to-worker control paths use authenticated calls.
- Project access is tied to authenticated users and workspace membership.
- Environment variables are managed as project configuration rather than pasted into prompts.
- Pull requests keep production-bound changes inside existing review practices.

Proof needed:

- Link to technical security documentation when available.
- Mention policy and runtime-control details only where parity is proven.

CTA placement:

- Technical CTA: "Discuss security and repo access."

What not to say:

- Do not claim compliance certifications that do not exist.
- Do not claim "the AI cannot make mistakes."
- Do not imply secrets are safe if users paste them directly into prompts.

### 13. Final CTA: Give AI Web Work A Reviewable Path

Section goal:

- Restate the transformation and drive one clear action.

Argument being made:

- If your team already has web requests, repos, previews, and review gates, Daytona gives AI work a controlled lane through that system.

Buyer psychology:

- The final CTA should feel like a next step for a serious workflow evaluation, not a casual toy trial.

Key copy points:

- Move from tickets and disconnected AI sessions to isolated previews, inspectable changes, and GitHub pull requests.
- Give growth, design, engineering, and agencies one place to request, supervise, review, and hand off web work.
- Start with one recurring web-change workflow and prove the loop.

Proof needed:

- Repeat the strongest proof cluster: isolated workspace, live preview, commit history, GitHub PR, usage visibility, reusable workflows.

CTA placement:

- Primary CTA: "Request a workflow demo".
- Secondary CTA: "Watch prompt to PR".
- Optional low-friction field: work email plus "What web workflow do you want to speed up?"

What not to say:

- Do not end with "start building anything."
- Do not introduce new feature claims in the final CTA.
- Do not shift from controlled workflow to generic automation.

## Proof Inventory To Gather Before Writing Final Copy

Must-have proof:

- Product screenshot or video of the workspace with chat, live preview, and code/history visible.
- Example GitHub-backed workflow ending in a pull request.
- Commit history or diff screenshot showing agent-produced changes.
- Usage dashboard screenshot showing per-project or per-turn cost.
- Visual feedback example using preview-region capture.

Strong later-stage proof:

- Named customer quote with a specific workflow outcome.
- Time-to-PR comparison for a recurring website task.
- Number of landing page variants, fixes, or client revisions shipped through Daytona.
- Internal security or architecture note for sandboxing, queueing, and repo access.
- Example library preset or AGENTS.md rule set that improved repeatability.

Do not use as proof:

- Generic AI screenshots with no repo or diff.
- Decorative dashboards without a real task story.
- Fabricated logos, metrics, or testimonials.
- Claims about replacing engineering review.

## Messaging Guardrails

Say:

- AI can make the change. Daytona makes the change inspectable, steerable, and reviewable.
- Turn website requests into isolated previews, commits, and pull requests.
- Keep code ownership and review discipline while using agents for implementation work.
- Give teams reusable workflows for recurring web changes.
- Control the loop around agent work: preview, inspect, revise, commit, hand off.

Do not say:

- No engineers needed.
- Fully autonomous developer.
- Build any app instantly.
- AI website maker.
- Vibe coding for teams.
- Safe AI coding.
- Production-ready every time.
- We replace your CMS, IDE, GitHub, and deployment pipeline.

## Recommended Page Tone

The tone should be practical, premium, and operationally literate.

Use:

- Clear verbs: request, run, preview, inspect, revise, commit, hand off.
- Specific nouns: sandbox, workspace, preview, diff, queue, runtime, pull request, usage.
- Calm confidence: "reviewable", "isolated", "observable", "recoverable".

Avoid:

- Breathless AI hype.
- Founder-mode slogans.
- Generic SaaS abstraction such as "streamline collaboration", "unlock productivity", or "supercharge workflows" unless followed by a concrete mechanism.
- Infrastructure-first language before the buyer understands the workflow value.
