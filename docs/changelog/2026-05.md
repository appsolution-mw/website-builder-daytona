### T-20260508-001 - Image-Input-Modelle filtern + Codex-Multimodal-Fix

- OpenRouter-Modell-Picker zeigt jetzt nur Modelle, deren
  `architecture.input_modalities` sowohl `text` als auch `image` enthält.

  Code-Pfad unverändert (CLI auto-detect funktioniert weiter).

## 2026-05-07

### T-20260507-012 - Recover broker tokens after worker-agent restart

- Worker-agent now exposes `POST /sandboxes/:id/attach-token` (HMAC).
  When a Watchtower update wipes the in-memory `brokerTokens` map, the
  host can re-feed an existing token without rebuilding the sandbox.
- New host helper `withTokenRecovery` retries any 409
  `broker-token-missing` once after fetching the persisted token from
  `SandboxToken` and re-attaching it. Wired into `executor-client.ts`
  (`executeProjectRun`, `cancelProjectRun`) and `broker-git.ts`
  (`getProjectGitStatus`, `pushProjectGitChanges`).
- Affected files: `worker-agent/src/server.ts`,
  `worker-agent/tests/server.test.ts`,
  `lib/runtime/worker-pool/{types,agent-client,fake-agent-client,with-token-recovery}.ts`,

### T-20260507-010 - Sandbox restart policy + Watchtower bootstrap script

- Sandbox containers now created with `RestartPolicy: { Name: "unless-stopped" }`
  so cold reboots of a worker VM bring them back up automatically (matching
  the worker-agent and watchtower policies).
- Added `scripts/bootstrap-watchtower.ts` — idempotent script that walks
  every active Hetzner worker in the DB and, via SSH on the stored
  Tailscale IP, installs Watchtower (Nicholas-Fedor fork, with
  docker-config auth mount) and recreates the worker-agent with the
  Watchtower label + `IMAGE_REGISTRY_*` env vars when needed. Supports
  `--only <name>` and `--dry-run`.
- Affected files: `worker-agent/src/docker.ts`,
  `worker-agent/tests/docker.test.ts`, `scripts/bootstrap-watchtower.ts`.

### T-20260507-009 - Project readiness gating + Watchtower auto-update

- Added explicit `Project.brokerReady` flag fed by a worker-agent → host
  HMAC callback so the Open button and workspace UI only unlock when the
  in-container broker is actually listening, instead of trusting a
  too-early `status=RUNNING` flip. Includes self-heal pass on worker-agent
  start that re-probes every running sandbox.
- Replaced `crypto.randomUUID()` (secure-context only) with a polyfilled
  `randomId()` so HTTP-served deployments stop throwing `TypeError`.
- Added Watchtower (Nicholas-Fedor fork) to the worker bootstrap, plus a
  CI step that joins the Tailnet via OAuth and triggers Watchtower's
  `/v1/update` on every `tag:wbd-worker` device. Worker-agent now passes
  `IMAGE_REGISTRY_*` to dockerode so private GHCR pulls succeed, and
  always re-pulls the sandbox image on boot. New workers pick up future
  builds automatically; existing workers needed a one-shot bootstrap.
- Hardened `POST /api/projects` so a `createRuntime()` env-var failure
  marks the project DESTROYED with a real error instead of stranding it
  in PROVISIONING.
- Affected files: `prisma/schema.prisma` + two migrations,
  `lib/random-id.ts` (new), `worker-agent/src/{index,server,broker-ready}.ts`,
  `container/sandbox/broker/src/ws-server.ts`,
  `app/api/internal/sandboxes/[id]/broker-ready/route.ts` (new),
  `app/api/projects/{route,[id]/restart/route}.ts`,
  `lib/runtime/{types,daytona/fake,provisioner/{cloud-init,hetzner}}.ts`,
  `app/page.tsx`, `app/project/[id]/page.tsx`,
  `components/workspace/XtermTerminal.tsx`,
  `.github/workflows/worker-agent.yml`.
