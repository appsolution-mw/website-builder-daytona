# Phase 1.4a-B — Rollback / Revert UI — Design

Task: [T-20260510-003](../../tasks/active/T-20260510-003.md)
Date: 2026-05-10
Status: Draft (awaiting user review)

## 1. Goal

Give the user a one-click way to roll the project back to the state of any
historical commit. The action is reversible, preserves audit history, leaves
chat untouched, and integrates with both the History panel (Phase 1.4a) and
the chat-turn commit chip.

This is the second slice of Phase 1.4 from the master spec
(`2026-04-22-website-builder-daytona-design.md` §7.3, §9 Phase 4). It builds
directly on the data model, broker primitives, and UI components delivered by
Phase 1.4a (`2026-05-09-phase-1.4a-commits-per-turn-design.md`).

The remaining slices ship later as their own specs:

- **1.4a-C** — User-edit commits with a separate author identity
- **1.4d**   — Pause/Resume + Archive lifecycle automation
- **1.4e**   — Per-turn / per-user cost quota and hard limits

## 2. Decisions (recap from brainstorming)

| # | Decision | Rationale |
|---|---|---|
| 1 | Rollback is implemented as a **revert-commit**, not `git reset --hard`. This explicitly overrides the `reset --hard` line in master spec §7.3. | Keeps history linear and audit-stable, makes rollback itself reversible (a revert of a revert is just another revert), and keeps GitHub push fast-forward. Consistent with 1.4a Decision 5. |
| 2 | Chat / session history is **not** modified by a rollback. | Audit trail stays intact; History panel remains the single source of "what changed in the codebase"; keeps DB writes scoped to the `Commit` table. |
| 3 | Rollback is allowed only when the project is **idle** — no `RUNNING` agent run, no `QUEUED` queue items. | Prevents agent / revert filesystem races. The existing skip / cancel controls (T-20260507-003) cover the not-idle case; we don't reinvent them. |
| 4 | Rollback is allowed only when the **working tree is clean**. | Editor edits are uncommitted between agent turns until 1.4a-C lands. Blocking is the conservative choice — no edit ever goes missing. 1.4a-C will relax this. |
| 5 | The Revert affordance lives **in CommitDetail (History mode)** and **on chat-turn commit chips**. | v0 / Lovable convention; discoverable from both contexts; one shared hook owns the state machine. |
| 6 | Revert is gated by a **confirm modal with a summary** — title, file list, +/− stats, and the "current state stays in history" reassurance. | Cheap protection against accidental clicks without slowing the common case. Diff preview is unnecessary because the user can already inspect the diff in History mode before clicking. |
| 7 | Revert commits stay **local** — no auto-push to GitHub. | Consistent with 1.4a Decision 8. Push remains an explicit user action through the existing `commitAndPushChanges` path. |
| 8 | Revert commit author = `launchnode-agent <agent@launchnode.de>`. New `CommitAuthorKind = ROLLBACK` distinguishes the row in the DB and UI. | Single bot identity matches the 1.4a author convention; the new enum value gives History mode an unambiguous icon / label. USER identity stays reserved for 1.4a-C. |
| 9 | Reverting to the current `HEAD` is a no-op: the UI hides the button on the HEAD row, and the broker rejects with `is_head` if a stale UI tries it. | Trivial guard; avoids creating empty revert commits. |

## 3. Architecture

```
Workspace (browser)
  ┊  Click on Revert button
  ┊    — CommitDetail (History mode), or
  ┊    — chat-turn commit chip
  ▼
RevertConfirmDialog
  ┊  Reads pre-loaded title / files / shortstat from existing
  ┊    /api/projects/[id]/commits/[sha]/files endpoint
  ▼  User confirms
POST /api/projects/[id]/commits/[sha]/revert
  ┊  Host pre-checks
  ┊    — Project run state idle (no RUNNING / QUEUED)
  ┊    — Commit row exists for this projectId+sha
  ▼
broker-rpc → Broker.revertToCommit({ sha })
  ┊  git status -z   → empty?  no  → reject with `dirty_tree`
  ┊  git rev-parse HEAD == sha?    → reject with `is_head`
  ┊  git checkout <sha> -- .
  ┊  git commit
  ┊      --author "launchnode-agent <agent@launchnode.de>"
  ┊      -m "Revert to <shortSha> — <originalTitle>"
  ┊      -m <body with original-sha + triggered-by>
  ┊  git rev-parse HEAD                       → newSha
  ┊  git diff --shortstat HEAD~1 HEAD          → files / + / −
  ┊  emit `git.commit` (existing channel) with authorKind=ROLLBACK
  ▼
Worker-Agent → Host event-sink
  ┊  Writes Commit row  (authorKind=ROLLBACK, agentRunId=null,
  ┊                      revertedFromSha=<original sha>)
  ▼
Browser (existing WS subscription)
  ┊  History prepends the new row, chat shows the new commit chip,
  ┊  toast: "Reverted to '<original title>'"
```

- **Broker** still owns all git operations. The agent-runner is uninvolved
  (the runner is idle by Decision 3).
- **Host** does not need a new event ingress — the existing
  `git.commit` event channel (Phase 1.4a) carries the rollback row, with the
  new `authorKind` value and `revertedFromSha` payload field.
- **Frontend** uses one shared hook (`use-revert-commit`) for both UI surfaces
  so the confirm / loading / error / disabled state machine lives in one
  place.

## 4. Schema

One additive Prisma migration. No data backfill — existing rows stay with
their current `authorKind`.

```prisma
enum CommitAuthorKind {
  AGENT
  USER
  ROLLBACK
}

model Commit {
  // ...existing fields from Phase 1.4a

  // Phase 1.4a-B additions
  revertedFromSha String?  // 40-char hex; only set when authorKind=ROLLBACK
}
```

`revertedFromSha` is **not** a foreign key to `Commit.sha` because `sha` is
not unique across the table (master spec §7.3 + 1.4a §4 leave uniqueness as
`(projectId, sha)` but never enforce it). The host event-sink stores it as a
plain string. The History route resolves it lazily by looking up the
`(projectId, revertedFromSha)` pair when present.

## 5. Protocol Extension (`@wbd/protocol`)

Bump to `1.16.0`. New RPC + extended event payload.

```ts
type RevertToCommitRequest = {
  type: "rpc.revert.to.commit";
  id: string;          // RPC correlation id
  sha: string;         // commit sha to revert to
};

type RevertToCommitResponse =
  | {
      type: "rpc.revert.to.commit.ok";
      id: string;
      newSha: string;
      shortSha: string;
      filesChanged: number;
      insertions: number;
      deletions: number;
      revertedFromSha: string;
    }
  | {
      type: "rpc.revert.to.commit.err";
      id: string;
      reason:
        | "dirty_tree"
        | "unknown_sha"
        | "is_head"
        | "commit_failed";
      detail?: string;   // sanitised git error, only when commit_failed
    };
```

The existing `git.commit` event (Phase 1.4a) is extended with two optional
fields. Older brokers omit them; older host code already ignores unknown
fields.

```ts
type GitCommitEvent = {
  // ...existing 1.4a fields
  authorKind: "AGENT" | "ROLLBACK";   // "USER" reserved for 1.4a-C
  revertedFromSha?: string;            // only when authorKind = ROLLBACK
};
```

`git.commit.skipped` is unchanged.

## 6. Host API

One new route. Existing `/commits/[sha]/files` and `/commits/[sha]/diff`
power the confirm modal — no new read endpoints.

```text
POST /api/projects/[id]/commits/[sha]/revert
```

- **Auth**: project ownership + session, identical to existing commit routes.
- **Body**: empty (the path already names the target sha).
- **Pre-checks (host, before broker call)**:
  1. Lookup `Commit { projectId, sha }`. Missing → 404 `unknown_sha`.
  2. Sha equals current `HEAD` (latest commit row for the project) → 409
     `is_head`. Cheap shortcut so we don't pay broker round-trip.
  3. Active run check: any `AgentRun` with `status ∈ {QUEUED, RUNNING}` for
     the project → 409 `not_idle`. Mirrors the existing queue-state checks
     used by skip / cancel.
- **Broker call**: `broker-rpc.revertToCommit({ sha })`. Errors map 1:1 to
  HTTP responses.
- **Success response** (201): `{ newSha, shortSha, filesChanged, insertions,
  deletions, revertedFromSha }`. The host does **not** wait for the
  `git.commit` event — the event-sink writes the Commit row asynchronously,
  and the WS subscription updates the UI from there. The 201 body is enough
  for the optimistic confirm-toast.

The host route does **not** write the Commit row directly. It relies on the
broker → worker-agent → host event channel that 1.4a already runs. This
keeps "write Commit row" in exactly one place.

Failure responses:

| HTTP | reason            | When                                   |
|------|-------------------|----------------------------------------|
| 400  | `dirty_tree`      | Broker found uncommitted changes.      |
| 404  | `unknown_sha`     | No Commit row for `(projectId, sha)`.  |
| 409  | `is_head`         | Sha equals project HEAD.               |
| 409  | `not_idle`        | RUNNING / QUEUED run blocking.         |
| 500  | `commit_failed`   | Broker's `git commit` failed.          |

## 7. Broker

New function in `container/sandbox/broker/src/git-handlers.ts`, peer to the
existing `commitAgentTurn`. Reuses `runGit`, `getGitStatus`, the
`sanitizeGitOutput` helper, and the `readShortstat` parser.

```ts
export interface RevertToCommitRequest {
  projectRoot: string;
  sha: string;            // 40-char hex
  triggeredBy: string;    // sanitised "user:<userId>" string
}

export type RevertToCommitResponse =
  | {
      ok: true;
      newSha: string;
      shortSha: string;
      filesChanged: number;
      insertions: number;
      deletions: number;
      revertedFromSha: string;
    }
  | {
      ok: false;
      reason: "dirty_tree" | "unknown_sha" | "is_head" | "commit_failed";
      detail?: string;
    };

export async function revertToCommit(
  input: RevertToCommitRequest
): Promise<RevertToCommitResponse>;
```

Sequence inside the function:

1. `git rev-parse --verify <sha>^{commit}` — unknown / non-commit → return
   `unknown_sha`.
2. `git rev-parse HEAD` — equals `<sha>` → return `is_head`.
3. `getGitStatus({ projectRoot })` — non-empty porcelain output → return
   `dirty_tree`.
4. `git checkout <sha> -- .` — restores files in working tree + index.
5. `git commit --author "launchnode-agent <agent@launchnode.de>" -m <title>
   -m <body>`. Title and body composed by the broker (sanitised, byte-bounded,
   reusing `sanitizeCommitTitle` for the title). On failure → `commit_failed`.
6. `git rev-parse HEAD` → `newSha`; `readShortstat(HEAD~1, HEAD)` → stats.
7. Emit existing `git.commit` event with `authorKind = ROLLBACK` and
   `revertedFromSha = <sha>`. The same channel that 1.4a uses; the
   worker-agent / host sink already trusts it.
8. Return `ok`.

Title and body composition:

```text
Revert to <shortSha> — <sanitised original title, truncated to 72 chars>

Reverted-from: <sha>
Triggered-by: user:<userId>                  # body only, never the title
```

`Triggered-by` is the authenticated `userId` from the host route's session.
The host RPC call passes it to the broker as part of the
`RevertToCommitRequest`. The event-sink already drops the body into
`Commit.bodyMessage`; no new storage path.

## 8. Frontend

### 8.1 Shared hook

```ts
// lib/workspace/use-revert-commit.ts
type RevertState =
  | { status: "idle" }
  | { status: "confirm"; commit: CommitSummary }
  | { status: "submitting"; sha: string }
  | { status: "error"; sha: string; reason: ApiReason; message: string }
  | { status: "success"; newSha: string };

function useRevertCommit(projectId: string): {
  open: (commit: CommitSummary) => void;
  cancel: () => void;
  confirm: () => Promise<void>;
  state: RevertState;
};
```

Both placements (`CommitDetail`, chat-turn chip) call `open(commit)`. The
hook owns the modal, the POST, the toast, and the disabled-while-not-idle
guard. The History mode receives the new Commit row through the existing WS
subscription, so the hook does not need to mutate local state on success
beyond closing the modal.

### 8.2 RevertConfirmDialog

`components/workspace/RevertConfirmDialog.tsx` — shadcn/ui `Dialog` shell.

The dialog has two render variants. The default variant shows the Revert
summary; if the POST returns `dirty_tree`, the dialog swaps in place to the
blocked variant rather than closing. `not_idle` is handled upstream by the
button itself (disabled with tooltip — see 8.3) and never reaches the
dialog.

| State after open | Variant                                                          |
|------------------|------------------------------------------------------------------|
| Default          | Title, file list, +/− stats, "current state stays in history", `[Cancel] [Revert]` |
| `dirty_tree` from POST | "Uncommitted changes — run a turn or discard them first", `[Got it]` |
| `unknown_sha` / `commit_failed` | Generic error message, `[Got it]`                            |

The dialog never pre-fetches git status. We trust the broker as the single
authoritative check; the rare dirty-tree case surfaces as a one-extra-click
swap, which keeps the implementation small and avoids an extra round-trip
on the common case.

### 8.3 CommitDetail integration

`components/workspace/CommitDetail.tsx` adds a Revert button in the header,
right of the title. Button states:

| State                              | Render                                  |
|------------------------------------|-----------------------------------------|
| Commit is HEAD                     | Hidden                                  |
| Project read-only (existing flag)  | Hidden                                  |
| Active run (`RUNNING` / `QUEUED`)  | Disabled, tooltip: "Active run — abort or wait" |
| Otherwise                          | Enabled                                 |

Run-state comes from the workspace's existing chat / queue state; no new
subscription. The disabled-state never opens the dialog.

### 8.4 Chat-turn integration

Today the chat agent message has no commit chip — the only commit feedback
on chat turns is the `footer` string used by `git.commit.skipped` events.
Phase 1.4a-B adds a real chip.

New component `components/chat/AgentTurnCommitChip.tsx`:

```tsx
function AgentTurnCommitChip(props: {
  commit: CommitView;
  isHead: boolean;
  isProjectIdle: boolean;
  onRevertClick: (commit: CommitView) => void;
});
```

Renders inline in the agent message footer area, right of the existing
`m.footer` slot. Layout:

```
✓ abc1234 · 3 files · +12 −5    ↶
```

The trailing `↶` icon button follows the same enabled / disabled / hidden
matrix as the History-mode button (8.3).

Wiring in `app/project/[id]/page.tsx`:

- Look up the matching `CommitView` per agent message by `agentRunId === m.turnId`
  via a memoised map derived from `commitHistory.commits`.
- Pass it as a prop to `Message` (new optional `commit?: CommitView` and
  `isHeadCommit?: boolean`); `Message` renders the chip when set.
- The shared `useRevertCommit` hook handles the click.

### 8.5 History row labelling

Rollback rows render with a distinct icon (curved-arrow-back) and a subdued
title format: `↶ Revert to abc1234 — add header`. The Detail view shows
`Reverted from: abc1234 — add header` as a clickable link that selects the
original commit row.

## 9. Edge cases

| Case | Behaviour |
|---|---|
| Reverting a revert | Treated as a normal commit. Linear history grows by one more rollback row. No special UI. |
| Race: user clicks Revert, agent-run starts before broker call | Host pre-check returns `not_idle`; UI re-renders with the not-idle dialog variant. |
| Race: working tree becomes dirty between dialog open and confirm | Broker returns `dirty_tree`; UI swaps the dialog into the dirty-tree variant. |
| Stale UI: commit row deleted server-side (very rare) | Host returns 404 `unknown_sha`; toast error. |
| Protocol mismatch: old broker, new host | Host's `revertToCommit` RPC times out / returns method-unknown; UI surfaces a generic error toast. Same pattern as other 1.4a RPCs. |
| GitHub-integrated project | Revert commit lands locally only. The existing "push to GitHub" affordance pushes it like any other commit. No special path. |

## 10. Out of scope (for 1.4a-B)

- Auto-push of revert commits to GitHub.
- Reverting individual files from a historical commit (only full-state revert).
- "Squash N reverts" or amend-style cleanup of the History panel.
- Performing a revert while a run is active (deferred — depends on Pause/Resume work in 1.4d).
- USER author identity for an opportunistic dirty-tree snapshot before revert (deferred to 1.4a-C).
- Hard-reset semantics (`git reset --hard`) anywhere in the codebase. Master spec §7.3 line about reset is overridden by Decision 1.

## 11. Test plan (high level)

- **Broker unit tests** (`git-handlers.test.ts`): each branch of
  `revertToCommit` — happy path, `unknown_sha`, `is_head`, `dirty_tree`,
  `commit_failed`, body truncation, title sanitisation, shortstat parsing.
- **Host route tests** (`__tests__/route.test.ts`): pre-check matrix
  (`unknown_sha`, `is_head`, `not_idle`), broker error mapping, success
  response shape. Uses the existing test broker stub.
- **Event-sink tests**: `git.commit` event with `authorKind=ROLLBACK` writes
  a Commit row with `revertedFromSha` populated and `agentRunId=null`.
- **UI hook tests** (Vitest + RTL): state transitions across all four
  RevertState variants.
- **Smoke**: full broker + host integration via the existing host-test
  harness — create commit, revert, assert HEAD changed, assert Commit row
  exists with ROLLBACK kind.

## 12. Migration / rollout

Single Prisma migration: extend `CommitAuthorKind` enum and add
`Commit.revertedFromSha` (nullable). Safe additive change, runs alongside
the still-unapplied `20260510120000_drop_vercel_ai_runtime` migration.

No feature flag; the Revert button only appears once the matching frontend
ships and the protocol bump lands. Brokers without `revertToCommit` simply
return RPC-not-found, which the UI surfaces as a generic error.
