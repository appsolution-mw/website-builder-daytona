# Phase 1.4e — Cost-Quota + Hard-Limits — Design

Task: [T-20260511-002](../../tasks/active/T-20260511-002.md)
Date: 2026-05-11
Status: Draft (awaiting user review)

## 1. Goal

Cap runaway agent spend with two USD-denominated limits:

- A **per-day quota** aggregated per user (sum of `TokenUsage.costUsd` across all the user's projects within the current UTC day). Enforced as a pre-flight check before dispatching new agent runs. Exhausted quota disables Send until the next UTC midnight.
- A **per-turn cap** observed live by the broker via the existing `agent.usage` event stream. When a run's running cost crosses the cap, the broker emits an error event and aborts the run via the existing `AbortController` path.

A constant cost badge in the workspace topbar shows the user's current daily spend / cap with a green → yellow (≥ 80 %) → red (≥ 100 %) colour ramp. Both caps are env-var-configured for solo-builder MVP; per-user DB-stored overrides arrive with Phase 5 (multi-user / billing).

This addresses the Master-Spec §4.2 "Hard Limits (Cost Control)" risk and the master-spec Top-2 risk ("Agent costs balloon → $5+ per turn"). Token-count and tool-call caps from §4.2 are explicitly deferred (see §11 Out of scope).

## 2. Decisions (recap from brainstorming)

| # | Decision | Rationale |
|---|---|---|
| 1 | Two USD caps: **per-day quota** (host pre-flight) and **per-turn cap** (broker mid-turn observer). | USD is the user-facing unit; tokens and tool-calls are reductive proxies. Two caps cover two distinct failure modes (aggregate drift vs single-turn explosion). |
| 2 | Aggregation **per-user** across all of the user's projects. | Multi-user-ready for Phase 5 / billing. For solo-builder, equivalent to per-project. |
| 3 | Reset window = **UTC calendar day** (`createdAt >= startOfUtcDay()`). | Deterministic, no timezone drama, easy to test, easy to communicate to user. |
| 4 | Limit source = **env vars** at MVP. `DEFAULT_DAILY_USD_CAP=5.00`, `DEFAULT_PER_TURN_USD_CAP=1.00`. Per-user DB-stored overrides deferred to Phase 5. | Solo-builder tunes via `.env`; no UI settings page needed yet. |
| 5 | Pre-flight check is **optimistic**: `todaySpend < dailyCap` → allow. A single final turn can push 1× per-turn-cap over the daily cap. Acceptable bounded over-spend ($1 max). | Stricter `todaySpend + perTurnCap > dailyCap` blocks even cheap turns when near cap — too aggressive for MVP. |
| 6 | Mid-turn abort uses **existing `AbortController.abort()`** + emits **`agent.error`** with descriptive message. No new event type, no protocol bump. | The existing abort path already correctly classifies the run as CANCELLED, persists the cost so far, and surfaces the error in chat. |
| 7 | UI = **always-on badge** in workspace topbar (green / yellow / red), **Send disabled at 100 %**, no separate 80 % toast. | Constant transparency invites self-throttling; hard stop prevents silent overruns; the colour change at 80 % is the warning. |
| 8 | Aborted-mid-turn cost **still counts** toward today's quota. | API tokens were billed; reflecting it keeps the badge truthful and prevents quota-skirting via deliberate self-abort. |

## 3. Architecture

```
Pre-flight (Host)
  POST /api/projects/[id]/runs (existing dispatch route)
     ┊  requireCurrentUserFromRequest()
     ┊  todaySpend = SUM(TokenUsage.costUsd) WHERE
     ┊    project.ownerId = currentUser.id AND
     ┊    createdAt >= startOfUtcDay()
     ┊  if todaySpend >= DAILY_USD_CAP:
     ┊    return 429 { reason: "daily_quota_exceeded",
     ┊                 todaySpend, dailyCap,
     ┊                 resetsAt: <next midnight UTC ISO> }
     ┊  else: continue existing dispatch logic

Worker-pool provisioner (lib/runtime/worker-pool/runtime.ts)
  ┊  Inject BROKER_PER_TURN_USD_CAP into the sandbox env block,
  ┊  alongside BROKER_USER_NAME / BROKER_USER_EMAIL from 1.4a-C.

Broker (in sandbox)
  ┊  At boot: read process.env.BROKER_PER_TURN_USD_CAP (null = disabled).
  ┊  Active-run map gains a runningCostUsd accumulator per turn.
  ┊  In agent-run-executor, after the existing persistEvent/broadcastEvent
  ┊  pipeline forwards an `agent.usage` event:
     ┊  runningCostUsd[runId] += event.costUsd
     ┊  if perTurnCap && runningCostUsd[runId] > perTurnCap:
     ┊    persistEvent + broadcastEvent {
     ┊      type: "agent.error",
     ┊      turnId,
     ┊      message: "Cost cap reached: $X.XX (cap $Y.YY)",
     ┊    }
     ┊    activeRuns.get(runId)?.controller.abort()
  ┊  On run end: drop the runId from the map.

Host live-update path
  ┊  After persistCommitEvent / persistTokenUsage writes a TokenUsage row:
  ┊    pushed via existing per-WS-connection event channel as
  ┊    a new event type `usage.daily_total`:
  ┊      { type: "usage.daily_total",
  ┊        todaySpend: <updated sum>,
  ┊        dailyCap, resetsAt }
  ┊  Workspace hydration includes the same shape for initial render.

Browser
  ┊  Receives `usage.daily_total` events on the WS connection;
  ┊  updates the badge state.
  ┊  Workspace hydration seeds initial badge state.
  ┊  Send button gating: disabled when todaySpend >= dailyCap.
```

The pre-flight check and mid-turn abort are independent enforcement points — either can trip without the other.

## 4. Schema

**No migration required.** `TokenUsage` (Phase 1.2 / 1.2b) already records every per-turn cost with `Decimal(18, 9)` precision, indexed by `(projectId, createdAt)`. The new aggregation queries reuse that index.

## 5. Protocol

**No `@wbd/protocol` bump required.** Mid-turn cost-cap abort is emitted as the existing `agent.error` event with a descriptive message. The browser-host `usage.daily_total` event is part of the browser↔ws-proxy↔host topology, not the broker protocol, so it lives in the host's own event-type definitions, not `@wbd/protocol`.

## 6. Host — daily-quota helper

New file `lib/usage/daily-quota.ts`. Pure functions; no Prisma writes.

```ts
export interface DailyQuotaConfig {
  dailyCapUsd: number;        // 0 = disabled
  perTurnCapUsd: number;      // 0 = disabled
}

export function readDailyQuotaConfigFromEnv(): DailyQuotaConfig {
  const dailyCapUsd = parseFloat(process.env.DEFAULT_DAILY_USD_CAP ?? "0") || 0;
  const perTurnCapUsd = parseFloat(process.env.DEFAULT_PER_TURN_USD_CAP ?? "0") || 0;
  return { dailyCapUsd, perTurnCapUsd };
}

export function startOfUtcDay(now: Date = new Date()): Date {
  const d = new Date(Date.UTC(
    now.getUTCFullYear(), now.getUTCMonth(), now.getUTCDate(),
  ));
  return d;
}

export function nextUtcMidnight(now: Date = new Date()): Date {
  const d = startOfUtcDay(now);
  d.setUTCDate(d.getUTCDate() + 1);
  return d;
}

export interface DailySpendQuery {
  ownerId: string;
  now?: Date;     // injectable for tests
}

export async function getUserDailySpend(
  prisma: PrismaClient,
  q: DailySpendQuery,
): Promise<number> {
  const startOfDay = startOfUtcDay(q.now);
  const agg = await prisma.tokenUsage.aggregate({
    _sum: { costUsd: true },
    where: {
      project: { ownerId: q.ownerId },
      createdAt: { gte: startOfDay },
    },
  });
  // Prisma Decimal → number; loss of precision is acceptable at quota
  // granularity (we compare against a $5 cap, not micro-cents).
  return agg._sum.costUsd ? Number(agg._sum.costUsd) : 0;
}

export interface DailyQuotaState {
  todaySpend: number;
  dailyCap: number;          // 0 = disabled
  perTurnCap: number;        // 0 = disabled (broker side)
  resetsAt: string;          // ISO-8601, next UTC midnight
  exceeded: boolean;         // todaySpend >= dailyCap
}

export async function getDailyQuotaState(
  prisma: PrismaClient,
  ownerId: string,
  now: Date = new Date(),
): Promise<DailyQuotaState>;
```

`getDailyQuotaState` composes the spend query with the config and returns the shape consumed by the dispatch route, the usage route, the workspace hydration payload, and the WS event.

## 7. Host — pre-flight quota check on run dispatch

Locate the existing dispatch route — `app/api/projects/[id]/runs/route.ts` (POST). After the existing auth and validation, before the broker dispatch, add:

```ts
const quota = await getDailyQuotaState(prisma, currentUser.user.id);
if (quota.dailyCap > 0 && quota.exceeded) {
  return NextResponse.json(
    {
      reason: "daily_quota_exceeded",
      todaySpend: quota.todaySpend,
      dailyCap: quota.dailyCap,
      resetsAt: quota.resetsAt,
    },
    { status: 429 },
  );
}
```

The 429 is the conventional "too many requests / quota" code. The body shape is consumed by the chat UI to render a clear failure message.

Existing dispatch logic is otherwise unchanged. If `dailyCap === 0` (env not set), the check is skipped — quota is disabled.

## 9. Host — live `usage.daily_total` events

After every `TokenUsage` row insert in `lib/agent-runs/executor-client.ts` (the existing `persistTokenUsage` / equivalent path), broadcast a `usage.daily_total` event over the per-project WS channel used by `page.tsx`.

```ts
const updated = await getDailyQuotaState(prisma, project.ownerId);
broadcaster.send({
  type: "usage.daily_total",
  todaySpend: updated.todaySpend,
  dailyCap: updated.dailyCap,
  resetsAt: updated.resetsAt,
});
```

The browser already subscribes via the existing event-channel topology. No new transport.

The `usage.daily_total` event type is defined in the host-internal event types module (not `@wbd/protocol`). Browser side adds a handler in `page.tsx`'s existing event switch.

## 10. Broker — per-turn cost observer + abort

### 10.1 Config read

`container/sandbox/broker/src/index.ts` (or wherever the broker config is composed) reads:

```ts
const perTurnCapUsd = parseFloat(process.env.BROKER_PER_TURN_USD_CAP ?? "0") || 0;
```

Pass it through `startBroker({ ..., perTurnCapUsd })`. `StartBrokerOptions` gains the field.

### 10.2 Wiring in `agent-run-executor.ts`

The existing `executeAgentRun` already streams `agent.usage` events to `persistEvent` and `broadcastEvent`. Add a counter and a cap check.

Sketch (touch only the event handler closure inside `executeAgentRun`):

```ts
let runningCostUsd = 0;

// Inside the onEvent callback that handles incoming provider events:
if (event.type === "agent.usage") {
  runningCostUsd += event.costUsd ?? 0;
  if (perTurnCapUsd > 0 && runningCostUsd > perTurnCapUsd) {
    const errEvt: BrokerToHost = {
      type: "agent.error",
      turnId: input.runId,
      message: `Cost cap reached: $${runningCostUsd.toFixed(2)} (cap $${perTurnCapUsd.toFixed(2)})`,
    };
    await input.persistEvent(errEvt);
    input.broadcastEvent(errEvt);
    // Surface the cap-exceeded condition to the run-execute handler in
    // ws-server.ts, which already owns the AbortController via activeRuns
    // and tears the run down cleanly. We throw a typed error rather than
    // plumbing a new abort path.
    throw new CostCapExceededError(runningCostUsd, perTurnCapUsd);
  }
}
```

Add the exception class:

```ts
export class CostCapExceededError extends Error {
  constructor(public observed: number, public cap: number) {
    super(`Cost cap reached: $${observed.toFixed(2)} (cap $${cap.toFixed(2)})`);
    this.name = "CostCapExceededError";
  }
}
```

The `executeMatch` HTTP handler in `ws-server.ts` already wraps `executeAgentRun` in try/catch for stream completion. Catching `CostCapExceededError` there logs cleanly and ends the run; the persistEvent+broadcastEvent of the `agent.error` already happened before the throw, so the user sees the message immediately and the run record receives the terminal error event.

### 10.3 Test plan for the broker piece

Mock a provider whose `onEvent` callback emits three `agent.usage` events with `costUsd: 0.4`, `0.4`, `0.4`. With `perTurnCapUsd: 1.0`, the second event keeps `runningCostUsd = 0.8` (no abort), and the third pushes to `1.2 > 1.0` → emit `agent.error` with a "Cost cap reached" message and throw `CostCapExceededError`. Verify the test harness sees both events (`agent.error` followed by no further events) and the thrown exception.

## 11. Browser — `DailyCostBadge` + Send gating + cap-error chat marker

### 11.1 `<DailyCostBadge />`

```tsx
// components/workspace/DailyCostBadge.tsx
export function DailyCostBadge(props: {
  todaySpend: number;
  dailyCap: number;
  resetsAt: string;
}) {
  if (props.dailyCap <= 0) return null;        // quota disabled
  const ratio = props.todaySpend / props.dailyCap;
  const color =
    ratio >= 1 ? "text-rose-600 border-rose-600/40 bg-rose-500/10" :
    ratio >= 0.8 ? "text-amber-600 border-amber-600/40 bg-amber-500/10" :
    "text-muted-foreground border-border";
  const title = `$${props.todaySpend.toFixed(2)} of $${props.dailyCap.toFixed(2)} today; resets ${new Date(props.resetsAt).toLocaleString()}`;
  return (
    <span title={title} className={`inline-flex items-center gap-1 rounded-md border px-2 py-0.5 text-xs font-medium ${color}`}>
      ${props.todaySpend.toFixed(2)} / ${props.dailyCap.toFixed(2)}
    </span>
  );
}
```

Mounted in the workspace topbar in `app/project/[id]/page.tsx` next to the existing Code/History switcher.

### 11.2 Send-button gating

The chat input's Send button (currently in `app/project/[id]/page.tsx`'s chat panel) gains an additional disabled condition: `dailyQuota.exceeded === true`. Tooltip: `"Daily $X.XX quota reached. Resets at <time>."`.

### 11.3 Cap-error chat marker

The existing `agent.error` chat rendering already exists. For cost-cap errors specifically, detect via a substring match on `"Cost cap reached"` and apply an alternate icon / colour. Minor visual tweak only.

## 12. Edge cases

| Case | Behaviour |
|---|---|
| `DEFAULT_DAILY_USD_CAP` env unset or `0` | Pre-flight check skipped; badge hidden; Send never gated by quota. |
| `DEFAULT_PER_TURN_USD_CAP` env unset, `0`, or NaN | Broker's per-turn observer disabled; runs never aborted for cost. |
| Concurrent runs same user across two projects | Pre-flight at dispatch sees combined cost; per-turn cap is per-run-id local. |
| Turn that starts under quota but pushes over (e.g. todaySpend $4.80, turn costs $0.70 → $5.50) | Allowed by Decision 5; one bounded over-spend per day. Next turn blocked. |
| UTC midnight crossing during a turn | Pre-flight uses `startOfUtcDay` at dispatch time. Cost lands when `TokenUsage` row inserts (run-end). One run might "carry over" cost into the new day; the badge resets correctly at midnight regardless. |
| `agent.usage` event with `costUsd === 0` or missing | Treated as 0; runningCostUsd unchanged; no abort. |
| Provider over-reports cost (broker observes $1.5 but real cost was $0.5) | Aborted regardless. The trusted-cost-source is the provider event; if the provider misreports, the cap is the user's only protection. |
| Aborted run's partial cost | TokenUsage row(s) already persisted before abort; counts toward today's quota. |

## 13. Out of scope (for 1.4e)

- Per-user DB-stored caps (User.dailyCapUsd, User.perTurnCapUsd). Phase 5 (multi-user / billing) adds this.
- Tool-call counter / max-iteration cap.
- Token-level caps via SDK `max_tokens` params.
- Per-model differential caps.
- ML-based cost prediction / pre-flight estimation.
- Monthly aggregates or billing-cycle alignment.
- Pause / soft-stop near cap (graceful wrap-up instead of hard abort).
- Different caps per Better-Auth user tier.
- Multi-currency or non-USD display.

## 15. Migration / rollout

No migration, no protocol bump. Env vars:

```dotenv
# .env (host) — both are optional; 0 / unset disables the cap.
DEFAULT_DAILY_USD_CAP=5.00
DEFAULT_PER_TURN_USD_CAP=1.00
```

`DEFAULT_PER_TURN_USD_CAP` is also forwarded to the broker as `BROKER_PER_TURN_USD_CAP` by the worker-pool provisioner. The broker's old behaviour (no cap) is preserved when the env var is missing — sandboxes booting against a not-yet-redeployed host won't crash.

Deploy order:
1. Update `.env` on the host with both caps.
2. Deploy host code.
3. Worker images and sandbox images get the broker change with the next worker rotation; until they do, only the host-side daily quota is enforced. Per-turn cap activates as workers refresh.

No DB migration to deploy. The Phase 1.4a-C migration is the only outstanding DB change.
