**Date:** 2026-04-28
**Status:** Completed and merged to `main` on 2026-04-28
**Project root:** `/Volumes/Extern/Projekte/website-builder-daytona`
**Parent spec:** `docs/superpowers/specs/2026-04-27-phase-h.1-hetzner-runtime-design.md`

---

## 1. Goal

Replace the Daytona-style ad-hoc container bring-up with a CI-built sandbox image and a per-Worker `worker-agent` that exposes a high-level RPC for sandbox lifecycle. After H.1b, with `RUNTIME_MODE=worker-pool-local` set, the UI spawns a real Docker sandbox from a pre-built image through a locally-running worker-agent — end-to-end, observable, debuggable, with the same three-panel UX as today's Daytona path.

H.1b deliberately stops short of Hetzner-side automation: no real `HetznerProvisioner`, no Tailscale, no Caddy, no wildcard-TLS. Those are H.1c and H.1d. The stake here is **the image and the agent**, validated against a Local-Docker target that exercises every code path the Hetzner build will reuse later.

**UX target:** unchanged from today's Daytona-fake mode — sub-10 s spawn, instant HMR, broker WS round-trip < 50 ms locally. The pre-built image cuts heavy installs out of the boot path; spawn time drops from ~2–3 min (Daytona) to ~5–10 s.

**Non-goals are documented in §9.**

## 2. Chosen Options Summary

| Decision | Chosen Option |
|---|---|
| H.1b-Scope | **Bausteine + `WorkerPoolRuntime`-MVP** gegen Local-Docker (UI-Live-E2E lauffähig, kein Caddy/Tailscale) |
| worker-agent API-Style | **High-Level-RPC** — `POST /sandboxes`, `DELETE /sandboxes/:id`, `GET /sandboxes/:id`, `GET /sandboxes`, `GET /health`. Outbound Heartbeat. |
| Auth Host ↔ worker-agent | HMAC-shared-secret. Body+Method+Path+Timestamp signed, ±5 min replay-window. |
| User-Code-Bringup | **Image-Snapshot** via `COPY container/sandbox/project-template/`. Kein Volume, kein Git-Pull. Persistenz = Phase 1.4. |
| worker-agent Form | **Eigener Docker-Container**, mountet `/var/run/docker.sock`. Identische Deployment-Form lokal und auf Worker-VM. |
| Image-Registry | `ghcr.io/appsolution-mw`, privat. Worker-VM auth'd via fine-grained PAT mit `read:packages`-Scope. |
| Runtime-Klasse + Mode-Naming | **`WorkerPoolRuntime`** (multi-provider, ersetzt das `HetznerRuntime` aus der Parent-Spec). Modes: `worker-pool-local` (H.1b) / `worker-pool-hetzner` (H.1c). |
| Image-Tag-Strategie | `:<sha>` (immutable) + `:latest` auf main. Production referenziert `:<sha>`. |

The `WorkerPoolRuntime` rename overrides the parent spec's `HetznerRuntime` — that name was misleading once Vultr/Linode/AWS provisioners join the pool. The class is provider-agnostic; only `WorkerProvisioner` implementations are provider-specific.

**Three layers in H.1b:**

1. **`WorkerPoolRuntime`** on the host — picks worker via `Scheduler`, calls `AgentClient`, persists `WorkerSandbox` + `SandboxToken` rows.
2. **worker-agent container** — HMAC gate, dockerode-driven container ops, picks free ports, outbound heartbeat.
3. **Sandbox container** — pre-built image with broker, claude-code CLI, codex CLI, project-template, `.claude/agents/`. Identical to today's Daytona container content, just baked instead of installed at boot.

**Three critical flows** (details in §5):

- **Spawn:** UI → API → `WorkerPoolRuntime.spawn` → `Scheduler.pickWorker` → if none, `Provisioner.provision` (FakeProvisioner registers a Worker pointing at `127.0.0.1`) → `AgentClient.createSandbox` → worker-agent runs `docker.containerCreate` + `start`, returns ports.
- **Heartbeat:** worker-agent → POST `http://host.docker.internal:3000/api/internal/workers/:id/heartbeat` every 10 s.
- **Destroy:** UI → API → `WorkerPoolRuntime.destroy` → `AgentClient.destroySandbox` → worker-agent → `docker stop` + `rm`.

**Unchanged from H.1a:**

- `ws-proxy/` — reads `project.brokerUrl` from DB; URL scheme is transparent.
- `container/broker/` source code — moves into the image build context but otherwise unchanged.
- `lib/runtime/index.ts` factory — gets a new case `worker-pool-local`. The `daytona-cloud` and `daytona-fake` paths are not touched.
- Prisma models `Worker` / `WorkerSandbox` / `SandboxToken` from H.1a — H.1b is the first phase that populates them with real data.

**Removed vs. today's Daytona path** (only inside `worker-pool-*` mode; Daytona path keeps its old behaviour):

- Tarball-clone of the builder codebase at container start.
- Runtime install of `claude-code`, broker, monorepo deps.
- `@daytona/sdk` calls.

### 4.4 worker-agent API

```
POST   /sandboxes
  Body: {
    sandboxId: string,        // host-supplied uuid
    projectId: string,
    image: string,            // e.g. "ghcr.io/.../sandbox:latest"
    env: Record<string,string>,
    brokerToken: string,      // forwarded as BROKER_TOKEN env to container
  }
  Response 201: {
    sandboxId, containerId, brokerPort, previewPort, status: 'spawning'
  }
  Errors: 400 (bad input), 401 (HMAC fail), 422 (image-not-found), 503 (port-exhausted)

DELETE /sandboxes/:sandboxId
  Response 204 (also when sandbox unknown — idempotent)

GET    /sandboxes/:sandboxId
  Response 200: {
    sandboxId, containerId, brokerPort, previewPort,
    status: 'spawning'|'running'|'stopped'|'gone'
  }

GET    /sandboxes
  Response 200: Array<above>          # for reconciliation

GET    /health
  Response 200: { ok, dockerVersion, uptime, count }   # NOT HMAC-protected
```

All routes except `/health` require:

```
X-Timestamp: <ISO-8601, ±5 min window>
X-Signature: hmac-sha256("${timestamp}.${method}.${path}.${body}", WORKER_AGENT_HMAC_SECRET)
```

Replay protection = timestamp window only (no nonce store) — acceptable in H.1b given Tailscale-only reachability in production and loopback in dev. A nonce store is on the H.2 hardening list if we ever expose the agent off-Tailnet.

### 4.5 WorkerPoolRuntime spawn sketch

```ts
async spawn(args: SpawnArgs): Promise<SandboxInfo> {
  let worker = await scheduler.pickWorker({});
  if (!worker) {
    worker = await provisioner.provision({
      region: 'local',          // FakeProvisioner ignores
      size: 'local',
      capacity: 8,
    });
  }

  const agent = agentClientFor(worker);   // baseUrl = http://${worker.tailscaleIp}:4500
  const sandboxId = cuid();
  const brokerToken = randomBytes(32).toString('hex');

  const sandbox = await agent.createSandbox({
    sandboxId,
    projectId: args.projectId,
    image: process.env.SANDBOX_IMAGE!,
    brokerToken,
    env: {
      PROJECT_ID: args.projectId,
      BROKER_TOKEN: brokerToken,
      ANTHROPIC_API_KEY: process.env.ANTHROPIC_API_KEY ?? '',
      CLAUDE_CODE_OAUTH_TOKEN: process.env.CLAUDE_CODE_OAUTH_TOKEN ?? '',
      AGENT_RUNTIME: process.env.AGENT_RUNTIME ?? 'claude-code',
    },
  });

  await prisma.workerSandbox.create({
    data: {
      id: sandboxId,
      workerId: worker.id,
      projectId: args.projectId,
      containerId: sandbox.containerId,
      brokerPort: sandbox.brokerPort,
      previewPort: sandbox.previewPort,
      status: 'SPAWNING',
    },
  });
  await prisma.sandboxToken.create({
    data: {
      sandboxId,
      token: brokerToken,
      expiresAt: new Date(Date.now() + 24 * 3600 * 1000),
    },
  });

  return {
    sandboxId,
    brokerUrl: `ws://${worker.tailscaleIp}:${sandbox.brokerPort}/?token=${brokerToken}`,
    brokerPreviewToken: brokerToken,
    previewUrl: `http://${worker.tailscaleIp}:${sandbox.previewPort}`,
  };
}
```

In H.1b `worker.tailscaleIp` holds `127.0.0.1` (set by `register-local-worker.ts`). Both consumers — `agent-client` (host process, calls `http://127.0.0.1:4500`) and `ws-proxy` (host process, connects to `ws://127.0.0.1:<brokerPort>`) — run on the host and reach the agent and sandbox containers through Docker's published-port loopback.

The agent's outbound heartbeat uses a separate `HOST_URL=http://host.docker.internal:3000` env var (§6.2) — that one *is* container-internal and needs the Docker DNS name to reach the host.

In H.1c the same code path runs against a real Tailscale IP without changes here — only `provisioner.provision` and the worker-agent's runtime location differ.

## 5. Data Flows

### 5.1 Spawn

```
1. UI "Open Project" → POST /api/projects/:id/start
2. createRuntime() → WorkerPoolRuntime  (RUNTIME_MODE=worker-pool-local)
3. runtime.spawn({projectId, ...})
   ├─ scheduler.pickWorker({})
   │  ├─ existing READY worker w/ capacity → return it
   │  └─ none                              → fall through
   ├─ provisioner.provision({region:'local', size:'local', capacity:8})
   │  └─ FakeProvisioner inserts Worker(provider='fake', tailscaleIp='127.0.0.1')
   ├─ const brokerToken = randomBytes(32).hex()
   ├─ agentClient.createSandbox({sandboxId, projectId, image, env, brokerToken})
   │  └─ POST http://127.0.0.1:4500/sandboxes  (HMAC-signed)
   │     └─ worker-agent:
   │        - pick brokerPort/previewPort from 30000-39999 (random + retry on EADDRINUSE)
   │        - dockerode.createContainer({Image, Env, HostConfig.PortBindings, Labels})
   │        - container.start()
   │        - return {containerId, brokerPort, previewPort, status:'spawning'}
   ├─ DB:
   │   INSERT WorkerSandbox(... status='SPAWNING')
   │   INSERT SandboxToken(sandboxId, token, expiresAt=+24h)
   │   UPDATE Project SET brokerUrl, previewUrl
   └─ return SandboxInfo

4. UI iframe loads previewUrl, ws-proxy connects to brokerUrl

Spawn time (Local-Mode, image already pulled):
  - schedule + DB:           ~50 ms
  - dockerode.create+start:  ~500 ms
  - container entrypoint:    ~3-8 s (next dev startup, no installs)
  → total ~4-9 s
```

In H.1b the `brokerUrl` `?token=<brokerToken>` URL parameter is the only authentication on the broker — unchanged from today's Daytona path. H.1d adds an `X-Sandbox-Token` header check at Caddy. The host already populates `SandboxToken` rows so H.1d's Caddy validator has data to query.

### 5.2 Heartbeat

```
worker-agent (every 10s):
  POST http://host.docker.internal:3000/api/internal/workers/<id>/heartbeat
  Body: { dockerVersion, runningSandboxes, uptime }
  Headers: X-Timestamp, X-Signature

Host /api/internal/workers/[id]/heartbeat:
  - HMAC verify (rejects 401 if bad)
  - SELECT Worker WHERE id=:id
    - if not found       → return 404 (do NOT auto-create — workers are only created by Provisioner)
    - if DECOMMISSIONED  → return 410 (agent should self-stop on this)
    - else               → UPDATE Worker SET lastHeartbeatAt=now(), status='READY' and return 204

Stale-detection (cron job, every 60s):
  UPDATE Worker SET status='OFFLINE'
   WHERE lastHeartbeatAt < now() - interval '30 seconds'
     AND status = 'READY'
```

The cron is a tiny Next.js route hit by an OS-level cron (or `node-cron` inside the host process) — implementation detail for the plan. In H.1b only one local worker exists, so the cron is mostly there to be tested before H.1c puts it under real load.

### 5.3 Destroy

```
1. UI "Close Project" → POST /api/projects/:id/stop
2. runtime.destroy(sandboxId)
   ├─ SELECT WorkerSandbox WHERE projectId=...
   ├─ agentClient.destroySandbox(sandboxId)
   │  └─ DELETE http://.../sandboxes/<id>  (HMAC-signed)
   │     └─ worker-agent: docker.kill + docker.rm
   ├─ UPDATE WorkerSandbox SET status='DESTROYED'
   ├─ DELETE SandboxToken WHERE sandboxId=...
   └─ UPDATE Project SET brokerUrl=null, previewUrl=null, status='STOPPED'
```

Destroy is idempotent on the agent side: a `DELETE /sandboxes/:id` for an unknown id returns 204.

### 5.4 Failure Matrix

| Scenario | Handling in H.1b |
|---|---|
| Image missing locally | `docker.containerCreate` raises `ImageNotFound`; agent returns 422 `image-not-found`. Runtime surfaces a clear error in the UI. Operator: `docker pull <SANDBOX_IMAGE>`. Mitigation: agent does `docker pull` at startup and only signals `/health=ok` afterwards (§9 open item 3). |
| Port range exhausted | Port-picker retries 10×, then agent returns 503. Runtime surfaces. |
| HMAC mismatch | Agent returns 401. Runtime logs + throws. Common cause: clock skew → check ±5 min window. |
| worker-agent down | `agent.createSandbox` times out (10 s). Scheduler re-marks Worker `OFFLINE` locally; subsequent calls fall through to `provisioner.provision` which re-registers the local worker. |
| Container crashes immediately after `start` | `agent.getStatus` returns `gone` within 3 s of create. Runtime makes one retry, then surfaces error. (Auto-respawn → H.1c.) |
| Race: double spawn for same project | Unique `WorkerSandbox.projectId` constraint catches it. Second spawn returns the existing sandbox. |
| `host.docker.internal` not resolvable on Linux without Docker Desktop | `start-local-worker-agent.sh` adds `--add-host=host.docker.internal:host-gateway` — Linux compatible. |

### 6.1 GitHub Actions

`.github/workflows/sandbox-image.yml`:

```yaml
name: Build sandbox image
on:
  push:
    branches: [main]
    paths:
      - 'container/sandbox/**'

      - 'container/sandbox/**'

jobs:
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write
    steps:
      - uses: actions/checkout@v4
      - uses: docker/setup-buildx-action@v3
      - if: github.event_name != 'pull_request'
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}
      - uses: docker/build-push-action@v5
        with:
          context: .
          file: container/sandbox/Dockerfile
          push: ${{ github.event_name != 'pull_request' }}
          tags: |
            ghcr.io/appsolution-mw/sandbox:${{ github.sha }}
            ghcr.io/appsolution-mw/sandbox:latest
          cache-from: type=gha
          cache-to: type=gha,mode=max
```

`.github/workflows/worker-agent.yml` is structurally identical, building `worker-agent/Dockerfile` to `ghcr.io/appsolution-mw/worker-agent:<sha>` + `:latest`.

**Tag strategy:**

- `:<sha>` is immutable. `.env` references `:<sha>` for reproducible spawns.
- `:latest` for local dev convenience.
- Production cutover (H.1c) deploys with `:<sha>`.

**Auth setup:**

- Both ghcr.io repos private. `GITHUB_TOKEN` in workflow has `packages: write`.
- Worker-VM (H.1c) auths via fine-grained PAT with `read:packages` scope.
- Local dev: one-time `docker login ghcr.io` with the developer's PAT.

```bash
: "${WORKER_AGENT_HMAC_SECRET:?set in .env}"

docker rm -f wbd-worker-agent 2>/dev/null || true
docker run -d --name wbd-worker-agent \
  --add-host=host.docker.internal:host-gateway \
  -p 4500:4500 \
  -v /var/run/docker.sock:/var/run/docker.sock \
  -e HOST_URL=http://host.docker.internal:3000 \
  -e HMAC_SECRET="${WORKER_AGENT_HMAC_SECRET}" \
  -e SANDBOX_IMAGE="${SANDBOX_IMAGE:-wbd/sandbox:dev}" \
  -e WORKER_ID="${WORKER_ID:-local-1}" \
  ghcr.io/appsolution-mw/worker-agent:latest

until curl -sf http://127.0.0.1:4500/health > /dev/null; do sleep 0.5; done

pnpm tsx scripts/register-local-worker.ts
```

`scripts/register-local-worker.ts` upserts a `Worker` row pointing at `127.0.0.1`. The `id` is set explicitly to match the agent's `WORKER_ID` env var so the heartbeat path `/api/internal/workers/<id>/heartbeat` resolves:

```ts
import { prisma } from '../lib/db/client';

const id = process.env.WORKER_ID ?? 'local-1';

await prisma.worker.upsert({
  where: { id },
  create: {
    id,
    tailscaleHostname: 'local-dev',
    tailscaleIp: '127.0.0.1',
    provider: 'fake',
    providerVmId: 'local-1',
    region: 'local',
    capacity: 8,
    status: 'READY',
  },
  update: { status: 'READY', lastHeartbeatAt: new Date() },
});
```

(Prisma's `@default(cuid())` only fires when `id` is not provided, so passing one is safe.)

A root `pnpm dev:worker` script wraps `start-local-worker-agent.sh` so local E2E setup is a single command.

### 6.3 New env vars

In `.env.example`:

```
# === Sandbox runtime (worker-pool mode) ===
RUNTIME_MODE=worker-pool-local                  # was: daytona-fake for local dev
SANDBOX_IMAGE=ghcr.io/appsolution-mw/sandbox:latest
WORKER_AGENT_URL=http://127.0.0.1:4500          # local-dev override; production reads from DB
WORKER_AGENT_HMAC_SECRET=                       # 32-byte random hex; required by both agent and host
WORKER_ID=local-1                               # only set in local-dev for the agent
```

In production (H.1c) the runtime resolves `WORKER_AGENT_URL` from `Worker.tailscaleIp:4500`. The env var stays as a local-dev override for standalone agent-client tests.

### 9.3 Open items for the plan-writing phase (non-blocking)

1. **Worker-agent HTTP library:** Recommendation **Fastify** — small surface, good TS typings, similar shape to broker's WS server. The plan locks this in.
2. **`dockerode` vs. `docker` CLI shell-out:** Recommendation **dockerode** — typed errors, no shell parsing.
3. **Pre-pull strategy:** worker-agent should `docker pull <SANDBOX_IMAGE>` at startup and only expose `/health=ok` after success — eliminates the "ImageNotFound on first spawn" race.
4. **HMAC-secret bootstrap in local dev:** `pnpm dev:worker` script generates `WORKER_AGENT_HMAC_SECRET` into `.env.local` if missing — saves friction.
5. **Image-size audit:** measure after first build; if > 1 GB, esbuild-bundle the broker before adding to the image.

## 10. Validation Criterion for End-of-Phase

H.1b is complete when, on a developer machine:

1. `docker login ghcr.io` is the only out-of-band setup step.
2. `pnpm dev:worker && pnpm dev` starts everything; `RUNTIME_MODE=worker-pool-local` is set in `.env`.
3. Clicking "Open Project" in the UI spawns a real Docker sandbox in ≤10 s.
4. The preview iframe loads and shows the project-template's Next.js page.
5. A chat turn ("change the heading to 'Hi'") edits the file inside the container, HMR fires in the iframe, broker WS delivers events without errors.
6. "Close Project" removes the container; `docker ps` confirms.
7. The worker-agent container survives a `docker restart wbd-worker-agent` and continues heartbeating without manual host intervention.
8. `pnpm test:host` passes, including the new agent-client / runtime / heartbeat suites; `worker-agent`-package tests pass.
9. CI builds + pushes both images to ghcr.io on a merged PR to main.

When all nine pass under live execution (not just unit tests), the phase ships.
