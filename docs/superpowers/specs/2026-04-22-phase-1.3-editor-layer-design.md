# Phase 1.3 — Editor Layer (Monaco + File Tree + Soft Lock) Design Spec

**Date:** 2026-04-22
**Status:** Design, awaiting user review
**Predecessor:** Phase 1.2 "Single-Agent Claude Code" — merged to main, tagged `phase-1.2-claude-code-agent`
**Successors:** Phase 1.4 (git commits per turn, rollback UI, chat history persistence), Phase 1.2b (multi-agent team — independent of 1.3)

---

## 1. Goal

Give the user a working file tree and a read/write Monaco editor next to the chat and preview, plus live file-change surfacing driven by `chokidar` inside the container. The user can open any project file, watch the agent edit it live, and save their own edits between agent turns. A soft lock makes the editor read-only while an agent turn is in flight; the broker refuses `file.write` during a turn as the authoritative enforcement point.

The result: "User opens `app/page.tsx` in Monaco → sees it highlighted → sends a prompt → editor goes read-only → file gets a badge in the tree when the agent writes it → content live-refreshes → editor becomes writable again → user tweaks a line → Ctrl+S writes it to disk → HMR updates the preview."

This phase stays in-memory only. Git commit plumbing for both agent turns and user edits is Phase 1.4.

## 2. Chosen Options Summary (from brainstorming)

| Decision | Choice |
|----------|--------|
| Scope | Full read/write + soft lock; no queued-save-after-done (Option C) |
| Layout | Chat \| Tab-switcher (Code / Preview) on the right (Option B) |
| File-change surface | Tree badge only + markdown rendering in agent chat (Option A) |
| Transport | Single WS pipe for list/read/write/changed (Spec §6 as-written, Approach 1) |
| Tree virtualisation | None (plain DOM — templates stay well under 500 files) |
| Monaco integration | `@monaco-editor/react` lazy-loaded via `next/dynamic` (SSR false) |
| Soft-lock signal source | Existing `turnInFlight` in UI + `currentTurn` in broker (no new broker state) |
| File-size cap | 1 MB, enforced in broker for both read and write |
| Binary detection | >10 % NULL bytes in first 8 KB → `error: "binary"` on read |
| Ignore list | `node_modules`, `.next`, `.git`, `dist`, `.agent-artifacts`, `**/*.log` |

## 3. Architecture Cut

```
┌─────────── BROWSER ───────────┐    ┌────────── CONTAINER ───────────┐
│                               │    │                                │
│ ProjectWorkspace (client)     │    │ broker :4000                   │
│  ├─ Chat panel                │    │  ├─ ws-server (dispatch)       │
│  ├─ RightPane                 │    │  ├─ claude-runner (unchanged)  │
│  │   ├─ Tab: Code             │    │  ├─ fs-tracker (NEW)           │
│  │   │   ├─ FileTree          │    │  │   └─ chokidar on            │
│  │   │   └─ CodeEditor        │    │  │      /workspace/project     │
│  │   └─ Tab: Preview (iframe) │◀──▶│  └─ fs-handlers (NEW)          │
│  └─ WS client                 │ WS │      list / read / write       │
│                               │    │                                │
└───────────────────────────────┘    └────────────────────────────────┘
             │                                       ▲
             │            ws-proxy (pass-through)    │
             └───────────────────────────────────────┘
```

**Broker** gains two modules:

- `container/broker/src/fs-tracker.ts` — owns a single `chokidar.watch('/workspace/project')`. Collects every `add` into an in-memory `Set<string>` until chokidar emits `ready`; suppresses WS events during that phase. After `ready`, every `add` / `change` / `unlink` produces one `file.changed` event broadcast to all connected sockets. Resolves the `source: "agent" | "external"` heuristic via an injected `getCurrentTurn()` callback.
- `container/broker/src/fs-handlers.ts` — pure functions `handleFileList(tracker)`, `handleFileRead(tracker, path)`, `handleFileWrite(tracker, path, content)`. All path inputs are resolved against `/workspace/project` and rejected if they escape it. `handleFileWrite` enforces the soft lock via an injected `isLocked()` callback; read/list are stateless.

`ws-server.ts` today tracks `currentTurn` per-connection. Phase 1.3 adds a **broker-level active-turn counter** alongside it: every `agent.prompt` that starts a turn increments it, every `finally` decrements. `isLocked()` and the `source: "agent"` heuristic in `fs-tracker` both read the broker-level counter, not per-connection state. This preserves the main-spec rule "only one code-mutating agent per container" even if two browser tabs are open against the same project: a `file.write` from tab B while tab A's turn is running returns `reason: "locked"` just like tab B trying to start a second prompt returns `agent.error`. The per-connection `currentTurn` is still used to route `agent.abort` to the right turn.

**Host UI** gets new components under `components/workspace/` and `components/chat/`:

- `RightPane.tsx` — tab bar + switch between Code and Preview (local `useState<'code' | 'preview'>`, default `'preview'`).
- `FileTree.tsx` — receives a flat path array + selection callback; renders indented tree via a small `buildTreeFromPaths` helper.
- `CodeEditor.tsx` — Monaco wrapper. Props: `content`, `language`, `readOnly`, `onSave(content)`. Lazy-loaded via `next/dynamic` with `ssr: false`.
- `components/chat/Message.tsx` — extracted chat bubble, agent messages rendered through `react-markdown` + `remark-gfm` (no `rehype-raw`, no HTML passthrough).

The existing `ProjectWorkspace` page stays the single orchestrator: holds the WS, turn state, file-path set, selected-path, open-file content, and passes everything down as props.

**ws-proxy**: no changes. It remains a blind WS pass-through.

**Prisma / DB**: no changes in Phase 1.3. Editor state is ephemeral memory in the container.

**Protocol package**: new message types, version bump to `1.3.0`.

## 4. Protocol Additions (v1.3.0)

All new messages flow over the existing WS pipe; `requestId` is a UUID generated by the browser per request and echoed in the response.

```ts
// Browser/Host → Broker
| { type: "file.list"; requestId: string }
| { type: "file.read"; requestId: string; path: string }
| { type: "file.write"; requestId: string; path: string; content: string }

// Broker → Browser/Host
| {
    type: "file.list.result";
    requestId: string;
    paths: string[]; // relative to /workspace/project, POSIX separators, sorted
  }
| {
    type: "file.content";
    requestId: string;
    path: string;
    content?: string;
    error?: "not_found" | "too_large" | "invalid_path" | "io_error" | "binary";
  }
| {
    type: "file.write.result";
    requestId: string;
    path: string;
    ok: boolean;
    reason?: "locked" | "too_large" | "invalid_path" | "io_error";
  }
| {
    type: "file.changed";
    path: string;
    event: "add" | "change" | "unlink";
    source: "agent" | "external";
  }
```

**Semantics:**

- `file.list` returns a **flat** path array, not a nested tree. The UI builds hierarchy from slashes client-side. Two serialisations (flat + nested) are not worth maintaining.
- `file.read` limit: 1 MB. Larger → `error: "too_large"` with no `content` field.
- Binary detection: reads the first 8 KB; if >10 % of bytes are `0x00`, responds with `error: "binary"` (Monaco does not receive binary garbage).
- `file.write` limit: 1 MB. Larger → `ok: false, reason: "too_large"`.
- `file.write` is **atomic**: `fs.writeFile(tmpPath)` → `fs.rename(tmpPath, finalPath)`. Produces a single chokidar `change` event, not `add` + `change`.
- `file.write` creates parent directories as needed (`mkdir -p` semantics). Agents occasionally create nested paths.
- `file.changed` is broadcast to all connected sockets on this broker (one tab = one socket, but two tabs both see updates).
- `source` is best-effort: the broker sets it to `"agent"` iff the broker-level active-turn counter is >0 at the moment of the chokidar callback, else `"external"`. Tiny race at turn boundary is acceptable — the UI only uses it for light tinting, no hard logic depends on it.
- `PROTOCOL_VERSION` bumps from `1.2.0` → `1.3.0`.

## 5. UI Behaviour

### 5.1 Layout

```
┌────────────────────── ProjectWorkspace ─────────────────────────┐
│ Header (project name · WS status)                                │
├─────────────┬────────────────────────────────────────────────────┤
│             │ Tab-Bar: [ Code ] [ Preview ]                     │
│ Chat        ├────────────────────────────────────────────────────┤
│ (resizable, │                                                    │
│ 22-45 %,    │  Code tab:                Preview tab:              │
│ default 28) │  ┌──────┬──────────┐      ┌───────────────────┐    │
│             │  │Tree  │ Monaco   │      │  <iframe>         │    │
│             │  │240px │          │      │                   │    │
│             │  └──────┴──────────┘      └───────────────────┘    │
└─────────────┴────────────────────────────────────────────────────┘
```

The existing chat-resize handle stays unchanged. Tree width is fixed at 240 px; no internal second resizer in Phase 1.3. Default tab is `preview` — the "app is alive" impression matters at first load; clicking a file in the tree auto-switches to `code`.

### 5.2 File-Tree Interaction

- On `ws.onopen`, browser sends `file.list`. Tree shows "loading…" until the response arrives.
- If the initial `file.list` fails (broker not ready, WS error), tree remains empty. No retry loop in 1.3 — user reloads the page. Phase 1.4 may add a refresh button.
- Clicking a file sends `file.read` and immediately switches the tab to `code`. Monaco shows a spinner until `file.content` arrives.
- `file.changed` updates the tree's path set (for `add`/`unlink`) and marks the path with a small dot badge for 3 s (CSS animation) regardless of `source`.
- If the user has a file open and a `file.changed` arrives for it **while their editor is not dirty**, the UI re-issues `file.read` and replaces Monaco content with the new version. This is how "watching the agent edit live" works.

### 5.3 Soft Lock

- While `turnInFlight !== null` in the UI: Monaco `readOnly={true}`, save button disabled, a subtle banner "Agent editing… save disabled" appears above the editor.
- Local keystrokes during a locked turn are blocked by Monaco's `readOnly` — no queue, no dirty-buffer, no flush-after-done.
- When `turnInFlight` becomes `null`, `readOnly` flips back. If `file.changed` already refreshed the content, Monaco shows the new version.
- Broker-side enforcement: `handleFileWrite` rejects with `reason: "locked"` whenever the broker-level active-turn counter is >0 (regardless of which connection started the turn). The UI check is a UX nicety; the broker is the source of truth.

### 5.4 User Save (outside of a turn)

- Ctrl/Cmd-S or clicking Save sends `file.write` with the current Monaco buffer.
- On `file.write.result { ok: true }` the UI shows a brief green "saved" indicator and clears the dirty flag.
- On `ok: false` it shows a red banner with `reason`; the editor stays dirty so the user can retry.
- **User writes are NOT committed in Phase 1.3.** Content reaches the container's filesystem and HMR fires, but no `git commit` is produced. Commit-per-turn (including pre-turn user edits) is Phase 1.4.

### 5.5 Markdown Rendering

- `react-markdown` + `remark-gfm`, strict safe defaults: no `rehype-raw`, no HTML passthrough.
- Applied to `agent` messages only; user bubbles remain plain text (no surprises if a user types `<script>` in a prompt).
- Tool-use labels stay plain italic as they are today.

### 5.6 Explicitly Not Included

- Create / rename / delete file from the UI
- Multi-file editor tabs
- Diff viewer between file versions
- Search-in-files
- File-type icons or syntax-differentiated themes
- "Discard my changes" merge hint when `file.changed` hits a locally dirty file

## 6. Error Handling & Edge Cases

### 6.1 Broker Side

| Scenario | Behaviour |
|----------|-----------|
| `file.read` path contains `..` or resolves outside `/workspace/project` | `file.content { error: "invalid_path" }` |
| `file.read` target does not exist | `file.content { error: "not_found" }` |
| `file.read` target >1 MB | `file.content { error: "too_large" }`, no content field |
| `file.read` target binary (>10 % NULL bytes in first 8 KB) | `file.content { error: "binary" }` |
| `file.write` while broker-level active-turn counter >0 | `file.write.result { ok: false, reason: "locked" }`, filesystem untouched |
| `file.write` content >1 MB | `file.write.result { ok: false, reason: "too_large" }` |
| `file.write` parent directory missing | `mkdir -p` then write |
| `file.write` I/O error (disk full, EACCES) | `file.write.result { ok: false, reason: "io_error" }` |
| chokidar `ready` takes >10 s | Logs a warning; broker still accepts WS connections; `file.list` returns `{ paths: [] }` until ready |
| chokidar emits `error` | Logs `console.error`, does not auto-restart; next container boot heals |

### 6.2 Host / UI Side

| Scenario | Behaviour |
|----------|-----------|
| WS drops mid-`file.read` (no `file.content` arrives) | Client-side 10 s timeout → red banner in editor, user can re-open the file |
| WS drops mid-`file.write` (no result arrives) | 10 s timeout → error banner, save button re-enabled |
| `file.changed` arrives for a file the user is dirty-editing | Update suppressed. The `file.write` will overwrite the external version. (In practice unreachable during a turn thanks to soft lock.) |
| Monaco lazy-load fails | `next/dynamic` loading fallback remains; user reloads page |
| Tab switches to Code mid-load while a new turn starts | Monaco renders content read-only as designed |

### 6.3 Agent Writes Currently-Open File

While `turnInFlight` is set, Monaco is already read-only. When `file.changed` arrives: if the file matches the currently-opened path AND the editor is not dirty, the UI re-issues `file.read` and replaces Monaco content. This produces the live "watch the agent edit" experience.

### 6.4 Paths Explicitly Without Error Handling

- Broker restart mid-session — container is gone, UI already shows "WS closed" via `wsStatus`.
- Concurrent agent + user write to the same file — impossible by construction via soft lock.

## 7. Testing Strategy

### 7.1 Broker Unit (TDD, red/green pairs)

- `fs-tracker.test.ts` — chokidar mock + temp dir:
  - initial-snapshot collection before `ready`
  - `ready` gate suppresses emission during startup
  - add/change/unlink events after ready
  - ignore-pattern filtering
  - source heuristic via injected `getCurrentTurn()`
- `fs-handlers.test.ts` — pure functions with real temp dir:
  - `handleFileList` returns sorted paths, excludes ignored
  - `handleFileRead` happy path + all four error variants
  - `handleFileWrite` happy path + all four error variants + atomic rename (verify no stray tmp file)
- `ws-server` new dispatch branches: `file.list` / `file.read` / `file.write` routing, soft-lock gate on write (mock `currentTurn`).

### 7.2 Protocol Package

No runtime tests (types-only module; `tsc --noEmit` at repo root is the gate).

### 7.3 Host Unit

- `buildTreeFromPaths(paths)` — pure helper, table-driven: empty, flat, nested, duplicate-prefix, deep.
- No React-component tests in Phase 1.3. UI logic is thin and live E2E covers behaviour.

### 7.4 Broker Integration

- `fs-e2e.test.ts` — starts broker + chokidar on a temp dir, connects a WS client, mutates files on disk, asserts that `file.changed` events arrive in expected order. Re-uses the `startBroker()` harness built in Phase 1.0 / 1.2.

### 7.5 Live E2E (final task of the phase, sonnet subagent)

Against a real Daytona sandbox:

1. Container boots, UI opens, WS connects, tree shows the template files (`app/layout.tsx`, `app/page.tsx`, `package.json`, `next.config.ts`, `tsconfig.json`).
2. Clicking `app/page.tsx` switches tab to Code; Monaco shows file content within 500 ms.
3. Sending a prompt ("change the heading to 'Foo'") puts editor into read-only; `file.changed` badge appears on `app/page.tsx`; after `agent.done` Monaco content auto-refreshes with the new heading; the preview iframe HMR-updates.
4. User edits file manually, Ctrl+S sends `file.write`; `file.write.result { ok: true }` returns; HMR updates the preview.
5. User attempts to save during an active turn: save button disabled, Monaco blocks keystrokes via `readOnly`.
6. User attempts to open a crafted >1 MB file: `file.content { error: "too_large" }` → banner in editor.
7. User attempts to open a binary file (e.g. any PNG in `/public`): `error: "binary"` → banner.
8. No event blocks the `agent.chunk` stream for more than 200 ms during a turn.

Expected new test count: ~20–25 tests (≈10 fs-tracker + ≈8 fs-handlers + ≈3 ws-server dispatch + ≈3 buildTree + integration).

## 8. Out-of-Scope and Carry-Overs

### 8.1 Explicitly Out of Phase 1.3

- `git commit` per agent turn (belongs to Phase 1.4)
- `git commit` for user edits before the next turn (Phase 1.4)
- Rollback UI / commit list
- Chat history persistence in Postgres `Message` table (Phase 1.4)
- Create / rename / delete files from the UI
- Multi-file editor tabs
- Diff viewer
- Search-in-files
- "File was changed externally" merge hint
- File-type icons and syntax-differentiated theme
- Tree virtualisation library (add when templates exceed ~500 files)

### 8.2 Carry-overs from Phase 1.2 Included in 1.3

- `file.changed` events via chokidar → covered by `fs-tracker`
- Markdown rendering of agent messages → covered by `components/chat/Message.tsx`

### 8.3 New Carry-overs Phase 1.3 → 1.4+

- **Pre-baked Docker image.** Container still runs on-boot `npm i -g @anthropic-ai/claude-code`. Remains a Phase 1.2b concern.
- **Second NDJSON fixture** for a later Claude Code version. Deferred again.
- **Nested `file.list`.** Flat paths suffice for Phase 1.3 tree sizes. Add an optional `pathPrefix` argument if later tree sizes demand lazy subtree loading.
- **`file.changed` debouncing.** If chokidar fires multiple events per atomic write, a 50 ms per-path debounce in the broker can be added. Introduce only if observed as noisy in live E2E.
