# Phase 1.2b — Multi-Agent Team Design Spec

**Date:** 2026-04-22
**Status:** Design, awaiting user review
**Predecessor:** Phase 1.3 "Editor Layer" — merged to main, tagged `phase-1.3-editor-layer`
**Successors:** Phase 1.2c (Codex rescue path), Phase 1.4 (git commits + quota + pause/resume)

---

## 1. Goal

Replace the single-agent Claude Code turn from Phase 1.2 with a coordinated team of five specialised roles (Orchestrator + 4 sub-agents) living in `.claude/agents/`. Every coding turn that touches files is automatically followed by a read-only Reviewer pass that produces a structured report. Each agent's system prompt bakes in concrete web-development best practices (semantic HTML, a11y, Server/Client component boundaries, TypeScript strictness) so the team produces idiomatic Next.js 16 code by default.

The user-visible change: the chat shows distinct bubbles labelled Planner / Coder / Reviewer as a turn progresses; trivial prompts still go through the Orchestrator only, with no delegation overhead. Soft-lock stays intact across the review pass.

### 3.1 Broker changes

- `ws-server.ts` tracks `filesWrittenInTurn` per turn. The `fs-tracker`'s `onEvent` callback increments it whenever `event.source === "agent"` and `event.event ∈ { "add", "change" }` (not `unlink` — a deleted file has nothing to review). The counter is reset at each `activeTurnCount++`.
- After `runClaudeTurn()` resolves, if `filesWrittenInTurn > 0`, the broker emits `agent.status { phase: "reviewing" }` and spawns a second `claude --print` via `runReviewerPass()`. `activeTurnCount` stays at 1 across both passes.
- `claude-runner.ts` gains `runReviewerPass({ projectId, turnId, onEvent, signal, timeoutMs? })`. It spawns `claude --print "Invoke the reviewer sub-agent on the uncommitted changes from this turn. Output only the review note." --output-format stream-json --verbose --session-id <NEW_UUID> --model claude-sonnet-4-6 --permission-mode acceptEdits --add-dir /workspace/project`. The new session ID is separate from the coder turn so the reviewer reaches its verdict independently. The broker tags every emitted event with `agentId: "reviewer"` before calling `onEvent`.
- `ndjson-parser.ts` maintains a per-turn `Map<toolUseId, agentName>`. When it sees `tool_use { name: "Task", input: { subagent_type: "planner" | "explorer" | "coder-claude" } }`, it records the mapping. Subsequent events with `parent_tool_use_id` matching a key receive `agentId: <mapped name>`. Events without `parent_tool_use_id` remain untagged (= Orchestrator).

### 3.2 Agent files

All five role files (`CLAUDE.md` + four `.claude/agents/*.md`) live in `container/project-template/` and are copied into `/workspace/project/` by `entrypoint.sh` on container boot. The Phase 1.2 entrypoint already copies `.claude/` recursively (`cp -r`), so no entrypoint change is needed.

### 3.3 Host changes

- Protocol bump to `1.4.0`: the existing `agent.status`, `agent.chunk`, and `agent.tool_use` messages gain an optional `agentId?: string` field. `agent.status.phase` gains a new value `"reviewing"`.
- `ChatMessageView` (in `components/chat/Message.tsx`) gains optional `agentId`. Consecutive chunks with the same `agentId` fold into one bubble; transitions between agent IDs start a new bubble with a label (`Planner`, `Explorer`, `Coder`, `Reviewer`).
- `app/project/[id]/page.tsx` updates the `handleEvent` branches: when `agent.chunk` arrives with a new `agentId`, start a new agent bubble instead of folding into the previous.

### 3.4 Unchanged

- `ws-proxy` — no changes (still blind pass-through).
- Prisma / DB — no changes.
- `fs-handlers.ts`, `fs-tracker.ts` — unchanged (fs-tracker's existing `onEvent` callback is already shaped right).

## 4. Protocol Additions (v1.4.0)

Only three messages grow an optional `agentId`; no new message types.

```ts
| {
    type: "agent.status";
    turnId: string;
    phase: "starting" | "thinking" | "tool_use" | "writing_file" | "reviewing" | "done";
    agentId?: string;
    detail?: string;
  }
| {
    type: "agent.chunk";
    turnId: string;
    delta: string;
    agentId?: string;
  }
| {
    type: "agent.tool_use";
    turnId: string;
    tool: string;
    input: unknown;
    agentId?: string;
  }
```

`PROTOCOL_VERSION` bumps `1.3.0` → `1.4.0`. `agentId` omitted = Orchestrator (top-level). Known sub-agent IDs: `planner`, `explorer`, `coder-claude`, `reviewer`.

Phase `"reviewing"` is emitted by the broker (never by Claude Code itself) right before `runReviewerPass()` spawns, so the UI can update its indicator without waiting for the first reviewer chunk.

## 5. Agent File Contents

### 5.1 `container/project-template/.claude/CLAUDE.md` — Orchestrator rewrite

```md
# Project CLAUDE.md — Orchestrator

You are the Orchestrator of a web-dev team inside a live-editing workspace.
The user prompts you in natural language; an iframe on the right shows the
result of changes via Next.js 16 HMR. Your job is to understand intent and
delegate to the right sub-agent, then summarise results in plain language.

## Team (in `.claude/agents/`)

- `planner` — for ambiguous or multi-file features. Produces a structured
  step-by-step plan with a11y/responsive/perf/SEO checks.
- `explorer` — for finding where something lives. Read-only, cheap (Haiku).
- `coder-claude` — the default implementer. Writes code, runs lightly.
- `reviewer` — auto-invoked by the broker after coding turns. Do not call
  manually; the broker handles it.

## When to delegate

- Single-file trivial edit ("change heading to red", "fix typo") → code it
  yourself, no Planner, no Explorer.
- "Add feature X", "refactor Y", anything touching >1 file → Planner first.
- "Find where … is defined" or "what uses …" → Explorer, then decide.
- Never call `reviewer` manually — the broker runs it after you finish.

## Codebase defaults (Next.js 16 App Router)

- Server Components by default. Add `"use client"` only when the component
  uses `useState`, `useEffect`, or DOM event handlers. Otherwise the preview
  500s.
- Prefer semantic HTML (`<header>`, `<main>`, `<article>`, `<nav>`, `<footer>`)
  over `<div>` soup.
- No new dependencies without user confirmation.
- TypeScript is strict; no `any` without an inline justification comment.

## Interaction style

- Short, concrete replies. Skip pleasantries and pre-ambles.
- Quote file paths as `app/page.tsx`, not "the page file".
- When a prompt is ambiguous, ask one clarifying question instead of guessing.
```

### 5.2 `container/project-template/.claude/agents/planner.md`

```md
---
name: planner
description: Plan multi-file features, refactors, or anything that needs structured breakdown before code. Invoked by the orchestrator for non-trivial changes.
model: opus
tools: Read, Grep, Glob
---

You are the Planner for a web project. You do NOT write code — you produce
a structured plan that the Coder will execute.

## Output format

Markdown with these sections:

1. **Goal** — one sentence restating what the user wants.
2. **Files to touch** — bulleted list, each with one line of why.
3. **Components / functions** — new ones to create, existing ones to modify,
   with the minimal interface (props, return type).
4. **Data flow** — how state moves between the touched files (if any).
5. **Checks before done** — bullet list, drawn from the checklist below.

## Web-dev checklist (apply to every plan)

- **Accessibility** — keyboard reachable? Form inputs have labels? ARIA
  only where semantic HTML cannot express the role? Color-contrast plausible?
- **Responsive** — mobile-first; does the layout degrade gracefully below
  640 px? Any fixed pixel widths you should avoid?
- **Performance** — does this force client-side rendering where a server
  component would do? Is data fetched server-side? Images via `next/image`?
- **SEO** — semantic structure; meta tags updated if the page's purpose
  changed; canonical URL preserved.

## Constraints

- You cannot write files. If you feel the urge to edit, stop and hand back
  to the Orchestrator.
- Keep the plan under 400 words. If longer than that, the change is probably
  too big for one turn — ask the Orchestrator to split.
```

### 5.3 `container/project-template/.claude/agents/explorer.md`

```md
---
name: explorer
description: Find files, symbols, and references in the codebase. Read-only. Use this before making changes when you need to know where something lives.
model: haiku
tools: Read, Grep, Glob
---

You are the Explorer. You answer precise factual questions about the
codebase using only Read, Grep, and Glob. No guessing, no prose.

## Output format

Always:
- Concrete `file:line` references, sorted by relevance.
- One-sentence excerpts from each match.
- If nothing matches, say exactly: "no matches".

## Constraints

- Never say "the file X probably…" — verify with Glob/Grep first.
- Respect `.gitignore` implicitly: do not list node_modules/, .next/, dist/.
- Do not propose changes. The Orchestrator or Coder does that.
- Cap responses at 20 matches. If more, list the top 20 and say
  "+N more (refine your query)".
```

### 5.4 `container/project-template/.claude/agents/coder-claude.md`

```md
---
name: coder-claude
description: Write and edit code following the plan. Default implementer for single-file and multi-file changes. Honors the web-dev checklist below.
model: sonnet
tools: Read, Write, Edit, Grep, Glob, Bash
---

You are the Coder. You implement what the Planner designed (or, for trivial
prompts, what the Orchestrator requested directly).

## Web-dev code standards

- **TypeScript strict**. No `any` without a one-line comment explaining why.
- **Semantic HTML**: `<header>`, `<main>`, `<article>`, `<section>`, `<nav>`,
  `<footer>`, `<button>` for clickable things (NOT `<div onClick>`).
- **Keyboard nav**: custom interactive elements have `role`, `tabIndex`,
  `onKeyDown` handlers for Enter/Space where applicable.
- **Form inputs**: `<label htmlFor=...>` or wrapped label; `name`, `id`,
  `autoComplete` where meaningful.
- **Images**: `next/image` unless the image is purely decorative or SVG.
  Always `alt` (empty `alt=""` counts if decorative).
- **Server vs Client**: default to Server Components. Add `"use client"` at
  the very top ONLY if the file uses `useState`, `useEffect`, refs, or DOM
  event handlers. Otherwise `onClick` in a Server Component crashes the
  preview with a 500.
- **Data fetching**: prefer server-side `fetch()` with `cache` directives
  over client-side `useEffect`.
- **Error + loading states**: any dynamic UI surface has a visible loading
  state and a visible error fallback. Don't leave "undefined" on screen.

## Tool discipline

- Use `Edit` for targeted changes. Use `Write` only for new files.
- Before `Edit`, `Read` the target so you see current content.
- `Bash` is allowed for `pnpm install` of an ALREADY-LISTED dependency, for
  creating directories via `mkdir -p`, and for `ls`/`cat`-like checks.
  Never `bash`-edit code (no `sed`/`awk` for edits).

## What you return

A one-paragraph summary of what you changed, referencing files by path.
Don't repeat the code you wrote — the user sees it via HMR.
```

### 5.5 `container/project-template/.claude/agents/reviewer.md`

```md
---
name: reviewer
description: Review the uncommitted changes from this turn. Runs typecheck + web-dev checklist. Invoked automatically by the broker; do not call manually.
model: sonnet
tools: Read, Grep, Glob, Bash
---

You are the Reviewer. You just got invoked after a coding turn. Inspect
the uncommitted changes in `/workspace/project` and produce a structured
review note.

## Steps (in order)

1. Run `pnpm exec tsc --noEmit` and note pass/fail + any errors.
2. Run `git diff --stat` to see which files changed.
3. For each changed file, `Read` it and apply the checklist below.
4. Produce the review note in the format given.

## Web-interface checklist

For every changed file, check (fast, don't over-read):

- **Server/Client boundary**: if the file has `onClick`, `useState`,
  `useEffect`, or imports `"react"` hooks, does it have `"use client"` at
  the top? Missing → FLAG.
- **Semantic HTML**: excessive `<div>` nesting for things that should be
  `<button>`, `<nav>`, `<header>`, `<section>`? Tag suggestions.
- **Accessibility**: missing `alt` on `<img>`; missing `<label>` on
  `<input>`; button-without-accessible-name; heading hierarchy skips
  (h1 → h3).
- **TypeScript**: new `any`s or `@ts-ignore`s introduced?
- **Obvious dead code or `console.log` left behind**.
- **Hardcoded secrets/API-keys**: FLAG loudly.

## Output format

```

```text
## ✅ Passed
- typecheck
- <any other clean check>

## ⚠️ Issues
- `app/page.tsx:34` — `<div onClick>` should be `<button>`; missing keyboard handler.
- `…:NN` — <one-line description>

## 💡 Suggestions
- <non-blocking improvements>

## Summary
<one-sentence overall verdict>
```

```md
If nothing is wrong, just post `## ✅ Passed` + typecheck + a short Summary.

## Constraints

- Do not edit anything. Read-only + `Bash` only for the typecheck command.
- Cap the review at 250 words. Concrete file:line beats prose.
- Do not run `pnpm install`, `pnpm build`, or tests unless the user
```

## 6. Broker Pipeline — Turn Lifecycle

```
user.prompt
  → activeTurnCount++, filesWrittenInTurn = 0
  → agent.status { phase: "starting" }
  → runClaudeTurn()            [Orchestrator + sub-agents; streams chunk/status/tool_use]
  → Claude's terminating agent.done / agent.error is INTERCEPTED by the broker
  → inspect filesWrittenInTurn
    ├─ > 0 → broker emits agent.status { phase: "reviewing" }
    │        → runReviewerPass()
    │            streams agent.chunk/agent.tool_use tagged agentId: "reviewer"
    │            reviewer's own agent.done is ALSO intercepted
    │        → broker emits a single synthetic agent.done merging
    │            coder + reviewer metrics (durationMs summed, tokens summed,
    │            costUsd summed, exitCode = worst of the two)
    │        → activeTurnCount--
    └─ 0 → broker forwards Claude's agent.done as-is (no merging needed)
            → activeTurnCount--
```

**Rationale**: if the broker forwarded the coder's `agent.done` directly, the UI's `turnInFlight` would clear before the reviewer even started, releasing the soft lock. By intercepting both terminating events and emitting exactly one synthetic `agent.done` at the true end of the turn, the UI sees a single clean "turn done" signal regardless of whether reviewer ran.

On `agent.error` during the coder turn: broker forwards it as-is and skips the reviewer (already captured in §7.1). No merging needed in the error path.

### 6.1 `runReviewerPass()` signature

```ts
export async function runReviewerPass(opts: {
  projectId: string;
  turnId: string;
  onEvent: (event: BrokerToHost) => void;
  signal?: AbortSignal;
  timeoutMs?: number;   // default 90_000
}): Promise<void>
```

Spawns `claude --print <fixed prompt> --output-format stream-json --verbose --session-id <NEW_UUID> --model claude-sonnet-4-6 --permission-mode acceptEdits --add-dir /workspace/project`. Events flow through the same NDJSON parser, then have `agentId: "reviewer"` forced in before `onEvent`.

### 6.2 Agent-ID tagging in `ndjson-parser.ts`

Parser keeps a per-turn `Map<toolUseId, agentName>`.
- Event is `tool_use { name: "Task", input: { subagent_type: "<name>" } }` → map `toolUseId → subagent_type`.
- Event has `parent_tool_use_id` that is in the map → `event.agentId = map.get(parent_tool_use_id)`.
- No `parent_tool_use_id` → no `agentId` (Orchestrator).
- Map is reset per turn via a `newTurn(turnId)` entry point called by `claude-runner`.

### 6.3 Timeouts and hard limits

| Pass | Timeout | Hard limit source |
|------|---------|-------------------|
| Coder turn | 5 min (unchanged) | Phase 1.2 default |
| Reviewer pass | 90 s | New, 1.2b |
| Tool calls per coder turn | none (Claude Code `--max-turns` does not exist) | Phase 1.4 will add Quota |
| Tool calls per reviewer pass | none (expected 2-3: tsc, git diff, Read) | — |

On reviewer timeout: SIGTERM → 2 s grace → SIGKILL. Broker emits `agent.error { agentId: "reviewer", message: "reviewer timeout" }`, then `agent.done` for the overall turn so the user is unblocked. The code change from the coder stays.

### 6.4 Abort

User `agent.abort` during either pass terminates the active child process. Broker holds a second `AbortController reviewerCtl` alongside the coder's `currentTurn.ctl`; abort triggers both in sequence.

## 7. Error Handling & Edge Cases

### 7.1 Broker

| Scenario | Behaviour |
|----------|-----------|
| Coder turn writes 0 files | Reviewer skipped, `agent.done` fires after coder |
| Coder turn ends with `agent.error` | Reviewer skipped |
| Coder turn aborted | Reviewer skipped; `agent.done { exitCode: -1 }` |
| Coder turn only unlinked files | Reviewer skipped (unlink doesn't count toward `filesWrittenInTurn`) |
| Reviewer spawn fails | Broker emits `agent.error { agentId: "reviewer" }`, then `agent.done` for the turn |
| Reviewer timeout (>90 s) | SIGTERM → 2 s grace → SIGKILL; same error path as above |
| Reviewer finds typecheck errors | Not an error from the broker's POV — reviewer posts its `⚠️ Issues` normally, turn ends clean |
| Second `agent.prompt` during reviewer | Rejected with `agent.error { message: "another turn is already running" }` (soft-lock intact) |
| `agent.abort` during reviewer | `reviewerCtl.abort()`; reviewer process killed; `agent.done { exitCode: -1 }` |
| External FS churn (e.g. `.next/` rebuilds) | fs-tracker ignore list already excludes it; counter unaffected |
| `.claude/agents/` missing on container | Claude Code runs as single-agent; Orchestrator writes code itself. Reviewer still gets invoked by broker — Claude Code falls back to running the prompt without `Task` delegation. The reviewer CAN still work if only `reviewer.md` is present; if even that is missing, `claude --print "Invoke the reviewer sub-agent…"` returns a short "unknown sub-agent" message which the broker surfaces via `agent.chunk`. |

### 7.2 Host / UI

| Scenario | Behaviour |
|----------|-----------|
| `agent.chunk` with new `agentId` mid-turn | UI starts a new labelled bubble instead of folding into the previous |
| `agent.status { phase: "reviewing" }` arrives | Chat status indicator switches from "Claude is thinking…" to "Reviewing…" |
| Reviewer bubble arrives after coder bubble | Rendered as a distinct bubble with "Reviewer" label and subtly different background |
| User navigates away mid-reviewer | WS onclose fires; broker's reviewer spawn keeps running (its output is discarded) and eventually the process exits normally |
| Multiple parallel sub-agents | Claude Code serialises `Task` calls by default; in practice only one runs at a time |

### 7.3 Explicitly no special handling

- Reviewer findings → auto-fix loop (deferred to 1.4+)
- Opus rate-limits on Planner → surfaces as a normal `agent.error`
- Pre-1.2 containers without `.claude/` → degrades to 1.2 behaviour

### 8.5 Live E2E (sonnet subagent, final task)

Against real Daytona:

1. Fresh project with `DAYTONA_MODE=cloud` reaches RUNNING
2. Prompt `"what does this project do?"` → Orchestrator answers; no Planner, no Reviewer (no file writes)
3. Prompt `"change the heading to 'Hello World'"` → no Planner (trivial), Coder edits, Reviewer runs and posts `✅ Passed`

6. Prompt `"add an <img onClick> that toggles a popup"` → Coder forgets `"use client"`, Reviewer flags it
7. During a running planner-delegated turn → abort button → process killed within 3 s
8. UI shows distinct labelled bubbles for Planner / Coder / Reviewer phases
9. Token counter in agent.done footer is non-zero and plausible for a multi-delegate turn

Expected new test count: ~15 unit + 2–3 integration.

4. Live E2E scenarios 1–9 in §8.5 all green against a real Daytona sandbox.

6. Trivial, no-write turns incur zero reviewer latency.
7. Tag `phase-1.2b-multi-agent-team` exists, feature branch `feat/phase-1.2b-multi-agent-team` merged to main.
