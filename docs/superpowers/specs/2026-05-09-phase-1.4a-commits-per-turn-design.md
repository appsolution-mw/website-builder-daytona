# Phase 1.4a — Commits per Turn — Design

Task: [T-20260509-006](../../tasks/active/T-20260509-006.md)
Date: 2026-05-09
Status: Draft (awaiting user review)

## 1. Goal

Capture every successful agent turn as a single git commit inside the sandbox,
persist a `Commit` row in the host DB, and expose a v0/Lovable-style History
mode in the workspace. The user gets an audit trail of what the agent
changed, and we lay the foundation for rollback (Phase 1.4a-B) and user-edit
commits (Phase 1.4a-C).

This is the first slice of Phase 1.4 from the master spec
(`2026-04-22-website-builder-daytona-design.md` §7.3, §9 Phase 4). The
remaining slices ship later as their own specs:

- **1.4a-B** — Rollback / Revert UI on top of the History mode
- **1.4a-C** — User-edit commits with a separate author identity
- **1.4d**   — Pause/Resume + Archive lifecycle automation
- **1.4e**   — Per-turn / per-user cost quota and hard limits

## 2. Decisions (recap from brainstorming)

| # | Decision | Rationale |
|---|---|---|
| 1 | Commit only when `AgentRun.status === SUCCEEDED` **and** working tree is dirty. | Keep history meaningful; no failed-state snapshots, no empty commits. |
| 2 | Commit author = `launchnode-agent <agent@launchnode.de>` for all runtimes. | Single bot identity matches v0/Lovable convention; runtime/model live in body + DB. |
| 3 | Commit title = first line of the user prompt, sanitised, truncated to 72 chars. | Makes `git log` read like a list of user wishes — most navigable for humans. |
| 4 | Git operations live in the broker (`git-handlers.ts`), reusing the existing `runGit` / `getGitStatus` / `sanitizeGitOutput` helpers. | FS locality, reuse of sanitisation that already protects against secret leakage. |
| 5 | Separate `Commit` model in Prisma, not fields on `AgentRun`. | Forward-compatible with rollback commits and user-edit commits, both of which exist without an AgentRun. |
| 6 | UI: top-bar **Code / History** mode switcher; History mode is master-detail (commit list + selected commit detail) over the full Files+Editor area. | v0/Lovable convention; gives diffs the horizontal space they need; clean foundation for Revert in 1.4a-B. |
| 7 | Skipped commits (`no_changes`, `commit_failed`) appear as small inline hint in the chat turn, **never** in the History panel. | History panel = "what changed in the codebase". Run-level signals belong in the chat. |
| 8 | Auto-commits stay **local**. GitHub-integrated projects keep using the existing `commitAndPushChanges` push path on explicit user action. | Avoids surprise pushes; preserves user control over the remote. |

## 3. Architecture

```
Agent-Runner (sandbox)
   ┊  AgentRun reaches SUCCEEDED, includes runtime/modelId/userMessage
   ▼
Broker (sandbox)             commitAgentTurn()
   ┊                           getGitStatus()  ─── clean ──▶ emit git.commit.skipped
   ┊                           git add -A
   ┊                           git commit --author=…  -m title  -m body
   ┊                           git rev-parse HEAD  → sha
   ┊                           git diff --shortstat HEAD~1 HEAD  → files/+/-
   ┊                         emit git.commit
   ▼
Worker-Agent (host VM)       HMAC-forwarded over the existing event channel
   ▼
Host (Next.js)               commit-recorder writes Commit row
   ┊                         broadcasts git.commit on the session WS
   ▼
Browser (Workspace)          History store prepends; Code/History badge bumps
```

- **Broker** owns all git operations; the agent-runner does **not** shell out
  to git. Locality + reuse of existing sanitisation.
- **Agent-runner** signals "run finished" with the metadata the broker needs
  to compose the commit (existing `agent.done` carries enough already).
- **Host** writes the row inside the same handler that already persists agent
  events; no new ingress endpoint.

## 4. Schema

New model + enum in `prisma/schema.prisma`. One Prisma migration.

```prisma
model Commit {
  id            String           @id @default(cuid())
  projectId     String
  project       Project          @relation(fields: [projectId], references: [id], onDelete: Cascade)
  sessionId     String?
  session       Session?         @relation(fields: [sessionId], references: [id], onDelete: SetNull)
  agentRunId    String?          @unique
  agentRun      AgentRun?        @relation(fields: [agentRunId], references: [id], onDelete: SetNull)
  sha           String           // 40-char hex
  shortSha      String           // first 7 chars, denormalised for cheap UI access
  authorKind    CommitAuthorKind
  runtime       AgentRuntime?    // null reserved for USER edits in 1.4a-C
  modelId       String?
  title         String           // sanitised, ≤ 72 chars
  bodyMessage   String           // full git body, capped at 8 KB
  filesChanged  Int
  insertions    Int
  deletions     Int
  createdAt     DateTime         @default(now())

  @@index([projectId, createdAt])
  @@index([sessionId, createdAt])
}

enum CommitAuthorKind {
  AGENT
  USER
}
```

`Project` and `Session` get back-relations. `AgentRun` gets an optional
back-relation to `Commit`.

The `sha` field is **not** unique across the table — different projects could
in principle share a sha. Uniqueness is `(projectId, sha)`, but we leave that
out for MVP; nothing depends on it and it would force a partial index.

## 5. Protocol Extension (`@wbd/protocol`)

Bump to `1.14.0`. Two new broker → host events.

```ts
type GitCommitEvent = {
  type: "git.commit";
  runId: string;
  sha: string;
  shortSha: string;
  title: string;
  bodyMessage: string;
  filesChanged: number;
  insertions: number;
  deletions: number;
  runtime: AgentRuntime;
  modelId: string | null;
  authorKind: "AGENT";        // "USER" reserved for 1.4a-C
  committedAt: string;        // ISO-8601
};

type GitCommitSkippedEvent = {
  type: "git.commit.skipped";
  runId: string;
  reason: "no_changes" | "commit_failed";
  detail?: string;            // sanitised git error, only when commit_failed
};
```

Both events flow through the existing broker → worker-agent → host bridge
that already carries `agent.*` events. No new transport.

## 6. Broker Implementation

### 6.1 New helper in `container/sandbox/broker/src/git-handlers.ts`

```ts
export interface CommitAgentTurnRequest {
  projectRoot: string;
  runId: string;
  userPromptFirstLine: string | null;
  userPromptFull: string | null;
  runtime: string;
  modelId: string | null;
}

export type CommitAgentTurnResponse =
  | {
      ok: true;
      sha: string;
      shortSha: string;
      title: string;
      bodyMessage: string;
      filesChanged: number;
      insertions: number;
      deletions: number;
      committedAt: string;
    }
  | { ok: false; reason: "no_changes" }
  | { ok: false; reason: "commit_failed"; detail: string };

export async function commitAgentTurn(
  input: CommitAgentTurnRequest,
): Promise<CommitAgentTurnResponse>;
```

Implementation:

1. `getGitStatus({ projectRoot })` → if `!hasChanges` → `{ ok: false, reason: "no_changes" }`.
2. Build title from `userPromptFirstLine` (or fall back to `agent turn <shortRunId>`).
   Sanitise: replace `[\x00-\x1f\x7f]` with single space, collapse runs of
   whitespace, trim, truncate to 72 chars with `…`.
3. Build body: `userPromptFull` (capped at 8 KB) + trailers
   `Runtime: <runtime>`, `Model: <modelId>`, `Run: <runId>`.
4. `git add -A`.
5. `git -c commit.gpgsign=false commit --author="launchnode-agent <agent@launchnode.de>" -m <title> -m <body>`.
   Wrap in try/catch. On failure return `{ ok: false, reason: "commit_failed",
   detail: sanitizeGitOutput(err.stderr ?? err.message) }`. Author flag is
   passed via `--author=` so the per-repo `user.email` set by entrypoint.sh
   does not bleed into the commit.
6. `git rev-parse HEAD` → sha.
7. `git diff-tree --shortstat --root <sha>` for the very first commit, else
   `git diff --shortstat HEAD~1 HEAD` → parse `N files changed, X insertions(+), Y deletions(-)`.
8. Return success with `committedAt = new Date().toISOString()`.

### 6.2 Wiring into the agent-run lifecycle

In the agent-runner ↔ broker boundary that already emits `agent.done`, the
broker recognises a successful terminal state and calls `commitAgentTurn` with
metadata pulled from the run context. Result is mapped to either
`git.commit` or `git.commit.skipped` and emitted on the same channel
`agent.done` uses.

The broker emits the commit event **after** the final `agent.done`. Order
matters for the chat UI: the turn-complete signal lands first, then the
commit-pill or skip-hint attaches to it.

## 7. Host Implementation

### 7.1 Event sink

Extend the broker-event handler that already persists `agent.*` events:

- On `git.commit` → `prisma.commit.create({ data: { ... } })` with
  `authorKind: "AGENT"`. Look up `sessionId` and `agentRunId` from the
  current run context.
- On `git.commit.skipped` → no DB row. Emit a UI-only event so the chat can
  show the inline hint.
- Re-emit both on the session WS that the browser already listens to.

If the DB write fails, log + retry once with backoff. On second failure log
an error and accept divergence — git remains source of truth.

### 7.2 REST

```
GET /api/projects/:projectId/commits?limit=50&cursor=<commitId>
GET /api/projects/:projectId/commits/:sha            // detail (single row)
GET /api/projects/:projectId/commits/:sha/files      // shortstat per file via broker
GET /api/projects/:projectId/commits/:sha/diff?path=<file>
```

- Auth: the same Project-ownership middleware used by the rest of `/api/projects`.
- Pagination: cursor over `(createdAt desc, id desc)`.
- The `/files` and `/diff` endpoints proxy a small request to the broker
  (`git show --stat <sha>` and `git show <sha> -- <path>`). Both pass the raw
  output through `sanitizeGitOutput`.

### 7.3 Project hydration

The existing project-detail loader returns the latest 20 `Commit` rows so
the History panel mounts populated. Older entries are paged in as the user
scrolls.

## 8. UI: History Mode

A top-bar mode switcher in the workspace toggles between **Code** (current
3-pane: FileTree | Editor | Preview, plus the chat) and **History**.

```
┌───────────────────────── Workspace ─────────────────────────┐
│  [ Code ] [ History ]                                       │
├──────────────────────────────────────────────────────────────┤
│  History mode (replaces Files+Editor area; Chat stays):     │
│                                                              │
│  ┌─ Commit list ─┬─ Selected commit detail ──────────────┐  │
│  │ ● abc1234     │ "Add a hero section to the landing"   │  │
│  │   "Add a …"   │ ─────────────────────────────────────  │  │
│  │   3 files     │ launchnode-agent · 2m ago              │  │
│  │   +47 −12     │ runtime: claude-code · sonnet-4-6      │  │
│  │   2m ago      │ Files changed:                         │  │
│  │               │   • src/app/page.tsx       +30 −5      │  │
│  │ ● def5678     │   • src/components/Hero.tsx +12        │  │
│  │   …           │   • styles/globals.css     +5 −7       │  │
│  │               │                                        │  │
│  │ (scroll)      │ Click a file → unified diff inline     │  │
│  └───────────────┴────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────────┘
```

### 8.1 Components

- `HistoryMode` — page-level container, owns the master-detail layout.
- `CommitList` — virtualised list, infinite scroll via cursor pagination.
- `CommitDetail` — header (title, author, runtime, model, time), file list,
  diff display.
- `CommitFileDiff` — collapsed by default, expands inline; uses a unified
  diff renderer (e.g. `react-diff-view` or a plain monospace block — picked
  in the implementation plan).
- `HistoryBadge` — count of new commits since the panel was last opened,
  shown on the History mode-switch label.

### 8.2 State + live updates

- Store: `useCommitHistory(projectId)` hook fed by initial hydration
  + cursor-paginated fetch.
- Live: hook listens for `git.commit` on the session WS and prepends.
- Skipped events: handled by a separate hook in the chat surface, not the
  history store.

### 8.3 Skipped indicator in the chat

After a turn that did not produce a commit, the chat-turn footer shows a
small de-emphasised line:

```
no code changes
```

For `commit_failed`, the same surface shows `couldn't save commit — <reason>`.
The detail string is already sanitised by the broker.

## 9. Edge Cases

| Case | Behaviour |
|---|---|
| AgentRun SUCCEEDED with no `userMessage` (programmatic trigger) | Title falls back to `agent turn <shortRunId>`. Body still carries runtime/model/run. |
| Working tree dirty, `git commit` fails (lock, corruption, hook reject) | Emits `git.commit.skipped` with `reason: "commit_failed"` and sanitised detail. AgentRun stays SUCCEEDED. |
| Sandbox restart mid-commit | Skip on restart. Next turn's `git add -A` sweeps any remnants in. |
| User edit in Monaco between turns | Edit gets folded into the **next** agent commit (`git add -A` is greedy). Documented limitation; clean separation lands in 1.4a-C. |
| Title contains newlines or control characters | Sanitised before commit and before DB persistence. |
| Body larger than 8 KB | Capped at 8 KB. Full prompt is already in the `Message` table. |
| Broker writes commit but host DB write fails | Git is source of truth. One retry, then log + accept divergence. |
| GitHub-integrated project (`PROJECT_SOURCE_TYPE=github`) | Auto-commit stays local. Push remains user-initiated through the existing `commitAndPushChanges` flow. |
| Two parallel runs for one project | SimpleScheduler already serialises per project. Out-of-scope to handle parallel writes. |
| `.gitignore` missing in `/opt/project-template` | Plan must verify the template ignores `node_modules/`, `.next/`, `.env`. |
| First commit in a new project (no `HEAD~1`) | Use `git diff-tree --shortstat --root <sha>` for the stats. |

## 10. Testing

### 10.1 Broker unit tests (`container/sandbox/broker/tests/`)

- `commitAgentTurn`: dirty tree → success; clean tree → `no_changes`;
  injected git failure → `commit_failed` with sanitised detail.
- Title sanitiser: newlines, control chars, length truncation, multi-byte
  unicode safety, fallback when prompt is null.
- Author flag wiring: assert `--author="launchnode-agent <agent@launchnode.de>"`
  is passed; assert per-repo `user.email` does not leak into the commit.
- First-commit case: `--root` shortstat path is exercised.

### 10.2 Host unit tests (`vitest.config.ts`)

- Event handler writes a Commit row with the correct fields.
- REST `GET /commits` paginates desc; rejects on missing project ownership.
- REST `/commits/:sha`, `/files`, `/diff` proxy to broker and sanitise.
- Project hydration returns the latest 20 commits.

### 10.3 Integration

- Real git temp dir end-to-end: dirty tree → commit → parse shortstat → event.
- Broker → worker-agent → host: HMAC-forwarded `git.commit` reaches the
  recorder.

### 10.4 E2E (live, on launchnode.de)

- New project → first prompt → commit appears in History panel within 2 s.
- Prompt that produces no edits → chat hint, panel unchanged.
- Force a commit failure (e.g. read-only `.git`) → `commit_failed` hint, panel unchanged.
- Multi-project: commits do not leak across projects.
- File click in CommitDetail → diff loads from the `/diff` endpoint.

## 11. Non-Goals

Explicit, called out so they do **not** silently expand the plan:

- ❌ Rollback / Revert UI (1.4a-B)
- ❌ User-edit commits with a `USER` author (1.4a-C)
- ❌ Branching, squash, amend, rebase, force-push
- ❌ Auto-push to GitHub remotes
- ❌ Pause / Resume / Archive (1.4d)
- ❌ Cost quota or hard token limits (1.4e)
- ❌ Per-runtime bot author identity
- ❌ Reconciliation between git and the `Commit` table
- ❌ Pre-commit linting, build gates, or commit-message validation

## 12. Implementation Order

1. Prisma schema migration (`Commit`, `CommitAuthorKind`).
2. `commitAgentTurn` in `git-handlers.ts` + unit tests.
3. Protocol bump to 1.14.0 + new event types.
4. Broker wiring on the agent-run lifecycle; emit `git.commit` /
   `git.commit.skipped`.
5. Worker-agent forwarding (verify the existing pipe carries the new event
   types, no new transport).
6. Host event recorder + project hydration patch.
7. REST endpoints (`/commits`, `/:sha`, `/:sha/files`, `/:sha/diff`).
8. UI: mode switcher, `HistoryMode`, `CommitList`, `CommitDetail`,
   `CommitFileDiff`, badge, chat-side skip hint.
