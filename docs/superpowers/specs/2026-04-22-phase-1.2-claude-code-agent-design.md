# Phase 1.2 — Single-Agent Claude Code in Container Design Spec

**Date:** 2026-04-22
**Status:** Design, awaiting user review
**Predecessor:** Phase 1.1 "Daytona Integration" — merged to main at `54cd12e`, tagged `phase-1.1-daytona-integration`
**Successors:** Phase 1.2b (multi-agent team), 1.3 (editor + file watcher), 1.4 (git commits per turn + quota)

---

## 1. Goal

Install Claude Code CLI inside each Daytona container and wire it to the broker. A user prompt from the workspace chat spawns a single `claude --print` subprocess that can read and write files in `/workspace/project/`. Output streams back to the chat as typed WebSocket events; the Next.js dev server in the container auto-reloads so the iframe preview updates live when Claude edits a file.

The result: "User types *'make the heading red'* → the text of `app/page.tsx` changes → iframe shows red heading." No sub-agents, no Planner/Reviewer/Codex, no file tree, no editor. Those come in later phases.

---

## 2. Chosen Options Summary (from brainstorming)

| Decision | Choice |
|----------|--------|
| Scope | Single Claude Code session per project, no sub-agents |
| Invocation | Claude Code CLI as broker subprocess, `--print --output-format stream-json` |
| Session persistence | `--session-id project-${id}` (Claude Code persists on disk) |
| Model | `claude-sonnet-4-6` hardcoded (env override deferred to Phase 1.4) |
| Turn concurrency | One at a time; new prompt during running turn → error reply |
| Abort | `SIGTERM` + 2s grace → `SIGKILL` |
| Permissions | `--dangerously-skip-permissions` (container is the sandbox) |
| Cost guard | `--max-turns 20` + 5 min hard timeout; no daily quota yet |
| API key | `ANTHROPIC_API_KEY` in host `.env` → passed via Daytona `envVars` → Claude Code reads env |

## 3. Container Changes

### 3.1 `container/entrypoint.sh` additions

After the existing `pnpm install` steps and before starting broker + next dev, add:

```sh
echo "[entrypoint] installing claude code cli…"
npm install -g @anthropic-ai/claude-code

echo "[entrypoint] copying .claude/ into project…"
cp -r "${REPO_DIR}/container/project-template/.claude" /workspace/project/.claude 2>/dev/null || true
```

Notes:
- `npm install -g` adds ~30s to first-spawn cold start (~90s in Phase 1.1 → ~120s now). Noted as known follow-up; Phase 1.2b may pre-bake Claude Code into a custom image.
- The `cp` uses `2>/dev/null || true` to tolerate the template already containing a `.claude/`; future-proof.

### 3.2 New: `container/project-template/.claude/CLAUDE.md`

Initial content — thin, because the project template itself is minimal:

```md
# Project CLAUDE.md

You are the coding agent for a live-editing web project. The user interacts
with you through a chat on the left of their browser; the iframe on the right
shows the result of your changes in real time via Next.js HMR.

## Codebase layout

- `app/` — Next.js 16 App Router pages. `app/page.tsx` is the home page; edits
  there are immediately visible in the preview.
- `app/layout.tsx` — shared HTML shell.
- No Tailwind; inline styles are fine for prototyping.

## Working style

- Make the smallest change that satisfies the user's request.
- Do not add dependencies without asking.
- Do not run tests, builds, or long commands unless specifically asked —
  the user sees the result live in the preview.
- When you edit a file, the change is reflected in the preview within ~1s.
```

Intentionally brief — Phase 1.3+ will let the user edit this file themselves for per-project customisation.

### 3.3 Environment variables passed at sandbox create

`lib/daytona/cloud.ts` `spawnProjectSandbox` extends `envVars`:

```ts
envVars: {
  PROJECT_ID: projectId,
  ANTHROPIC_API_KEY: process.env.ANTHROPIC_API_KEY ?? "",

```

If `ANTHROPIC_API_KEY` is not set on the host, the create still succeeds but any agent turn will fail with Claude Code's own "API key missing" error — surfaced to the user as `agent.error`.

`.env.example` gains:

## 4. Protocol (`@wbd/protocol`) Additions

### 4.1 Browser → Broker

```ts
| { type: "agent.prompt"; prompt: string; turnId: string }
| { type: "agent.abort"; turnId: string }
```

`turnId` is a client-generated UUID per user prompt. The broker echoes it on every event, letting the UI correlate streams to chat messages.

### 4.2 Broker → Browser

```ts
| { type: "agent.status";   turnId: string; phase: "starting" | "thinking" | "tool_use" | "writing_file" | "done"; detail?: string }
| { type: "agent.chunk";    turnId: string; delta: string }                                   // streamed assistant text
| { type: "agent.tool_use"; turnId: string; tool: string; input: unknown }                    // e.g. tool="Write", input={path,content}
| { type: "agent.done";     turnId: string; durationMs: number; tokensIn: number; tokensOut: number; costUsd: number; exitCode: number }
| { type: "agent.error";    turnId: string; message: string }
```

Existing `ping`/`pong`/`error` types remain for connection health checks. Phase 1.2 does not remove them; the chat UI just stops showing them prominently.

### 4.3 Event ordering guarantee

For any single turn, the broker emits events in this order:

```
agent.status{phase=starting}
agent.status{phase=thinking} (possibly multiple)
agent.chunk (zero or more, interleaved with status)
agent.tool_use (zero or more)
agent.status{phase=writing_file} (during tool execution)
… repeat thinking/chunk/tool_use until CLI exits
agent.done
```

or if failure:

```
agent.status{phase=starting}
… possibly partial output …
agent.error
```

The UI can safely assume `turnId` events arrive in timestamp order.

## 5. Broker Implementation

### 5.1 New module: `container/broker/src/claude-runner.ts`

Public surface:

```ts
export interface ClaudeRunnerOptions {
  projectId: string;
  prompt: string;
  turnId: string;
  onEvent: (event: BrokerToHost) => void;
  signal?: AbortSignal;
}

export async function runClaudeTurn(opts: ClaudeRunnerOptions): Promise<void>;
```

Responsibilities:
- Validate only one turn runs at a time (reject immediately if another is active — see §5.2)
- Spawn `claude --print` with correct flags
- Parse NDJSON from stdout, map each line to a protocol event, call `onEvent`
- Enforce 5-minute hard timeout
- Propagate `AbortSignal` to process kill

### 5.2 Turn serialisation in `container/broker/src/ws-server.ts`

The ws-server keeps a module-level `currentTurn: { turnId: string; abortCtl: AbortController } | null`.

Handler logic in pseudo-code:

```
on "agent.prompt" (prompt, turnId):
  if currentTurn != null:
    reply { type: "agent.error", turnId, message: "another turn is already running" }
    return
  ctl = new AbortController()
  currentTurn = { turnId, abortCtl: ctl }
  runClaudeTurn({ ...opts, onEvent: socket.send, signal: ctl.signal })
    .finally(() => { currentTurn = null })

on "agent.abort" (turnId):
  if currentTurn?.turnId != turnId:
    return // stale abort, ignore
  currentTurn.abortCtl.abort()
```

Note: the broker currently handles `ping` via `handleMessage` (a pure function). Agent prompts are stateful (they spawn processes, track currentTurn), so they must be handled inline in the WS connection handler rather than routed through `handleMessage`. The ping handler stays as-is; `agent.prompt` gets its own branch.

### 5.3 NDJSON event mapping

Claude Code's stream-json lines (observed format):

```json
{"type":"system","subtype":"init","session_id":"..."}
{"type":"assistant","message":{"content":[{"type":"text","text":"..."},{"type":"tool_use","name":"Edit","input":{...}}]}}
{"type":"user","message":{"content":[{"type":"tool_result","tool_use_id":"...","content":"..."}]}}
{"type":"result","subtype":"success","duration_ms":1234,"num_turns":3,"total_cost_usd":0.012,"usage":{"input_tokens":1234,"output_tokens":456}}
```

Mapping:

| NDJSON | Protocol event |
|--------|----------------|
| `{type:"system"}` | → `agent.status{phase:"starting"}` (only once, on first) |
| `{type:"assistant", ...content text}` | split content; emit `agent.chunk{delta}` for each text block |
| `{type:"assistant", ...content tool_use}` | emit `agent.tool_use{tool, input}` + `agent.status{phase:"tool_use", detail: tool}` |
| `{type:"user", tool_result}` | swallow; the status already signalled tool_use, no event needed (Phase 1.2 doesn't show tool_result detail) |
| `{type:"result", subtype:"success"}` | emit `agent.done{durationMs, tokens*, costUsd, exitCode:0}` |
| `{type:"result", subtype:"error" or process exits nonzero}` | emit `agent.error{message}` |

Defensive parsing: any NDJSON line that doesn't match the above is logged to broker stdout but does not crash the stream or surface to UI.

### 5.4 Process lifecycle details

- Spawn with `node:child_process spawn`, options: `{ env: {...process.env, ANTHROPIC_API_KEY}, cwd: "/workspace/project", stdio: ["pipe", "pipe", "pipe"] }`
- Prompt goes via `--print <prompt>` flag (not stdin) to avoid shell-escape issues; Claude Code handles quoting per its own argv parser
- stdout → NDJSON parser → `onEvent`
- stderr → buffered; on error exit, include last 4KB of stderr in `agent.error.message`
- Abort path: `SIGTERM` → wait 2s → `SIGKILL` → emit `agent.done{exitCode:-1}` (not `agent.error`, because abort is intentional)

## 6. Workspace UI

### 6.1 Chat input area (replaces ping button)

Below the existing chat list:

```tsx
<form onSubmit={onSubmit}>
  <textarea
    value={prompt}
    onChange={e => setPrompt(e.target.value)}
    placeholder="Tell Claude what to change…"
    rows={3}
    disabled={turnInFlight}
  />
  <div className="flex justify-between items-center">
    <span className="text-xs text-gray-500">{turnInFlight ? "Claude is thinking…" : ""}</span>
    <div>
      {turnInFlight && <button onClick={abort}>Abort</button>}
      <button type="submit" disabled={turnInFlight || !prompt.trim()}>Send</button>
    </div>
  </div>
</form>
```

### 6.2 Chat message rendering

Message types in the chat list:

```ts
type ChatMessage =
  | { kind: "user";   turnId: string; text: string }
  | { kind: "agent";  turnId: string; text: string; streaming: boolean; tools: string[]; footer?: string }
  | { kind: "error";  turnId?: string; text: string };
```

Streaming agent message:
- Starts empty on `agent.status{phase:"starting"}`
- Appends `delta` on each `agent.chunk`
- Shows a pulsing cursor while `streaming: true`
- Gray italic line "Claude is using **Write** on `app/page.tsx`" on `agent.tool_use`
- Gray footer on `agent.done`: `"12s · 1,234 in / 456 out · $0.012"`

Error message:
- Red background, shows error text, no streaming

Markdown rendering: **out of scope for Phase 1.2.** Agent text is rendered as `<pre>` / monospace plain text. Phase 1.3 or 1.4 adds markdown.

### 6.3 Existing ping mechanism

The ping test path stays in the protocol (useful for connection diagnostics) but the UI removes the prominent "Send ping" button. A small "ping" link in the header, visible only when WS status is degraded, can stay for debugging.

## 9. Risks

1. **Claude Code CLI NDJSON format instability.** The exact shape may change across Claude Code versions. Mitigation: defensive parsing, structured tests against recorded fixtures, log unknown lines without crashing. If Claude Code ships a breaking change we'll see it in the live E2E.

2. **Cold start bloat.** `npm i -g @anthropic-ai/claude-code` adds ~30s. Combined with Phase 1.1's 90s, first-spawn is now ~120s. Acceptable for MVP. Phase 1.2b will pre-bake.

3. **`--dangerously-skip-permissions` gives Claude full FS access.** By design — the container is the security boundary. Review: no secrets sit in `/workspace/project/`; `ANTHROPIC_API_KEY` and `GITHUB_CLONE_TOKEN` are env vars on the broker process, not files Claude can read.

4. **Prompt-injection via user input.** User prompt goes as a CLI argument, not shell-interpreted. Claude Code's argv parser handles quoting. No shell injection path.

5. **Turn concurrency edge cases.** Two rapid prompts in succession → the second gets `agent.error`. Acceptable; UI disables the button during turns anyway. A user hitting send twice before JS disables the button would see a single error, not a crash.

6. **Model name format.** Claude Code's exact model identifier for Sonnet 4.6 needs verification (`claude-sonnet-4-6` vs `claude-sonnet-4-6-20260101` vs other). Plan-1.2-Task-1 verifies from `claude --help` and `docs/DAYTONA_SDK_NOTES.md` convention.

7. **Large tool outputs.** Claude Code's NDJSON can emit large tool_result payloads (e.g., a file's full content). Phase 1.2 swallows these, but we still receive them via stdin. Could cause back-pressure. Mitigation: Node pipe default is 64KB buffer; drain continuously. Not expected to be an issue for small user-projects, but flagged.
