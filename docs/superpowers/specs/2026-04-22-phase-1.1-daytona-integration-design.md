## 1. Goal

Replace the local broker process with a real Daytona-hosted sandbox per project. After this milestone, creating a project via `POST /api/projects` provisions a Daytona container that clones the monorepo, starts the broker on port 4000, and a placeholder Next.js `next dev` on port 3000. The workspace UI connects through the ws-proxy to the container's broker and embeds the container's port-3000 preview URL in an iframe. Ping/pong continues to work end-to-end, now across the real network, against a real container.

No Claude Code integration yet (Phase 1.2). No editable project code (Phase 1.3). No pause/resume (Phase 1.4).

---

## 2. Chosen Options (from brainstorming)

| Decision | Choice |
|---|---|
| Base image | `node:24-alpine` (stock) |
| Code into container | Git clone of monorepo on container start |
| Private-repo auth | Fine-grained GitHub PAT (`Contents: Read-only`, single repo) |
| POST /api/projects flow | **Synchronous** — POST blocks until Daytona sandbox is `RUNNING` or error |
| Preview mechanism | Container exposes ports 4000 (broker) + 3000 (next dev); Daytona returns preview URLs for each |
| Testing | `DAYTONA_MODE=fake\|cloud` env toggle; fake spawns a local broker for unit tests |

## 3. Container Design

### 3.1 Image & Boot Sequence

**Base image:** `node:24-alpine`. No custom image build; we rely on Daytona to pull and cache this.

**Entrypoint script** lives at `container/entrypoint.sh` inside the repo and runs when Daytona starts the sandbox:

```sh
#!/bin/sh
set -e

# Required env (injected by host when calling daytona.create):
#   GITHUB_CLONE_TOKEN  — fine-grained PAT, Contents:Read on this repo
#   PROJECT_ID          — used by broker for session routing (future phases)
#   BROKER_PORT         — defaults 4000
#   PREVIEW_PORT        — defaults 3000

apk add --no-cache git
corepack enable pnpm

# Clone the monorepo
cd /workspace
git clone --depth=1 \
  "https://x-access-token:${GITHUB_CLONE_TOKEN}@github.com/${REPO_OWNER}/${REPO_NAME}.git" repo

cd repo
pnpm install --frozen-lockfile

# Stage the user-project template in its own directory
cp -r container/project-template /workspace/project
cd /workspace/project
pnpm install --frozen-lockfile

# Start next dev (background)
cd /workspace/project
pnpm dev &

# Start broker (foreground — if it exits, container exits)
cd /workspace/repo
exec pnpm -F @wbd/broker start
```

**Notes:**
- `--frozen-lockfile` keeps boots deterministic
- `exec` on broker ensures Daytona's process monitoring tracks it correctly
- Process supervision is intentionally minimal for Phase 1.1; if `next dev` crashes silently the preview stops working but broker keeps serving. Acceptable for MVP.

### 3.2 Container Filesystem Layout

```
/workspace/
├── repo/                    ← full monorepo clone (broker code lives here)
│   ├── container/broker/
│   ├── packages/protocol/
│   ├── ws-proxy/
│   └── container/project-template/  ← copy source
└── project/                 ← the "user project" for this container
    ├── app/
    │   └── page.tsx         ← "Hello from project <id>"
    ├── package.json
    ├── next.config.ts
    └── tsconfig.json
```

Two separate `pnpm install` runs because the broker and the user project are independent node_modules trees. Phase 1.3+ will make `/workspace/project` its own git repo.

### 3.3 Exposed Ports

| Port | Purpose | Public URL (from Daytona) |
|------|---------|---------------------------|
| 4000 | Broker WebSocket | `wss://<hash>-4000.proxy.daytona.app` |
| 3000 | Next.js dev (preview) | `https://<hash>-3000.proxy.daytona.app` |

The exact URL format depends on Daytona's preview-link API. We capture whatever `sandbox.getPreviewLink(port)` returns.

**Security in Phase 1.1:** URLs are world-reachable by default. Accepted because there is no user-sensitive content in the container yet. Signed-proxy gating is a Phase 5 (Auth) concern.

## 4. Project-Template Contents

A new directory `container/project-template/` in the monorepo contains the Phase-1.1 placeholder project. Minimal Next.js 16 app:

- `package.json` — deps: `next@^16`, `react@^19`, `react-dom@^19`
- `app/layout.tsx` — standard HTML shell
- `app/page.tsx` — `<h1>Hello from project</h1>` + reads `process.env.PROJECT_ID` if available
- `next.config.ts` — defaults
- `tsconfig.json` — Next.js defaults
- No Tailwind for the template (keep it minimal)

This is the *template*; later phases may offer multiple templates in a registry, but 1.1 has only this one.

## 5. Host-Side Changes

### 5.1 New Module: `lib/daytona/`

**`lib/daytona/client.ts`** — thin wrapper around the Daytona TypeScript SDK. Not a fat abstraction; just enough to keep the SDK's surface from leaking into the rest of the host.

```ts
export interface SandboxInfo {
  sandboxId: string;
  brokerUrl: string;   // wss://
  previewUrl: string;  // https://
}

export interface DaytonaClient {
  spawnProjectSandbox(args: {
    projectId: string;
    cloneToken: string;
    repoOwner: string;
    repoName: string;
  }): Promise<SandboxInfo>;

  destroyProjectSandbox(sandboxId: string): Promise<void>;

  getSandboxStatus(sandboxId: string): Promise<
    "provisioning" | "running" | "stopped" | "destroyed" | "error"
  >;
}

export function createDaytonaClient(): DaytonaClient {
  return process.env.DAYTONA_MODE === "fake"
    ? createFakeClient()
    : createCloudClient();
}
```

**`lib/daytona/cloud.ts`** — real SDK calls. Reads `DAYTONA_API_KEY`, `DAYTONA_API_URL` from env. Uses Daytona's current SDK (must be consulted during implementation; don't assume API shape from this spec).

**`lib/daytona/fake.ts`** — unit-test/dev-safe implementation. `spawnProjectSandbox` starts a local broker (same `startBroker` from `@wbd/broker`) on an OS-assigned port and returns `{ sandboxId: 'fake-xxx', brokerUrl: 'ws://localhost:NNNN', previewUrl: 'http://localhost:9999/fake-preview' }`. Keeps a map of sandboxId → local broker handle so destroy can clean up.

### 5.2 Environment Variables (additions)

Appended to `.env.example` and `.env`:

```
GITHUB_CLONE_TOKEN=                      # fine-grained PAT, Contents:Read
GITHUB_REPO_OWNER=appsolution-mw         # clone target (fixed for this project)
GITHUB_REPO_NAME=website-builder-daytona
DAYTONA_MODE=cloud                       # or 'fake' for tests / offline dev
```

`DAYTONA_API_KEY` and `DAYTONA_API_URL` already exist from Phase 1.0.

### 5.3 Prisma Additions

Migration `add_project_daytona_fields`:

```prisma
model Project {
  // ... existing fields
  daytonaSandboxId  String?
  brokerUrl         String?
  previewUrl        String?
  provisioningError String?
}
```

All nullable — additive migration, no data backfill needed.

### 5.4 API Endpoint Changes

**`POST /api/projects`** now:

1. Inserts `Project` row with status `PROVISIONING`
2. Calls `daytonaClient.spawnProjectSandbox(...)` — synchronous, blocks until success/failure. Target <60s typical; abort and mark `DESTROYED` if >120s (reasonable upper bound for first-spawn `pnpm install` + Daytona provisioning)
3. On success: updates row → `sandboxId`, `brokerUrl`, `previewUrl`, status `RUNNING`. Returns 201 with the full project.
4. On failure: updates row → `provisioningError`, status `DESTROYED`. Returns 500 with error.

This is synchronous on purpose. No polling state machine in 1.1. If Daytona's spawn time becomes unbearable we add polling in a follow-up.

**`DELETE /api/projects/[id]`** (new):

1. Look up project, get `sandboxId`
2. Call `daytonaClient.destroyProjectSandbox(sandboxId)` — tolerate `not found` errors (idempotent)
3. Update row → status `DESTROYED`, clear URL fields
4. Return 204

**`GET /api/projects`** / **`GET /api/projects/[id]`** — unchanged in shape but now project rows include the new fields (browser will use them).

### 5.5 ws-proxy Integration

`startProxy.resolveBrokerUrl(projectId)` in the runnable entry now consults the DB:

```ts
async resolveBrokerUrl(projectId: string): Promise<string> {
  const project = await prisma.project.findUnique({
    where: { id: projectId },
    select: { brokerUrl: true, status: true },
  });
  if (!project || project.status !== "RUNNING" || !project.brokerUrl) {
    throw new Error(`project ${projectId} is not running`);
  }
  return project.brokerUrl;
}
```

The ws-proxy entry-point file imports Prisma. This gives ws-proxy a runtime dep on `@prisma/client` and the adapter, which is acceptable (ws-proxy is a host-local sidecar).

Tests remain unchanged — they still pass a fake resolver.

## 6. UI Changes

### 6.1 Dashboard (`app/page.tsx`)

- Status badge next to each project: `PROVISIONING` (spinner), `RUNNING` (green dot), `DESTROYED` (grey, strikethrough)
- While any project is `PROVISIONING`, poll `/api/projects` every 3s
- "Delete" button next to each project — calls `DELETE /api/projects/<id>`
- Empty/loading/error states preserved from Phase 1.0

### 6.2 Workspace (`app/project/[id]/page.tsx`)

Two-state layout:

**If project status !== RUNNING:**
- Full-page "Provisioning container…" with a spinner
- Copy: "First boot takes about a minute while we pull dependencies"
- Polls `/api/projects/[id]` every 2s until `RUNNING` (or `DESTROYED` → error state)

**If project status === RUNNING:**
- 50/50 split: left = chat panel (existing ping/pong), right = `<iframe src={project.previewUrl}>` with `sandbox="allow-scripts allow-same-origin allow-forms"`
- WS connection opens on mount (as Phase 1.0)
- iframe loads the container's next-dev URL

Full three-panel Lovable-style layout (File-Tree + Monaco Editor) is Phase 1.3.

## 7. Lifecycle (Phase 1.1 Subset)

```
POST /api/projects
  ↓
PROVISIONING (DB insert)
  ↓ daytona.create (sync, ~30–90s first-time)
  ├─ ok   → RUNNING    (sandboxId, brokerUrl, previewUrl set)
  └─ fail → DESTROYED  (provisioningError set)

DELETE /api/projects/[id]
  ↓ daytona.destroy
  → DESTROYED (urls cleared)
```

Phase 1.4 will add `PAUSED` / `ARCHIVED` transitions.

## 10. Risks & Mitigations

1. **Daytona SDK API drift** — my design assumes a shape; implementer must read the current `@daytonaio/sdk` docs before writing `lib/daytona/cloud.ts`. Plan-1.1-Task 1 includes a docs-verification step.

2. **Cold-start >90s** — if first-time spawn exceeds 90s due to `pnpm install`, the UX gets poor. Mitigation: ensure Daytona's disk-snapshot caching is enabled after first spawn; consider switching to custom pre-built image (Phase 1.2b) if persistent.

3. **PAT token exposure** — `GITHUB_CLONE_TOKEN` is passed to the container as an env var and appears briefly in the clone URL. Daytona may log this. Mitigation: scope token narrowly (Contents:Read single repo), rotate on known compromise, plan to migrate to GitHub App for proper short-lived tokens.

4. **Port mapping for WSS** — if Daytona returns `wss://` but our `ws-proxy` has hardcoded `ws://` assumptions anywhere, the connection fails silently. Mitigation: the ws-proxy was designed to be URL-scheme-agnostic; verify during implementation.

5. **`pnpm install` compatibility** — `node:24-alpine` is musl-based. Some native deps (sharp, bcrypt, pg native) may need additional packages (`apk add build-base python3`). We have `pg`, which uses pure-JS when `@prisma/adapter-pg` is configured with it. Should be fine; verify during integration test.

6. **Sync POST risks browser timeout** — if Daytona takes 90s, the POST HTTP request may time out at Vercel/nginx/Next.js dev default limits. Mitigation: keep Next.js dev settings default (no aggressive timeout); if this becomes an issue, switch to 202 Accepted + poll pattern in 1.1b.
