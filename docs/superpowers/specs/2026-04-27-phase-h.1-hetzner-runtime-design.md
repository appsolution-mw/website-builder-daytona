# Phase H.1 — Hetzner-Runtime (Multi-Cloud) Design Spec

**Date:** 2026-04-27
**Status:** Design, awaiting user review
**Project root:** `/Volumes/Extern/Projekte/website-builder-daytona`

---

## 1. Goal

Replace Daytona Cloud as the per-project sandbox runtime with a self-operated, multi-cloud, production-near alternative. The system must:

- Run on Hetzner Cloud as the first provider, with a clean abstraction that lets us add Vultr, Linode, AWS, Azure, GCP, OVH later without rewrites.
- Auto-provision Worker-VMs on demand (no manual VM creation per spawn).
- Connect Host and Workers exclusively over Tailscale (no public Docker sockets, no port-forwarding).
- Serve customer preview URLs on a wildcard subdomain of our own domain (`*.preview.<our-domain>`) with valid TLS.
- Survive Worker-VM crashes without losing customer data (state lives in Postgres + Git, not on Workers).
- Reach functional parity with the existing Daytona integration: same `Runtime` API, same UI, same WS-protocol — only the implementation behind the interface changes.

**Trust model:** Customers are vetted, each works on their own projects under operator supervision. Container-level hardening (gVisor/Kata) is intentionally deferred to a later phase.

**UX target for Phase H.1:** Smooth experience for up to ~50 active sandboxes on a single Host-VM with 1–3 Worker-VMs. Spawn time ≤30 s when a Worker has capacity, ≤120 s when a fresh Worker must be provisioned.

Non-goals are documented in section 11.

---

## 2. Chosen Options Summary

| Decision | Chosen Option |
|---|---|
| Skala / Trust | Multi-Tenant SaaS, vertraute Kunden, "production-near" |
| Container-Runtime | Plain Docker mit strikter `Scheduler`-Abstraktion |
| Topologie | Host-VM separat von Worker-VMs |
| Networking-Backbone | Tailscale Free Tier (Migrationspfad zu Headscale offen) |
| Provider-Strategie | Multi-Cloud via `WorkerProvisioner`-Interface; Phase H.1 implementiert Hetzner |
| Provisioning | Automatisch ab Phase H.1 (Hetzner Cloud API + Tailscale Auth-Key API) |
| Datenbank | Self-hosted Postgres auf Host-VM, tägliches Backup auf Hetzner Storage Box |
| TLS-Termination | Caddy zentral auf Host-VM, Wildcard-Cert via DNS-Challenge |
| Container-Image | Pre-built Sandbox-Image in Container-Registry (ghcr.io oder Hetzner Registry) |
| Auto-Pause | Ja — nach 30 min Idle, Resume on demand |
| Worker-Capacity | Hart konfiguriert (z.B. 8 Sandboxes pro CCX33), erweiterbar |

## 3. Architecture Overview

```
┌─────────────────────────────────────────────────────────────────────┐
│                          BROWSER                                     │
│  - Chat / FileTree / Editor / Preview-iframe                         │
└────────────┬────────────────────────────────────────┬───────────────┘
             │ WSS (host)                             │ HTTPS (preview)
             ▼                                        ▼
┌──────────────────────────────┐         ┌────────────────────────────┐
│      HOST-VM (CX22 → CCX33)  │         │   Caddy Reverse-Proxy      │
│  ┌────────────────────────┐  │         │   (auf Host-VM)            │
│  │ Next.js 16 App         │  │         │  - Wildcard *.preview.…    │
│  │  - Auth, Projects, UI  │  │         │  - Routet zu Worker:Port   │
│  │  - WS-Proxy            │  │         │    via Tailscale-IP        │
│  │  - HetznerRuntime      │  │         │  - Token-Auth für Broker   │
│  │     ├─ Scheduler       │  │         │    (X-Sandbox-Token)       │
│  │     ├─ HetznerProvis.  │  │         └─────────────┬──────────────┘
│  │     ├─ CaddyControl    │  │                       │
│  │     └─ TailscaleClient │  │                       │
│  └────────────────────────┘  │                       │
│  ┌────────────────────────┐  │                       │
│  │ Postgres 16            │  │                       │
│  │  - Projects, Workers,  │  │                       │
│  │    Sandboxes, Tokens   │  │                       │
│  └────────────────────────┘  │                       │
│  + Tailscale-Daemon          │                       │
└─────────────┬────────────────┘                       │
              │ Docker-API + WSS                       │
              │ über Tailscale (100.x.y.z)             │
              ▼                                        ▼
┌──────────────────────────────────────────────────────────────────────┐
│  WORKER-VM #1  (Hetzner CCX33)        WORKER-VM #N  (beliebig)       │
│  ┌────────────────────────┐           ┌────────────────────────┐     │
│  │ Docker Engine          │           │ Docker Engine          │ ... │
│  │  ├─ Sandbox A          │           │  ├─ Sandbox D          │     │
│  │  ├─ Sandbox B          │           │  └─ Sandbox E          │     │
│  │  └─ Sandbox C          │           └────────────────────────┘     │
│  └────────────────────────┘           + Tailscale, Worker-Agent      │
│  + Tailscale, Worker-Agent                                           │
└──────────────────────────────────────────────────────────────────────┘
```

**Drei Schichten, klar entkoppelt:**

1. **Host-VM** — steuert alles, hält State, ist im Tailnet, bleibt bewusst klein und stabil
2. **Caddy auf Host-VM** — Wildcard-TLS-Termination, routet eingehenden Preview/Broker-Traffic per Tailscale zur richtigen Worker:Port. Validiert Sandbox-Token auf WSS-Verbindungen
3. **Worker-VMs** — überall (Hetzner / Vultr / Linode / AWS / …), zustandslos, hosten Docker-Container für Sandboxes

**Drei kritische Abstraktionen:**

- **`Runtime`-Interface** — heutiger `DaytonaClient` wird umbenannt zu `Runtime` (Drop-in-API: `spawn`, `destroy`, `getStatus`). Bestehende `cloud.ts`/`fake.ts` bleiben, neu: `hetzner.ts`.
- **`Scheduler`-Interface** — entscheidet "auf welchem Worker spawne ich Sandbox X?". Phase H.1: `SimpleScheduler` (least-loaded). Später austauschbar.
- **`WorkerProvisioner`-Interface** — fährt Worker-VMs hoch/runter. Phase H.1: `HetznerProvisioner` + `FakeProvisioner` (für Tests).

## 4. Components

### 4.1 Host-VM (neu / geändert)

| Modul | Zweck | Größe |
|---|---|---|
| `lib/runtime/` | Umbenennung von `lib/daytona/` zu generischem Runtime-Layer (alte Daytona-Implementierung bleibt drin) | Refactor |
| `lib/runtime/hetzner/` | Neue Implementierung des `Runtime`-Interface, intern: ruft Scheduler+Provisioner+Caddy | ~400 LoC |
| `lib/scheduler/` | Wählt Worker für neue Sandbox, hält Mapping aktuell | ~250 LoC |
| `lib/provisioner/` | `WorkerProvisioner`-Interface + `HetznerProvisioner` + `FakeProvisioner` | ~350 LoC |
| `lib/caddy-control/` | Schreibt Caddy-Config-Snippets (per Sandbox eine Route), reload via Admin-API | ~150 LoC |
| `lib/tailscale/` | Tailscale-Auth-Key-Generierung via API-Token, Worker-IP-Lookup | ~120 LoC |
| `services/host-agent/` | Empfängt Heartbeats von Workern, registriert/deregistriert sie in DB, dispatched auto-pause-Job | ~200 LoC |

### 4.2 Worker-VMs (per cloud-init beim Provisioning)

| Komponente | Zweck |
|---|---|
| Docker Engine 24+ | Container-Runtime |
| Tailscale-Daemon | Joint Tailnet via Auth-Key, ephemeral=true |
| `worker-agent` (Node.js, single binary) | Heartbeat zum Host alle 10 s, exponiert Docker-Socket nur über Tailscale-IP, signed mit shared HMAC |

### 4.3 Pre-built Sandbox-Image

In Container-Registry (Default: ghcr.io/<org>/sandbox:<tag>):

- Basis: `node:24-bookworm-slim`
- Vorinstalliert: pnpm, `@anthropic-ai/claude-code`, codex CLI, broker (kompiliert), git, ca-certificates
- ENTRYPOINT: leichtgewichtiges Bootstrap-Script — empfängt `PROJECT_ID`, `GITHUB_CLONE_TOKEN`, `BROKER_TOKEN` via Env, klont Repo, startet Broker + `next dev`
- Ziel-Spawn-Zeit: ~15 s (vs. heute ~2–3 min)
- CI-Workflow baut + pushed Image bei jedem PR-Merge in main

### 4.4 Was unverändert bleibt

- `ws-proxy/` (Browser ↔ Host) — liest schon `project.brokerUrl` aus DB, URL-Format ist transparent
- `container/broker/` (Agent-Broker im Container) — wird nur in das Image gebacken statt ad-hoc gebootet
- `lib/runtime/index.ts` (umbenannt von `lib/daytona/index.ts`) — bleibt als Factory, bekommt einen weiteren Mode `hetzner`

## 5. Data Flows

### 5.1 Sandbox spawnen

```
1. User klickt "Open Project" im UI
2. Next.js API-Route POST /api/projects/:id/start
3. runtime.spawnProjectSandbox({...})
4. ↳ scheduler.pickWorker(projectId)
   ├─ Wenn Worker mit Kapazität: nimm ihn
   └─ Wenn keiner: provisioner.provision({region, size}) → blockiert ~60–90s
5. ↳ docker.containerCreate auf gewähltem Worker (über Tailscale-IP)
   - Image: ghcr.io/<org>/sandbox:<tag>
   - Env: PROJECT_ID, GITHUB_CLONE_TOKEN, BROKER_TOKEN (random JWT-32),
          AGENT_RUNTIME, ANTHROPIC_API_KEY/CLAUDE_CODE_OAUTH_TOKEN, …
   - Labels: sandbox-id=…, project-id=…, customer-id=…
   - Published ports: brokerPort, previewPort (zufällig in 30000-39999)
6. ↳ caddyControl.addRoute({
     hostname: `${sandboxId}-preview.preview.<our-domain>`,
     upstream: `${worker.tailscaleIp}:${previewPort}`,
   })
7. ↳ caddyControl.addRoute({
     hostname: `${sandboxId}-broker.preview.<our-domain>`,
     upstream: `${worker.tailscaleIp}:${brokerPort}`,
     auth: { header: 'X-Sandbox-Token', value: brokerToken },
   })
8. caddy.adminApi.reload() — graceful, ~50ms, keine Connection-Drops
9. DB:
   - INSERT WorkerSandbox(workerId, projectId, containerId, brokerPort, previewPort, status=RUNNING)
   - INSERT SandboxToken(sandboxId, token=brokerToken, expiresAt=now+24h)
   - UPDATE Project SET brokerUrl, previewUrl, status='RUNNING'
10. Return SandboxInfo zum UI

Spawn-Zeit gesamt:
  ~15–30s (Worker existiert)
  ~90–120s (Worker muss provisioniert werden)
```

### 5.2 Browser ↔ Sandbox (Live-Sitzung)

```
Browser
  │ wss://app.<our-domain>/p/:projectId          ← Host-WS-Proxy
  ▼
Host (ws-proxy)
  │ liest project.brokerUrl
  │ wss://${sandboxId}-broker.preview.<our-domain>/?token=…
  ▼
Caddy (auf Host-VM)
  │ Wildcard-TLS-Termination
  │ Validiert X-Sandbox-Token-Header gegen `SandboxToken`-Tabelle
  │ (32-byte Random-Token, DB-Lookup pro Verbindungsaufbau, nicht pro Frame)
  │ Forward zu http://${worker-tailscale-ip}:${brokerPort}
  ▼
Worker-VM (Tailscale-IP, Docker-published-port)
  │ Forward zu Container localhost:${brokerPort}
  ▼
Broker im Container — Claude/Codex laufen, streamen Events
```

Preview läuft analog, nur ohne Token-Auth (öffentlich, wie bei Daytona heute) und über separaten Hostname.

### 5.3 Auto-Provisioning Flow (HetznerProvisioner)

```
provisioner.provision({region: 'fsn1', size: 'ccx33'}) {
  1. Tailscale-API: POST /api/v2/tailnet/-/keys
     Body: { capabilities: { devices: { create: { ephemeral: true,
            reusable: false, tags: ['tag:worker'] } } } }
     → tsAuthKey (gültig 5 min)

  2. Hetzner Cloud API: POST /v1/servers
     - image: 'docker-ce' oder 'ubuntu-24.04' + cloud-init
     - server_type: 'ccx33'
     - location: 'fsn1'
     - user_data: cloud-init-Script:
         - apt install tailscale
         - tailscale up --authkey=$TS --hostname=worker-$RAND --ephemeral
         - docker login ghcr.io -u … -p …
         - docker pull ghcr.io/<org>/sandbox:<tag>
         - docker run -d --restart=always worker-agent \
             --host-url=$HOST --hmac=$SHARED_SECRET
     → blockt ~30–60s bis "running"

  3. Polling: Tailscale API bis Worker im Tailnet auftaucht (~30s)
     ↳ liefert tailscale-IP (100.x.y.z)

  4. Polling: GET http://${tailscaleIp}:${agentPort}/health bis 200 (~10s)

  5. DB: INSERT INTO workers (
       tailscale_hostname, tailscale_ip, provider='hetzner',
       provider_vm_id, region, capacity=8, status='READY')

  6. Return Worker-Record
}
```

**Failure-Handling:**

- Hetzner-Provisioning hängt → Timeout 180 s, Worker wird `force-deleted`, scheduler.pickWorker propagiert Fehler.
- Tailscale joint nicht → Timeout 60 s, Server wird gelöscht, Auth-Key revoked (ephemeral cleanup automatisch).
- Health-Check fehlt → Timeout 30 s, Server gelöscht.

**Decommissioning** (Cron alle 10 min für Idle-Worker, nach 60 min ohne Sandboxes):

- DB-Status auf `DRAINING` (kein neuer Schedule)
- Hetzner: server delete
- Tailscale: ephemeral key cleanup automatisch nach disconnect
- DB: Worker-Eintrag bleibt (Audit), Status `DECOMMISSIONED`

## 6. Data Model

Drei neue Prisma-Modelle:

```prisma
enum WorkerStatus {
  PROVISIONING
  READY
  DRAINING
  DECOMMISSIONED
  OFFLINE
}

enum SandboxLifecycleStatus {
  SPAWNING
  RUNNING
  PAUSED
  DESTROYED
}

model Worker {
  id                String         @id @default(uuid())
  tailscaleHostname String         @unique  // worker-abc123
  tailscaleIp       String                   // 100.x.y.z
  provider          String                   // 'hetzner' | 'vultr' | 'linode' | …
  providerVmId      String                   // hetzner server id
  region            String                   // 'fsn1' | 'ash' | …
  capacity          Int                      // max concurrent sandboxes
  status            WorkerStatus
  lastHeartbeatAt   DateTime?
  createdAt         DateTime       @default(now())
  decommissionedAt  DateTime?
  sandboxes         WorkerSandbox[]
}

model WorkerSandbox {
  id          String                  @id @default(uuid())
  workerId    String
  worker      Worker                  @relation(fields: [workerId], references: [id])
  projectId   String                  @unique  // ein Sandbox pro Projekt
  containerId String                          // docker container id
  brokerPort  Int
  previewPort Int
  status      SandboxLifecycleStatus
  createdAt   DateTime                @default(now())
  pausedAt    DateTime?
  lastActivityAt DateTime              @default(now())
}

model SandboxToken {
  id          String   @id @default(uuid())
  sandboxId   String   @unique
  token       String   @unique         // 32-byte random; gesendet als X-Sandbox-Token
  expiresAt   DateTime
}
```

`Project.brokerUrl` und `Project.previewUrl` bleiben — sie zeigen jetzt auf `https://<sandboxId>-broker.preview.<our-domain>` bzw. `https://<sandboxId>-preview.preview.<our-domain>`.

## 7. Error Handling & Sandbox-Lifecycle

### 7.1 Failure-Matrix

| Szenario | Behandlung |
|---|---|
| Worker-VM crasht | Worker-Agent sendet keine Heartbeats. Host markiert nach 30 s `status=OFFLINE`. Sandboxen darauf werden in DB als `needs_respawn` markiert. Browser bekommt `sandbox.lost`-Event → UI bietet "Restart" → spawnt auf anderem Worker (Repo neu geclont, kein Datenverlust weil Code in Git). |
| Sandbox-Container crasht | Worker-Agent erkennt es per Docker-Event-Stream, meldet zum Host, Host updated DB, Browser bekommt `sandbox.crashed`. UI zeigt Restart-Button. |
| Tailscale-Verbindung Worker → Host bricht | Worker-Agent hat exponential backoff Reconnect. Host markiert Worker als `unhealthy`, neuer Schedule geht woanders hin. Existierende Sandboxen werden nicht sofort als kaputt markiert (kann Hiccup sein) — erst nach 60 s. |
| Hetzner Cloud API down | `provisioner.provision` schlägt fehl, Scheduler fällt auf bestehende Worker zurück. Wenn alle voll → User-facing Fehler "capacity, try in a moment". |
| Caddy reload schlägt fehl | Spawn-Workflow ist transaktional: Container existiert, Routing nicht. Cleanup: Container stoppen, DB-Record löschen, Fehler zurück. |
| Postgres auf Host-VM down | Host-App liefert 503. Bestehende Browser-WS brechen ab (ws-proxy braucht DB für Lookup), neue Sessions schlagen fehl. Sandboxen auf Workern laufen weiter. Recovery: Postgres-Restart. |
| Tailscale-Auth-Key abgelaufen | Provisioning-Code holt frisch einen Key bei jedem `provision()`-Call (kein Caching) → vermeidet Klasse von Bugs. |
| Image-Pull fehlt auf Worker | Worker-Agent macht beim Boot `docker pull` mit retry-Loop. Worker meldet erst dann `READY`. |
| Doppel-Spawn (Race) für gleiches Project | DB-Constraint: `WorkerSandbox.projectId` unique. Zweiter Spawn fängt Constraint-Violation ab → nutzt bestehende Sandbox. |
| Sandbox läuft, Browser-WS bricht | Sandbox läuft weiter (Auto-Pause greift erst nach 30 min). Browser kann jederzeit reconnecten. |

### 7.2 Sandbox-Lifecycle States

```
                   ┌──────────────┐
   spawn() ─────→  │  SPAWNING   │ ─── creation fail ─→ destroyed (cleanup)
                   └──────┬───────┘
                          │ container ready + caddy routed
                          ▼
                   ┌──────────────┐
                   │   RUNNING    │ ←──┐
                   └──────┬───────┘    │ user activity
                          │            │
                  30min idle           │
                          ▼            │
                   ┌──────────────┐    │
                   │   PAUSED     │ ───┘ user reopens project
                   │ (container   │
                   │  stopped,    │ ── 7 days idle ─→ ARCHIVED
                   │  routes      │
                   │  removed)    │
                   └──────┬───────┘
                          │
                          ▼
                   ┌──────────────┐
                   │  DESTROYED   │
                   └──────────────┘
```

- **PAUSED**: `docker stop` (kein delete) + Caddy-Routes entfernt. Resume = `docker start` + Routes rein. Resume-Zeit ~3–5 s.
- **ARCHIVED**: Container weg, `WorkerSandbox`-Record bleibt für 30 Tage zur Wiederherstellung (Code ist eh in Git).

## 9. Sub-Phasen-Aufteilung von Phase H.1

### H.1a — Foundation: Runtime-Abstraktion + FakeProvisioner (~3 Tage)

- Refactor `lib/daytona/` → `lib/runtime/` (rename, keine Verhaltensänderung; bestehende Daytona-Pfade bleiben)
- Definiere `Runtime`, `Scheduler`, `WorkerProvisioner` Interfaces
- Implementiere `FakeProvisioner` + `LocalDockerWorker` für Testing
- Implementiere `SimpleScheduler` (least-loaded)
- DB-Migrations: `Worker`, `WorkerSandbox`, `SandboxToken`
- ENV: `RUNTIME_MODE` ergänzt `DAYTONA_MODE` (alte ENV bleibt für Backward-Compat)

**Validation:** Lokaler `next dev` mit `RUNTIME_MODE=hetzner-fake` spawnt funktionierende Sandbox auf Local-Docker, der bestehende `fake`-Mode läuft weiter.

### H.1b — Pre-built Sandbox-Image + Worker-Agent (~3 Tage)

- Dockerfile für Sandbox-Image (Node + Tools + Broker vorinstalliert)
- CI-Workflow zum Image-Build und Push (ghcr.io)
- `worker-agent` schreiben (Heartbeat, Docker-Socket-Proxy, HMAC-Auth)
- cloud-init-Template für Worker-VMs

**Validation:** Manuell provisionierte Hetzner-VM mit Image läuft — Host kann Sandbox darauf spawnen via Tailscale.

### H.1c — Auto-Provisioning + HetznerProvisioner (~4 Tage)

- `HetznerProvisioner` mit Hetzner Cloud API
- Tailscale-API-Integration (Auth-Key-Generierung)
- Provisioning-Failure-Handling (Timeouts, Cleanup)
- Decommission-Worker-Job (Cron alle 10 min: Worker ohne Sandboxes seit >60 min → drain + destroy)

**Validation:** Host startet komplett leer, User klickt "Open Project" → Worker wird provisioniert → Sandbox spawnt → Preview läuft, alles automatisch.

### H.1d — Caddy-Wildcard-TLS + Auto-Pause (~3 Tage)

- Caddy auf Host-VM, Wildcard-Cert via DNS-Challenge (Cloudflare oder Hetzner DNS)
- `CaddyControl` schreibt Routes als JSON, reload via Caddy-Admin-API
- Auto-Pause-Job (Cron alle 5 min: Sandboxen ohne Activity seit >30 min → docker stop + Routes raus)
- Resume on activity (User-Klick im UI → API-Call → docker start + Routes rein)

**Validation:** Volle End-to-End-Reise mit echter Domain, TLS-Cert sauber, Idle-Sandbox geht in Pause, Resume klappt.

**Gesamt:** ~13 Werktage solide Arbeit, ~3 Wochen Kalenderzeit bei Solo-Builder mit Subagent-Driven-Development.

Jede Sub-Phase bekommt einen eigenen Implementation-Plan (analog zum Pattern der bisherigen Phasen 1.0–1.3) und wird sequenziell ausgeführt — H.1a → H.1b → H.1c → H.1d.

## 10. Skalierungs-Stufen (zur Orientierung)

| Stufe | Aktive Sandboxes | Setup | Monatskosten |
|---|---|---|---|
| Phase H.1 Launch | bis ~10–20 | 1× CX22 Host + 1× CCX33 Worker + Storage Box | ~35 € |
| Solides Wachstum | bis ~50 | 1× CCX33 Host + 2–3× CCX33 Worker + Storage | ~120 € |
| Production stretched | bis ~150 | CCX43 Host + 5× CCX33 Worker + Postgres-Replica + Cloudflare-Free | ~250 € |
| Multi-Region | beliebig | + Worker-Pools in zusätzlichen Regionen + Geo-DNS | linear pro Region |

Die Stufen sind migrierbar **ohne Code-Rewrite**, weil `Scheduler`/`Provisioner`/`CaddyControl` austauschbar und stateless gegen DB sind.

## 11. Non-Goals (bewusst aus Phase H.1 raus)

- gVisor / Kata / Firecracker Container-Härtung → Phase H.2
- Multi-Region Worker-Pools → Phase H.3
- Postgres-Replikation / Failover → Phase H.4
- Quotas / Billing pro Customer → Phase H.4
- Monitoring-Stack (Prometheus / Grafana / Loki) → später, `docker stats` + Caddy-Access-Logs reichen initial
- Caddy-auf-Worker (TLS-Termination dezentral) → Phase H.3, zentraler Caddy reicht für ~50 Sandboxes
- Pre-Warm-Container-Pool → Pre-built-Image gibt schon 80 % des Effekts
- Cloudflare oder anderer CDN/Edge-Layer → kann jederzeit nachgerüstet werden, kein Lock-in
- Weitere Provisioner (Vultr, Linode, AWS, Azure, GCP, OVH) → erst wenn ein konkreter Bedarf da ist; Architektur sieht sie vor

## 12. Migration vom heutigen Daytona-Setup

- Bestehende `DAYTONA_MODE=cloud|fake` Pfade bleiben unverändert. Neuer Mode: `RUNTIME_MODE=hetzner` (optional fallback auf `DAYTONA_MODE`).
- Existierende laufende Projekte auf Daytona bleiben dort. Cutover pro Projekt: Sandbox auf Daytona destroy, neu spawn auf Hetzner — Code ist in Git, keine State-Migration nötig.
- Feature-Flag-frei, da Auswahl per ENV pro Host-Deployment erfolgt.
