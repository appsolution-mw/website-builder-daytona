# Phase 1.4a-C — User-Edit Commits — Design

Task: [T-20260511-001](../../tasks/active/T-20260511-001.md)
Date: 2026-05-11
Status: Draft (awaiting user review)

## 1. Goal

Persist Editor (and Terminal) changes between agent turns as standalone
`USER`-authored commits. Triggered automatically by a 5-second debounce on
the broker's existing chokidar file watcher. Eliminates the
"dirty-tree-blocks-revert" gap from Phase 1.4a-B and gives every user edit
its own audit row in the History panel.

This is the third slice of Phase 1.4 from the master spec
(`2026-04-22-website-builder-daytona-design.md` §7.3, §9 Phase 4). It builds
on the `Commit` model + History UI from Phase 1.4a and the revert flow from
Phase 1.4a-B.

The remaining Phase 1.4 slices ship later as their own specs:

- **1.4d** — Pause/Resume + Archive lifecycle automation
- **1.4e** — Per-turn / per-user cost quota and hard limits

## 2. Decisions (recap from brainstorming)

| # | Decision | Rationale |
|---|---|---|
| 1 | **Trigger** is a broker-side 5-second chokidar debounce on `/workspace`. Each FS event resets the timer; on fire, the broker auto-commits. | Robust against WS disconnects, single source of truth, covers terminal-side edits with no extra plumbing. The chokidar watcher already exists for `file.changed` events. |
| 2 | **Skip while running**: the debounce timer is paused while any agent run is active (`activeRuns.size > 0`). | Prevents the broker from claiming the agent's writes as USER edits. Phase 1.3 soft-lock already covers the editor-side. |
| 3 | **Auto-flush before agent runs and before reverts**: `flushUserCommitsIfPending()` runs synchronously before `provider.runTurn` and before `revertToCommit`. | Preserves authorship (otherwise pre-run dirty edits would be bundled into the AGENT commit). Eliminates the 1.4a-B "dirty tree blocks revert" race window. |
| 4 | **Author identity** is the project owner's Better-Auth `user.name` + `user.email`, passed to the broker via env vars `BROKER_USER_NAME` + `BROKER_USER_EMAIL` at sandbox spawn time. | Survives broker / sandbox restart; one source; trivial to wire up through the existing worker-pool provisioner. If env vars are missing, USER commits are skipped silently (defensive fallback). |
| 5 | **Auto-generated commit message** — title is `Edit <file1>` (1 file), `Edit <file1> and <file2>` (2 files), or `Edit N files` (≥ 3); body is per-file shortstat. | Readable in `git log`, consistent with the AGENT-commit pattern from 1.4a, no user prompt required at debounce time. |
| 6 | **Granularity**: one commit per debounce window batches all dirty files. | Matches Lovable convention; prevents per-file commit spam. |
| 7 | **In-progress race**: while a commit is mid-flight, the broker sets a `commitInProgress` flag; chokidar events that arrive during it queue a fresh timer for after completion. | Atomicity without deadlocks. |
| 8 | **Reuse Phase 1.4a infrastructure** — the `Commit` schema, `git.commit` event channel, and `CommitAuthorKind.USER` enum value are already in place. No schema migration. | Phase 1.4a Decision 5 anticipated this. |

## 3. Architecture

```
Workspace (browser)
  ┊  User edits Monaco / Terminal
  ┊  → ws-proxy → broker file.write → /workspace/foo.tsx changes
  ▼
Broker (in sandbox)
  ┊  chokidar 'change' / 'add' / 'unlink' fires
  ┊    if activeRuns.size > 0: ignore
  ┊    if commitInProgress: set pendingAfter = true; return
  ┊    else: clearTimeout(userCommitTimer); setTimeout(5_000) ─┐
  ┊                                                              │
  ┊  Timer fires:                                              ◄┘
  ┊    commitUserEditsIfDirty()
  ┊      git status -z       → empty?  yes → skip
  ┊      git add -A
  ┊      git diff --cached --quiet  → no staged change? yes → skip
  ┊      git -c commit.gpgsign=false commit
  ┊        --author "<user.name> <user.email>"
  ┊        -m "Edit foo.tsx and bar.css"
  ┊        -m <body with per-file shortstat + Author trailer>
  ┊      git rev-parse HEAD, readShortstat → emit git.commit (authorKind=USER)

  Broker.revertToCommit() / runAgentTurn():
    flushUserCommitsIfPending()      ◄── synchronous before existing flow
    proceed with existing logic

Host (provisioner)
  ┊  At sandbox spawn: BROKER_USER_NAME, BROKER_USER_EMAIL env vars
  ┊    sourced from project.owner.{name, email}

Host (persistence) — see §7.2 for details
  ┊  broker enqueues USER-commit payload after successful commit
  ▼
  worker-agent long-poll → POST /api/internal/projects/[id]/commits/user
  ┊  HMAC-authed; idempotent upsert on (projectId, sha)
  ▼
  Commit row written with authorKind=USER, userEmail=<owner-email>,
  agentRunId=null, sessionId=null

Browser (existing WS subscription, parallel to persistence)
  ┊  page.tsx git.commit handler prepends to commitHistory immediately
  ┊  on the WS broadcast — does not wait for the host DB write.
  ┊  History UI shows the row with a USER icon + author email.
```

The broker still owns all git operations. The chokidar watcher is single-
instance per broker process. The flush hooks are direct in-process calls,
not RPCs. The persistence path is decoupled from the WS path so a slow
or failing host DB write does not delay live UI updates.

## 4. Schema

One additive Prisma migration. Phase 1.4a's `CommitAuthorKind.USER` enum
value is already in place, but USER commits need a dedicated email column so
the History UI can render the right author label without parsing the commit
body.

```prisma
model Commit {
  // ...existing fields from Phase 1.4a + 1.4a-B
  userEmail String?   // populated only when authorKind = USER
}
```

Safe `ALTER TABLE … ADD COLUMN` with no default. Migration name:
`add_user_email_to_commit`.

## 5. Protocol Extension (`@wbd/protocol`)

Bump to `1.17.0`. One-character widening of the `git.commit` event's
`authorKind` union.

```ts
type GitCommitEvent = {
  // ...existing fields
  authorKind: "AGENT" | "ROLLBACK" | "USER";
  // ...
};
```

For `USER` events:
- `turnId: null` (no associated agent run)
- `runtime: null`
- `modelId: null`
- `revertedFromSha` omitted

`git.commit.skipped` and other events are unchanged. No new event types.

## 6. Broker

### 6.1 New primitive: `commitUserEditsIfDirty`

New function in `container/sandbox/broker/src/git-handlers.ts`, peer to
`commitAgentTurn`. Signature:

```ts
export interface CommitUserEditsRequest {
  projectRoot: string;
  authorName: string;
  authorEmail: string;
}

export type CommitUserEditsResponse =
  | {
      ok: true;
      sha: string;
      shortSha: string;
      title: string;
      bodyMessage: string;
      filesChanged: number;
      insertions: number;
      deletions: number;
      committedAt: string;
    }
  | { ok: false; reason: "no_changes" }
  | { ok: false; reason: "no_identity" }       // env vars not set
  | { ok: false; reason: "commit_failed"; detail: string };

export async function commitUserEditsIfDirty(
  input: CommitUserEditsRequest,
): Promise<CommitUserEditsResponse>;
```

Sequence:

1. If `authorName` or `authorEmail` is empty → return `no_identity`.
2. `getGitStatus({ projectRoot })` — empty porcelain? → return `no_changes`.
3. `git add -A`.
4. `git diff --cached --quiet` — exit 0 means index matches HEAD (only
   gitignored or no-op edits were "dirty") → reset index, return
   `no_changes`.
5. Read changed-file list via `git diff --cached --name-status`. Compose
   title with the same `Edit <…>` rules as in §2 Decision 5.
6. Compose body: per-file `<path> | +<insertions> -<deletions>` via
   `git diff --cached --numstat`, plus a trailer line
   `Author: <email>` for symmetry with the agent body trailers.
7. `git -c commit.gpgsign=false commit --author "<name> <email>" -m <title>
   -m <body>` (use `--author`, not `git config`, to leave the existing
   `commitAndPushChanges` push-flow identity untouched).
8. `git rev-parse HEAD` → `sha`; `readShortstat(projectRoot, sha)` → stats.
9. Return the `ok: true` payload.

### 6.2 New debounce manager

A new module `container/sandbox/broker/src/user-edit-watcher.ts` (or
co-located in `ws-server.ts` if the surface stays small):

```ts
export function startUserEditWatcher(opts: {
  projectRoot: string;
  activeRuns: Map<string, ActiveRunState>;
  broadcastEvent: (event: BrokerToHost) => void;
  getIdentity: () => { name: string; email: string } | null;
}): { dispose: () => void; flushPending: () => Promise<void> };
```

State machine:

```
State: { timer: NodeJS.Timeout | null; inProgress: boolean; pendingAfter: boolean }

on chokidar 'change' | 'add' | 'unlink':
  if activeRuns.size > 0: return
  if inProgress: pendingAfter = true; return
  if timer: clearTimeout(timer)
  timer = setTimeout(fireCommit, 5_000)

fireCommit():
  if activeRuns.size > 0: timer = null; return    // race: run started in window
  inProgress = true; timer = null
  identity = getIdentity()
  if !identity: emit nothing, log once, inProgress = false; return
  result = await commitUserEditsIfDirty({ projectRoot, ...identity })
  if result.ok: emit git.commit event with authorKind=USER
  inProgress = false
  if pendingAfter: pendingAfter = false; schedule another timer immediately

flushPending():               // called by run-start and revert
  if timer: clearTimeout(timer); timer = null
  if inProgress: await until inProgress=false
  await fireCommit()          // one synchronous attempt
```

The watcher is started once per broker process inside the existing
`startServer` flow (after `projectRoot` is known but before the WS server
accepts clients).

Chokidar ignore globs MUST exclude:
- `node_modules/**`
- `.next/**`
- `.git/**`
- `.agent-artifacts/**`

These are the same globs the project template's `.gitignore` covers.

### 6.3 Identity wiring

The broker reads `BROKER_USER_NAME` and `BROKER_USER_EMAIL` from
`process.env` at boot. `getIdentity()` returns `{ name, email }` or
`null` if either is unset or empty.

### 6.4 Flush hooks

Two existing code paths call `flushPending()`:

1. **`agent-run-executor.ts`** — before `provider.runTurn(...)` for every
   run. Adds at most one synchronous commit before the agent runs. Acceptable
   delay (~100–500 ms for a small dirty tree).
2. **`ws-server.ts` `/git/revert` HTTP handler** — before
   `revertToCommit({...})`. Removes the dirty-tree race entirely.

### 6.5 Event shape for USER commits

`commitUserEditsIfDirty` returns a payload; the watcher constructs the WS
broadcast event:

```ts
opts.broadcastEvent({
  type: "git.commit",
  turnId: null,
  sha: result.sha,
  shortSha: result.shortSha,
  title: result.title,
  bodyMessage: result.bodyMessage,
  filesChanged: result.filesChanged,
  insertions: result.insertions,
  deletions: result.deletions,
  runtime: null,
  modelId: null,
  authorKind: "USER",
  committedAt: result.committedAt,
});
```

`revertedFromSha` is omitted (USER commits never set it).

## 7. Host

### 7.1 Provisioner env vars

`lib/runtime/worker-pool/*` (and / or `lib/runtime/provisioner/*`) builds
the env block for each sandbox / broker process. Add:

```ts
const ownerUser = await prisma.user.findUnique({
  where: { id: project.ownerId },
  select: { name: true, email: true },
});

env.BROKER_USER_NAME = ownerUser?.name ?? ownerUser?.email ?? "";
env.BROKER_USER_EMAIL = ownerUser?.email ?? "";
```

If the owner has no email at all, both vars are empty and the broker
silently skips USER commits (`no_identity` path). The user can still ship
changes via the normal `commitAndPushChanges` flow.

The exact file to touch depends on how worker-pool composes the broker
env today — that's discovered in the implementation plan.

### 7.2 Persistence path (out-of-run commits)

Phase 1.4a's `persistCommitEvent` in `lib/agent-runs/executor-client.ts`
only fires inside the per-run NDJSON stream. USER commits happen outside
runs, so they need a different path to reach the host DB.

The broker cannot write to the DB directly (no credentials by design). The
browser sees the WS broadcast but cannot be the source of truth (tab might
not be open). The clean path is **broker → worker-agent → host**, mirroring
the existing run-event chain.

```
Broker (in-process queue of pending user-commits)
   ┊  on commitUserEditsIfDirty success: enqueue payload
   ▼
Broker HTTP endpoint  GET /internal/projects/host/user-commits/pull
   ┊  long-poll (30 s) — returns when queue non-empty, or empty array on timeout
   ▼
Worker-agent has a persistent loop:
   ┊  GET .../user-commits/pull (long-poll)
   ┊  for each event:
   ┊    POST <host>/api/internal/projects/{projectId}/commits/user
   ┊      headers: x-hmac-signature: <hmac>, x-worker-id: <id>
   ┊      body: <user-commit payload>
   ▼
Host route writes Commit row (auth: HMAC matches the worker's stored secret).
```

Long-poll over a separate WS / push channel keeps the worker-agent's
existing HTTP-only outbound model intact. Pull-on-the-broker means the
broker doesn't need to know the worker-agent or host URLs (the worker-agent
already knows the broker URL via the existing broker-rpc plumbing).

The broker's WS broadcast (§6.5) is unchanged — it remains the live UI
update channel. The pull+POST path is purely for durability.

### 7.3 New host route

`POST /api/internal/projects/[id]/commits/user`

- **Auth**: HMAC signed by the worker-agent (same secret as the existing
  per-run event ingress).
- **Body**: `{ sha, shortSha, title, bodyMessage, filesChanged, insertions,
  deletions, committedAt, userEmail }`.
- **Behaviour**:
  - Validate HMAC.
  - Validate sha format (`/^[a-f0-9]{40}$/`).
  - `prisma.commit.upsert({ where: { id: sha-derived }, … })` — idempotent
    on `(projectId, sha)` so duplicate POSTs (worker-agent retries) are
    harmless.
  - Write row with `authorKind: "USER"`, `runtime: null`, `modelId: null`,
    `agentRunId: null`, `sessionId: null`, `userEmail: <email>`,
    `revertedFromSha: null`.
- **Response**: 201 with `{ id }`, or 409 if the row already exists, or
  401 on bad HMAC.

### 7.4 Worker-agent changes

`worker-agent/src/server.ts` (or its broker-bridge module) grows a new
long-lived loop per provisioned project:

```ts
while (running) {
  const events = await pullBrokerUserCommits(brokerUrl, brokerToken);
  for (const ev of events) {
    await postToHost(`/api/internal/projects/${projectId}/commits/user`, ev);
  }
}
```

Started when the worker-agent provisions a sandbox; stopped on sandbox
teardown. Reuses the existing project-id-keyed lifecycle map. Failures are
logged and retried with simple exponential backoff capped at 30 s — the
broker keeps the pending queue durable in memory until acknowledged.

### 7.5 Browser update

The browser already receives `git.commit` events with `authorKind: "USER"`
via the existing ws-proxy → broker WS broadcast (§6.5). The page.tsx
event handler (Phase 1.4a-B Task 13) prepends to `commitHistory` for any
`git.commit` event regardless of `authorKind`. No additional work needed
on the browser side beyond the §8 UI tweaks.

Phase 1.4a-B already widened `CommitView.authorKind` to include

Three small UI tweaks:

### 8.1 `CommitList.tsx`

The current row prefixes ROLLBACK titles with `↶`. Add a USER variant:

```tsx
<span className="truncate">

   commit.authorKind === "USER" ? "✎ " : ""}{commit.title}

```

### 8.2 `CommitDetail.tsx`

The author display currently hardcodes `"launchnode-agent"`. For USER
commits, show the email (or `"you"` if it matches the currently logged-in

```tsx
<span>
  {isUser ? "user" : "launchnode-agent"}
</span>
{isUser && <span className="text-xs">{commit.userEmail ?? ""}</span>}
```

`CommitView` extends with `userEmail: string | null` — sourced from the

added in Phase 1.4a-B Task 7.

### 8.3 `AgentTurnCommitChip` (chat)

No change. USER commits have no `agentRunId` → `commitsByRunId.get(turnId)`
returns `undefined` → chip is not rendered. Decision 5 in 1.4a anticipated
this.

### 8.4 `RevertConfirmDialog` `dirty_tree` variant

Becomes essentially dead code in normal operation (the pre-revert flush
clears dirty trees first). **Keep it** as a safety net — the broker still
returns `dirty_tree` if, for example, the identity env vars are missing
and so the flush silently skipped.

## 9. Edge cases

| Case | Behaviour |
|---|---|
| Dirty tree, all files gitignored | `git add -A` stages nothing → `git diff --cached --quiet` exits 0 → return `no_changes`. |
| Identity env vars unset / empty | Return `no_identity`. Watcher logs once and skips; no event emitted. The user can still ship through `commitAndPushChanges`. |
| File-change during commit | `commitInProgress = true`, `pendingAfter = true`; after commit completes, schedule a fresh 5 s timer. |
| User edits during active run (terminal bypass; soft-lock failure) | chokidar event suppressed by `activeRuns.size > 0` check. Edits get bundled into the AGENT commit at run end. Acceptable per Phase 1.3 soft-lock contract. |
| Broker / sandbox restart | Identity re-read from env vars at boot. Pending dirty tree triggers a fresh debounce on the first chokidar event after boot. |
| Many small saves in a tight burst | Timer resets on each event → one commit batches them all once the 5 s of silence elapses. |
| Multiple tabs editing the same project | Each tab issues its own file writes; chokidar sees them as a stream and debounces them together. Identity is per-project, not per-tab — all commits attributed to the project owner. |

## 10. Out of scope (for 1.4a-C)

- Multi-user project sharing (the USER identity is always the project
  owner).
- Per-file commit splitting.
- Custom commit messages typed by the user.
- Commit squashing or amend.
- Auto-push of USER commits to GitHub (still explicit user action via
  `commitAndPushChanges`).
- GPG signing of USER commits.
- A "saving…" / "saved" UI indicator driven by the debounce timer (could
  be added as a follow-up — broker emits a `user.commit.pending` event the
  browser shows; explicitly deferred for MVP).

## 11. Test plan (high level)

- **Broker unit tests** (`git-handlers.test.ts`): each branch of
  `commitUserEditsIfDirty` — happy path, `no_changes` (clean), `no_changes`
  (all gitignored), `no_identity`, `commit_failed`, title composition
  variants (1, 2, ≥ 3 files), body shortstat parsing.
- **Watcher state-machine unit tests**: the `commitInProgress` /
  `pendingAfter` race; `activeRuns.size > 0` skip; flush during in-progress
  commit; timer reset on consecutive events; flush idempotency.
- **Broker integration test** (`ws-server.test.ts`): a sandbox-style temp
  repo, edit a file via direct FS write, advance fake timers 5 s, assert
  the WS broadcast carries `authorKind: "USER"` with the right title.
- **Pre-flush integration**: trigger a run while a debounce timer is
  pending — assert the user commit lands before the agent run starts.
- **Host route tests** for the new `/api/internal/projects/[id]/commits/user`
  ingress: HMAC auth, payload validation, Commit row creation, idempotency
  (same sha twice).
- **Frontend smoke**: edit a file in Monaco, wait 5 s, see the USER commit
  appear in the History panel with the pencil icon and owner email.
