## Scope

Enable image attachments end-to-end for queued durable runs. Currently
`page.tsx` hard-blocks any submit with attachments. The protocol type
`PromptImageAttachment` exists, the OpenHands runner already consumes
attachments via a manifest, but every host/worker/broker boundary in
between drops them.

In-scope runtimes: OpenHands, Claude Code, OpenAI Codex.
Out-of-scope: Vercel AI runtime.

## Plan

Storage: Postgres base64 in a new `MessageAttachment` table (1:N to
`Message`, ON DELETE CASCADE).

Phase A — local, sequential (DB-coupled):
- Schema: add `MessageAttachment` model + Prisma migration.
- Host POST `/api/projects/[id]/runs`: accept `attachments` array, validate
  size/count, persist with the user `Message`.
- `enqueueAgentRun`: accept `attachments`, write `MessageAttachment` rows in
  the same transaction.
- Drain (`executor-client.ts`): load attachments alongside `userMessage`,
  forward in `ExecuteProjectRunRequest`.
- Session-load endpoint: include attachment metadata in message reads.

Phase B — parallel sub-agents:
- SA1 (medium) — Worker-agent + Broker protocol plumbing.
  Add `attachments?` to `ExecuteProjectRunRequest`, `ExecuteRunCommand`,
  validators, and `executeAgentRun`. Pass through to `runTurn`.
- SA2 (medium) — Runner consumption.
  Broker writes images to `/workspace/project/.agent-artifacts/chat-attachments/<turnId>/<i>.<ext>` once before `runTurn`. Claude/Codex
  receive an augmented prompt with `@<path>` references; OpenHands runner
  uses its existing manifest mechanism (already implemented).
- SA3 (medium) — Frontend.
  Remove the gate at `page.tsx:2210`. Persist attachments through submit.
  Render attachments inside the user `Message` view both for new submits
  and for session-loaded history (extend the message read pipeline).

Phase C — verification, finalisation:
- `pnpm lint`, `pnpm build`, focused tests for queue / executor-client /
  worker-agent / broker / route handlers.

Limits:
- Max 5 images per turn, 8 MB each (matches existing UI constants).
- Total payload guarded server-side too.

## Notes

- Host enqueue still writes a single user `Message` row; attachments are a
  child table. Existing event/log replay paths stay untouched.
- The "queued durable" in the error message refers to the host-API queue;
  the deprecated WS `agent.prompt` path is unaffected.
- Resume runs must not rewrite attachments to disk on every retry —
  broker-side write is idempotent (run-scoped directory) and runs once per
  attempt.

## Outcome

Image attachments now flow end-to-end through the durable-run pipeline.

- Schema: `MessageAttachment` (1:N to `Message`, ON DELETE CASCADE).
  Migration `20260507140000_add_message_attachments`.
- Host: `POST /api/projects/[id]/runs` accepts `attachments`, validates
  count/MIME/size/base64, persists rows, rejects vercel-ai + attachments.
  Drain lookup forwards attachments into `ExecuteProjectRunRequest`.
  Session GET surfaces attachments as `dataUrl` for chat history.
- Protocol: bumped to 1.13.0. `attachments?` field added to
  `ExecuteProjectRunRequest`, worker-agent + broker validators, broker
  `ExecuteRunCommand`, and `executeAgentRun` input.
- Broker: new `chat-attachments.ts` writes images under
  `${projectRoot}/.agent-artifacts/chat-attachments/<runId>/<i>.<ext>`.
  Claude/Codex receive an augmented prompt with `@<abs-path>` references;
  OpenHands gets attachments passed through (existing manifest path).
  Vercel-AI ignored.
- Frontend: gate at `page.tsx` removed; submit POSTs `attachments` and
  optimistically renders them in the user message. Session-load rehydrates
  attachments from the server.
- Side fix: pre-existing Dockerode type issue in
  `worker-agent/src/index.ts` blocked `pnpm build` on main; widened the
  `docker.pull` callback to optional `stream` and added a guard. Two-line
  fix, no runtime behavior change.
