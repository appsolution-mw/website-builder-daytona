## Scope

Review the current OpenHands integration against the current OpenHands SDK and
Remote Agent Server documentation, with a focus on whether the project should
continue using the local Python bridge or adopt a Remote Agent Server path.

## Plan

- Inspect the local OpenHands runtime flow and sandbox setup.
- Compare the implementation with current OpenHands SDK and Remote Agent Server
  guidance.
- Identify concrete risks, gaps, and recommended next steps.

## Notes

- Current implementation uses `openhands-sdk` and `openhands-tools` in
  standalone/local SDK mode inside the existing project sandbox.
- The project does not install or use `openhands-workspace`,
  `openhands-agent-server`, `DockerWorkspace`, `APIRemoteWorkspace`, or
  `RemoteConversation`.
- A Remote Agent Server would overlap with the existing worker/sandbox/broker
  architecture unless the project deliberately moves OpenHands execution out of
  the current broker process.
- Highest-value improvements before a remote migration are conversation
  persistence/resume, safer environment handling, and a less brittle SDK event
  integration.

## Outcome

Completed architecture review. Recommendation: keep the local SDK bridge for
the current worker-pool/Daytona architecture, but plan a focused spike for an
optional Remote Agent Server mode only if centralized multi-user OpenHands
execution, OpenHands Cloud workspaces, or SDK-native remote observability become
product goals.
