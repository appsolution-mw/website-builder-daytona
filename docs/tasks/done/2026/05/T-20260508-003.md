## Scope

Nach T-20260508-001 berichtete der User, dass Codex trotz Multimodal-Patch
weiter „kein eingebundenes Bild-Asset" sieht. Ursache war primär, dass die
Patch-Commits nicht gepusht waren — das Sandbox-Image lief noch mit der alten
Version.

Während wir den Push vorbereiten: zwei kleine Robustheits-Verbesserungen +
Diagnose-Logging einbauen, damit das nächste Mal (falls es trotzdem hakt)
sofort sichtbar ist, was los ist.

Hintergrund-Recherche bestätigt: `@openai/codex-sdk@0.122.0` propagiert
`local_image` korrekt zu `--image` (siehe `dist/index.js:217-221`); `gpt-5.4`
unterstützt Vision; bekannte Codex-CLI-Issues (#2473, #14895, Vaughan-Blog)
zeigen aber, dass das Modell trotz korrekter Übergabe manchmal das Bild
ignoriert, vor allem wenn der Prompt keinen expliziten Bild-Verweis enthält.

## Plan

- Pre-flight-Existence-Check via `fs.stat` für jede `attachmentPath`-Eintrag.
  Nur Pfade durchreichen, die als nicht-leere reguläre Datei existieren.
- Prompt-Preface „The user attached N images via the runtime image-input
  channel. Examine each one carefully before responding." vor den
  Original-Prompt setzen, wenn Bilder vorhanden sind. Verbessert laut
  Issue-Reports die Reliability bei Codex.
- Diagnose-Log `[broker codex] turnId=… attachments=N dropped=… paths=[…]`
  per `console.log` vor dem `runStreamed`-Call. Reicht um beim nächsten Bug-
  Report sofort zu sehen, ob Pfade ankamen.
- Tests: zwei neue Cases — Filter dropt fehlende/leere Files; alle-fehlend
  fällt auf String-Prompt zurück. Bestehende Cases nutzen jetzt echte tmp-
  Files (sonst würde der Filter sie wegwerfen).

## Notes

- Kein stderr-Capture vom Codex-CLI: das SDK kapselt den Spawn ohne externe
  Hooks; Hijacking wäre Monkey-Patching. Wenn nötig später via Env-Var
  `RUST_LOG=…` an den CLI durchreichen.
