## Scope

Diagnose why the worker-agent returns `502 broker-command-failed` when executing a queued chat run after the queue is unblocked.

## Plan

- Inspect the latest failed run and queue state.
- Test the worker-agent to broker forwarding path with a safe command.
- Check sandbox and broker lifecycle evidence.
- Propose the smallest next fix.

## Notes

- The latest runs start after skip, then fail at the Worker-Agent forwarding boundary with `502 broker-command-failed`.
- A signed direct `git/status` request to the worker-agent also returns `502 {"error":"broker-command-failed","reason":"fetch failed"}`.
- The sandbox container is `running`, so the failure is not queue state or OpenHands execution.
- The worker-agent was hard-coded to forward broker commands to `127.0.0.1:<brokerPort>`, which is wrong when the worker-agent itself runs in Docker.
- Added `BROKER_HOST` support and configured local/Hetzner worker-agent startup to use `host.docker.internal` with Docker host-gateway mapping.

## Outcome

Completed. New worker-agent deployments can reach host-published sandbox broker ports through `host.docker.internal`; the currently running Hetzner worker still needs to be redeployed/restarted with the updated worker-agent image/config.
