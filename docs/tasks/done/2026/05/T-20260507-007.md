## Scope

Replace the fixed worker-agent timeout for streaming agent executions with
open-ended streaming semantics aligned with OpenHands long-running tasks.

## Plan

- Research OpenHands timeout, pause, cancel, and streaming behavior from
  primary sources.
- Inspect the local host, worker-agent, and broker run path.
- Update tests to require no fixed abort timeout for agent streams.
- Implement a focused fix and update runtime documentation.
- Verify the focused tests and lint.

## Notes

The previous 30 minute run timeout still had the wrong semantics for tasks that
can intentionally run for hours. Abort should represent explicit cancellation or
transport failure, not a fixed elapsed time.

OpenHands primary sources describe agent work as a long-lived conversation with
state, event streaming, pause/resume, and stuck detection. Remote runs can be
triggered separately from waiting, and OpenHands notes that timeouts on blocking
waits may leave the server-side conversation still running.

## Outcome

Removed the fixed run-stream timeout and `WORKER_AGENT_RUN_TIMEOUT_MS` runtime
configuration. `executeProjectRun` no longer attaches an `AbortSignal` to the
long-running NDJSON stream. Short worker-agent calls still use
`WORKER_AGENT_TIMEOUT_MS`.

Verified with:
- `pnpm test:host lib/runtime/worker-pool/__tests__/agent-client.test.ts`
- `pnpm lint`

DB-backed host tests were not run because the current environment points at
database `wbd`, and the test safety guard requires `TEST_DATABASE_URL` to point
at an isolated test database.
