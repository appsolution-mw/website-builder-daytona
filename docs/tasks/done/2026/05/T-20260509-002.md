# T-20260509-002 - Bump worker-agent broker-readiness timeout to 180s

Status: Done
Date: 2026-05-09
Owner: AI Agent

Related files:
- worker-agent/src/broker-ready.ts

## Scope

Follow-up #1 of the Claude Agent SDK integration. The broker-readiness probe in
worker-agent timed out after 60s. With the new sandbox entrypoint that starts
agent-runner alongside the broker, first-time bring-up can exceed 60s
(next dev compile + agent-runner spin-up + broker start). When the timeout
fires, the host never receives the broker-ready callback and the project
sits in "provisioning" forever.

## Plan

- Bump the default `timeoutMs` in `watchBrokerReadiness` from 60_000 to
  180_000 (3 minutes).
- Update the JSDoc comment to reflect the new default.
- Keep the 250ms polling interval unchanged.
- Do not add a second probe; the broker is started last in the new
  entrypoint, so a successful broker `/health` implies agent-runner is up.

## Notes

- Both call sites (`worker-agent/src/index.ts` and
  `worker-agent/src/server.ts`) use the default timeout — no per-call
  overrides need to change.
- No existing test asserts the previous 60_000 default.

## Outcome

Done. Default `timeoutMs` bumped from 60_000 to 180_000 in
`worker-agent/src/broker-ready.ts` (and JSDoc updated). All 45
worker-agent vitest tests pass. Pre-existing typecheck error in
`tests/docker.test.ts(147,26)` is on `main` and unrelated to this change.
