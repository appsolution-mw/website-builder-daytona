# T-20260507-008 - LAN/prod-aware default for ws-proxy URL

Status: Done
Date: 2026-05-07
Owner: AI Agent

Related files:
- app/project/[id]/page.tsx

## Scope

Replace the hardcoded `ws://localhost:4100` fallback for the workspace ws-proxy
connection with a protocol- and host-aware default derived from
`window.location`. This makes the dev server reachable from other devices on
the LAN without setting `NEXT_PUBLIC_WS_PROXY_URL`, and keeps the production
path on `wss://<same-origin>` (proxied by Caddy on 443) when no explicit
override is configured.

## Plan

- Replace the static fallback in the workspace page's ws connect effect with a
  derived value that uses `window.location.protocol` (`wss:` for `https:`,
  `ws:` otherwise) and `window.location.hostname`. Append `:4100` only for
  the `ws:` (dev) case.
- Keep `NEXT_PUBLIC_WS_PROXY_URL` as an explicit override.
- Verify with `pnpm lint` and `pnpm build`.

## Notes

- Dev/localhost: `http://localhost:3000` -> `ws://localhost:4100` (unchanged).
- Dev/LAN: `http://192.168.x.y:3000` -> `ws://192.168.x.y:4100` (newly works).
- Prod: `https://<domain>` -> `wss://<domain>` (Caddy reverse-proxies the
  `/p/*` path to the host ws-proxy on port 4100).

## Outcome

Replaced the `ws://localhost:4100` fallback in
`app/project/[id]/page.tsx` with a `window.location`-derived default. The dev
ws-proxy is now reachable from any device on the LAN that can open the host
Next.js dev server, without setting `NEXT_PUBLIC_WS_PROXY_URL`. In production
(`https://`) the default resolves to `wss://<same-origin>`, matching the
existing Caddy reverse-proxy assumption. `NEXT_PUBLIC_WS_PROXY_URL` remains
available as an explicit override.

Verified with:
- `pnpm lint`
- `pnpm build`
