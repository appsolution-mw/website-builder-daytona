## Scope

Drive the H.1c + H.1d live validation against real Hetzner, Tailscale,
Cloudflare, and GHCR by writing a bootstrap script for the Caddy edge
server and fixing every issue surfaced during the validation run.

## Plan

  edge server, joins it to Tailscale, installs Caddy with the Cloudflare
  DNS plugin, points the Cloudflare wildcard DNS at the edge, and writes
  `CADDY_ADMIN_URL` into `.env`.
- Walk through the validation script `scripts/validate-hetzner-worker-pool.md`,
  fixing every blocker as it appeared.

## Notes

Issues found and fixed during the live walk-through:

- Missing Prisma migration after the H.1c merge â€” applied
  `20260505180000_managed_worker_pool` against the local dev DB.
- Tailscale `createAuthKey` 400 because the description contained
  parentheses; both the bootstrap script and the Hetzner provisioner
  were generating descriptions with `()`.
- Hetzner `cx22` server type was deprecated; switched the script default
  to `cx23` (then took the user override `cx23` as the working value).
- Worker-agent Docker build failed in CI because `node-pty` needs
  `python3` / `make` / `g++` to compile native bindings; added them to
  the deps stage of `worker-agent/Dockerfile`.
- GHCR images were private after the first push; resolved with a
  classic PAT (`read:packages`) and three new env vars
  (`IMAGE_REGISTRY_HOST`, `IMAGE_REGISTRY_USERNAME`, `IMAGE_REGISTRY_TOKEN`).
  Cloud-init now runs `docker login` before the pulls, and the token is
  redacted in `redactCloudInit`.
- `waitForTailscaleIp` default of 60 s was too short for cold-boot
  (apt install + Caddy + Tailscale + image pulls real takes 3-6 min);
  defaults are now 90 attempts at 5 s with `TAILSCALE_LOOKUP_ATTEMPTS`
  / `TAILSCALE_LOOKUP_INTERVAL_MS` overrides.
- Tailscale + Hetzner + Caddy clients now include the upstream response
  body in their error messages so 4xx/5xx failures are debuggable.
- `HOST_URL` was being rendered as the full heartbeat path which the
  worker-agent then duplicated, producing 404s on every heartbeat. Now
  the cloud-init passes only the bare app base URL and the worker
  appends the path itself.
- Caddy admin API rejected calls from the host because (a) only
  localhost origins were allowed and (b) the request had no `Origin`
  header. Edge Caddyfile now whitelists `http://<edge-tailscale-ip>:2019`
  and the Caddy client sends an explicit `Origin` header.
- Caddy admin API rejected route writes with `invalid traversal path`
  because the Caddyfile had no site block, so `srv0` was never created.
  Added a `nothing.invalid` placeholder site (with `tls internal` so
  ACME doesn't try to issue a public certificate for it).
- `app/api/projects/route.ts` swallowed the spawn error in its catch;
  now logs it via `console.error` so dev-time provisioning failures are
  diagnosable from the dev log instead of just `sandbox provisioning failed`.

Known follow-ups (out of scope for this task):

- ws-proxy issues `broker connection failed` when the browser opens
  a freshly-spawned project before its broker is listening. Browser
  reconnects on its own so the UX is not blocked, but a small retry
  loop (e.g. 3 attempts, 500 ms backoff) would silence the log noise.
- Tailscale OAuth client support in `tailscale-client.ts` so the system
  can drop the long-lived `tskey-api-...` token in favor of an OAuth
  client + auto-refresh. Sketched out in chat; not blocking validation.
