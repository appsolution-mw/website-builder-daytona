## Scope

Two follow-ups identified at the close of T-20260507-009:

1. Sandbox containers were created with `RestartPolicy: { Name: "no" }` —
   a worker VM cold reboot left every sandbox in `Exited (255)` and
   required manual `docker start` to recover.
2. Watchtower-based auto-update was wired into the bootstrap path for
   *new* workers (cloud-init), but legacy workers (e.g. worker-4
   provisioned before the change) still need a one-shot bootstrap to
   pick up Watchtower + the worker-agent label + `IMAGE_REGISTRY_*` env
   vars. Doing this by hand is error-prone.

## Plan

- Sandbox restart policy: switch to `unless-stopped` in
  `worker-agent/src/docker.ts`; add a docker-test that asserts the
  RestartPolicy on createSandbox.
- Bootstrap script: `scripts/bootstrap-watchtower.ts` — TypeScript, run via
  `pnpm tsx`, reads worker list from the host DB (filter:
  `provider = hetzner`, `status NOT IN (DECOMMISSIONED, OFFLINE)`), SSHes
  via stored `tailscaleIp`, and applies an idempotent bootstrap script:
    - Watchtower (Nicholas-Fedor fork) with docker-config mount, recreated
      only when image / mount drift.
    - worker-agent recreated only when missing the watchtower label OR the
      `IMAGE_REGISTRY_TOKEN` env var, preserving its existing config.
  Supports `--only <worker-name>` and `--dry-run`.

## Notes

- Bootstrap script uses `process.loadEnvFile` from `.env` and a dynamic
  `import()` of the prisma client to avoid ES-module import-hoisting
  beating the env load (same gotcha hit in
  `scripts/setup-hetzner-edge.ts`).
- The script is safe to re-run: each worker reports
  "already current/configured" if no drift.
