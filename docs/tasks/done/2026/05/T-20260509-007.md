## Scope

12 host tests have been failing on `main` since T-20260507-001 (watchtower
env-var addition) and a separate auth-refactor that introduced
`requireAdminUserFromRequest` against ambient `ADMIN_USER_IDS` / `ADMIN_EMAILS`
env. Surfaced during Phase 1.4a final sweep.

Two clusters:

**A. Admin auth (9 tests, "403 instead of 200/201/204/409"):** the dev-admin
fallback in `isAdminUser` only kicks in when both `ADMIN_USER_IDS` AND
`ADMIN_EMAILS` are empty. Local `.env` populates `ADMIN_USER_IDS`, so Next.js
auto-loads it during `pnpm test:host`, the dev-admin gate stays closed, and
admin-route tests get 403.

**B. Hetzner / worker-pool env (3 tests, "didn't throw"):** validator now also
requires `WATCHTOWER_HTTP_API_TOKEN`. Happy-path test envs miss this var; the
second-call success expectation throws instead.

## Plan

- Cluster A: in each affected test file, clear `ADMIN_USER_IDS` and
  `ADMIN_EMAILS` from the runtime env in `beforeEach` so the dev-admin path
  activates. Restore in `afterAll` like the existing `DEV_USER_ID` pattern.
- Cluster B: add `WATCHTOWER_HTTP_API_TOKEN` to the happy-path env objects in
  the three failing tests. Where the test asserts a missing-env throw, add a
  matching test for the new var (or accept that the existing first-call
  assertion still covers "validation runs").

## Notes

Not part of Phase 1.4a but flagged during its sweep. Out-of-scope variants
(other missing-env tests, unrelated 403 paths) stay untouched.

## Outcome

now reports 381/381 (was 369/12). `pnpm lint` clean.

Three commits:

   `.../drain/__tests__/route.test.ts`,

2. `fix(tests): align project restart tests with broker-readiness flow` —

   would otherwise leak through `{ ...originalEnv, ... }` and suppress
