## Scope

Two related issues seen on http://launchnode.de (Hetzner) right after creating
a project:

1. The host marks the Project `RUNNING` as soon as `spawnProjectSandbox` returns
   from the worker-agent — but the broker process inside the sandbox container
   may not yet be listening, so the WS-proxy gets `ECONNREFUSED`. The Open
   button on the homepage and the workspace UI on the project page are shown
   too early.
2. `crypto.randomUUID()` is undefined in non-secure contexts (HTTP without
   localhost). On `http://launchnode.de` every WS-driven RPC call from the
   project page throws `TypeError`.

## Plan

Patch A — crypto.randomUUID polyfill (independent, smallest):
- Add `lib/random-id.ts` with a `randomId()` helper that prefers
  `globalThis.crypto.randomUUID` and falls back to `crypto.getRandomValues`
  RFC4122 v4 hex output.
- Replace all `crypto.randomUUID()` call sites in
  `app/project/[id]/page.tsx` and `components/workspace/XtermTerminal.tsx`
  with `randomId()`.

Patch B — explicit broker readiness signal (push-based):
- Schema: add `brokerReady Boolean @default(false)` and
  `brokerReadyAt DateTime?` to `Project`. Generate a focused migration.
- Broker: add a tiny `GET /health` route on the sandbox broker HTTP/WS port
  that returns `{ ok: true }` once the listener is up.
- Worker-agent: after `docker.createSandbox` succeeds in `POST /sandboxes`,
  start an async readiness watcher for that sandbox. It polls
  `http://${brokerHost}:${brokerPort}/health` every 250 ms (capped at 60 s).
  On first success, it sends an HMAC-signed
  `POST /api/internal/sandboxes/{sandboxId}/broker-ready` to the host.
- Host: new route `app/api/internal/sandboxes/[id]/broker-ready/route.ts`,
  HMAC-verified, looks up the `WorkerSandbox` row, updates the linked
  `Project` with `brokerReady=true, brokerReadyAt=now()` (idempotent).
- Reset on lifecycle transitions:
  - Initial spawn (`POST /api/projects`) writes `brokerReady=false` together
    with `status=RUNNING`.
  - `POST /api/projects/[id]/restart` writes `brokerReady=false` before
    triggering a respawn; the worker-agent's readiness watcher flips it back.

Patch C — UI gating:
- Add `brokerReady` (+ `brokerReadyAt`) to `projectSelect` and
  `serializeProject`; surface it on the homepage list response and on the
  detail endpoint.
- Homepage: gate the Open button on
  `status === "RUNNING" && brokerReady === true`. While `RUNNING` but not
  yet ready, show a small "wird vorbereitet..." badge with a spinner. The
  existing polling already runs while any project is `PROVISIONING`; extend
  the trigger to also poll while any project has `RUNNING && !brokerReady`.
- Project page: extend the existing `WorkspaceLoadingState` gate to require
  both `project.brokerReady === true` and `wsStatus === "open"` (current
  gate uses `brokerReadyProjectId`, which is set on the first WS-open even
  if the broker actually crashed). Reset the gate when either flips to
  false. Poll `/api/projects/[id]` while waiting for `brokerReady`.

## Notes

- HMAC scheme on the new internal route mirrors
  `app/api/internal/workers/[id]/heartbeat/route.ts` exactly (same secret,
  same 5 min window, same `${ts}.${method}.${path}.${body}` canonical form).
- Worker-agent already has the host URL via heartbeat config — reuse it for
  the broker-ready signal.
- The broker `/health` route must answer over the same HTTP server that
  hosts the WS upgrade (it already mounts `/internal/...`).
- Polyfill must be plain `node`/browser-safe, no `react`/`window`-only code,
  so it works in tests and SSR alike.

One-time bootstrap on each existing Hetzner worker (only needed once;
new workers pick this up automatically via cloud-init):

```bash
ssh root@<worker-public-ip>   # or `tailscale ssh root@<hostname>`

# 1. Pull and start Watchtower next to the existing worker-agent
WATCHTOWER_TOKEN=<32+ random chars, store in 1Password>
docker pull containrrr/watchtower:latest
docker run -d --name watchtower --restart unless-stopped \
  -v /var/run/docker.sock:/var/run/docker.sock \
  -e WATCHTOWER_LABEL_ENABLE=true \
  -e WATCHTOWER_CLEANUP=true \
  -e WATCHTOWER_INCLUDE_RESTARTING=true \
  -e WATCHTOWER_HTTP_API_UPDATE=true \
  -e WATCHTOWER_HTTP_API_TOKEN=$WATCHTOWER_TOKEN \
  -p 8080:8080 \
  containrrr/watchtower:latest

# 2. Recreate the worker-agent with the watchtower-enable label
docker rm -f worker-agent
docker run -d --name worker-agent --restart unless-stopped \
  --label com.centurylinklabs.watchtower.enable=true \
  -p 4500:4500 \
  --add-host=host.docker.internal:host-gateway \
  -v /var/run/docker.sock:/var/run/docker.sock \
  -e WORKER_ID=<existing worker id> \
  -e HMAC_SECRET=<existing hmac secret> \
  -e HOST_URL=<APP_BASE_URL> \
  -e SANDBOX_IMAGE=<SANDBOX_IMAGE> \
  -e BROKER_HOST=host.docker.internal \
  ghcr.io/appsolution-mw/worker-agent:latest
```

New env vars / GitHub secrets:
- `WATCHTOWER_HTTP_API_TOKEN` — bearer token shared between the workers'
  Watchtower instances and the GitHub Action that triggers them. Set both
  as a host env var (so re-provisioning bakes it into cloud-init) and as
  a GitHub Actions secret.
- `TAILSCALE_OAUTH_CLIENT_ID` / `TAILSCALE_OAUTH_SECRET` — OAuth client
  with `auth_keys` scope and tag `tag:wbd-ci`. The GitHub Action exchanges
  it for a short-lived ephemeral auth-key on every run (no expiration to
  manage). Created once in Tailscale Admin → Settings → OAuth clients.
- `TAILSCALE_API_KEY` / `TAILSCALE_TAILNET` — already in place for the
  worker provisioner; reused by the workflow to enumerate `tag:wbd-worker`
  devices.

After the one-time bootstrap, the loop is:
1. Push to `main` (or merge a PR).
2. CI builds and pushes `ghcr.io/appsolution-mw/worker-agent:latest`.
3. CI joins the Tailnet, queries Tailscale for `tag:wbd-worker` devices,
   POSTs to each worker's `:8080/v1/update` Watchtower endpoint.
4. Watchtower pulls the new image and restarts the worker-agent on each
   worker — which triggers our self-heal pass (worker-agent/src/index.ts)
   that probes every running sandbox's broker and reports readiness back
   to the host. The user-visible UI catches up automatically.
